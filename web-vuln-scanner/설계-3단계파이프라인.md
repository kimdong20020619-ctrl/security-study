# 설계: 3단계 파이프라인 (scanner → analyzer → reporter)

허브: [[허브]]

## 핵심 결정
1번 프로젝트(`llm-redteam-framework`)와 달리 **ABC 기반 다중 구현체 스왑 구조를 채택하지 않음**. 스캐너는 ZAP 하나, 판정기는 LLM 하나, 리포터는 HTML 하나로 확정됐고 당장 교체할 두 번째 구현체 계획이 없어서, 인터페이스를 미리 추상화하면 YAGNI 위반이 된다고 판단.

## 구조
`scanner.py`(ZAP API 스캔) → `analyzer.py`(LLM 재판정) → `reporter.py`(HTML 출력), `models.py`(Alert/AnalyzedAlert/Verdict)와 `persistence.py`(단계 간 JSON 저장/로드)가 이를 지원. 스캔이 가장 오래 걸리는 단계라 중간 결과를 JSON으로 남겨 재실행 가능하게 함(`--skip-scan` 플래그).

## 스캔 대상
로컬 Docker의 OWASP Juice Shop(연습용 취약 웹앱), Docker Compose로 ZAP과 함께 실행. 실제 서비스는 절대 대상으로 하지 않음.

## 설계 단계에서 미리 반영한 교훈
1번 프로젝트의 `HtmlReporter`는 이스케이프 처리를 사후에 추가했음(리뷰에서 HTML 인젝션 발견). 이번엔 설계 스펙 단계부터 "alert 텍스트는 외부 응답을 담고 있어 인젝션 위험이 있으니 html.escape() 처리를 처음부터 포함한다"고 명시 — 실제로 태스크별 리뷰에서 5개 필드 전부 이스케이프 확인됨.
