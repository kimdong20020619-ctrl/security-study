# 모듈 1 — Start Your Cyber Security Journey

> **구성**: 3개 룸 — 공격적 보안 입문 · 방어적 보안 입문 · 검색 스킬
> **복습용 노트**: 한 번 훑고 와도 이것만 읽으면 바로 이해되게 정리.

---

## 🧠 이 모듈 한 눈에 (먼저 이것만 봐도 됨)
- 보안엔 두 편이 있다: **공격(Offensive)** = 뚫는 사람 / **방어(Defensive)** = 막는 사람.
- 룸1에서 내가 직접 **웹을 뚫어봤고**(공격), 룸2에서 그 공격을 **탐지·차단해봤다**(방어).
- 룸3에서 **보안 정보를 어디서 찾는지**(검색 도구) 배웠다.
- **한 문장 요약: "공격을 알아야 방어할 수 있다."**

---

## 룸 1 — 공격적 보안 (내가 가짜 은행을 털어봄)

### 🔎 쉽게 말하면
공격적 보안 = **도둑처럼 생각해서 내 집 약점을 먼저 찾는 것**. (도둑 오기 전에 안 잠긴 창문 확인하기)

### 🖐️ 내가 실제로 한 것 (단계별)
1. FakeBank(가짜 은행 웹) 접속 → 내 계좌 **8881**, 잔액 마이너스.
2. 터미널에 명령을 침: `dirb http://fakebank.thm`
   - **dirb가 뭐냐면**: 웹사이트의 **숨은 페이지를 자동으로 찾아주는 도구**. `admin`, `login`, `images`... 같은 단어를 주소 뒤에 하나씩 붙여보며 "이 페이지 있어?" 두드려본다. 있으면 결과에 `+`로 표시.
   - **왜 하냐면**: 개발자가 링크는 안 걸었지만 **삭제 안 한 페이지**가 종종 있다.
3. dirb가 숨은 페이지 **`/bank-transfer`**(관리자 송금 페이지) 발견.
4. 그 페이지에 들어가 계좌 8881에 $2000 입금 → 잔액이 양수로 → 초록 플래그 **`BANK-HACKED`** 팝업.

### 💡 그래서 핵심
1. **"숨겨두면 안전하겠지"는 착각** — dirb로 몇 초면 들통(= Security through Obscurity의 한계).
2. 공격 순서 = **정보 모으기 → 약점 찾기 → 악용**.
3. 실제 사이트에 하면 **범죄**. 허가된 실습 환경에서만.

### ✅ 실습 기록 (내 답)
| Task | 질문 | 답 |
|:--:|------|-----|
| 1 | 해커처럼 흉내 내 취약점을 찾는 것은? | `Offensive Security` |
| 2 | FakeBank 계좌번호는? | `8881` |
| 3 | dirb가 찾은 또 다른 숨은 URL? | `http://fakebank.thm/bank-transfer` |
| 4 | 초록 플래그(대문자)? | `BANK-HACKED` |

## 룸 2 — 방어적 보안 (내가 SOC 분석가가 되어 막음)

### 🔎 쉽게 말하면
방어적 보안 = **경비원처럼 CCTV(로그)를 보고 수상한 사람을 찾아 막는 것**. 공격과 달리 **공격 안 하고, 지켜보고 보호**한다.

### 🖐️ 내가 실제로 한 것 (신입 SOC 분석가 Joe를 도움)
1. **모니터링 대시보드**(경보판) 확인 → 수상한 알림이 뜸.
2. 수상한 **출발지 IP `32.122.195.63`** 를 찾음.
3. 그 공격자가 뭘 하는지 조사 → "URL 탐색 시도" 목록에서 **`https://fakebank.com/admin`** 을 찾으려 함
   - = **관리자 페이지를 찾는 중**. ❗ 이게 바로 룸1에서 내가 dirb로 한 짓과 똑같다.
4. 방화벽에 그 IP를 **BLOCK**(차단) → 성공 → 플래그 **`THM{FAKEBANK-SECURED}`**.

### 💡 그래서 핵심
1. 방어의 시작 = **가시성** — 로그·대시보드로 **지켜봐야** 대응한다.
2. 공격 당하면 **일단 격리(차단) 먼저** — 피해가 더 커지기 전에 막는 것(예: IP 차단).
3. **룸1(공격)을 룸2(방어)에서 막았다** → 공격을 알아야 방어한다(동전의 양면).

### ✅ 실습 기록 (내 답)
| Task | 질문 | 답 |
|:--:|------|-----|
| 1 | 방어적 보안의 주 목표는? | `Detect and respond to attacks` |
| 2 | 의심 트래픽 출발지 IP? | `32.122.195.63` |
| 3 | 공격자가 찾으려던 URL? | `https://fakebank.com/admin` |
| 4 | 성공 플래그? | `THM{FAKEBANK-SECURED}` |

## 룸 3 — 검색 스킬 (보안 정보 어디서 찾나)

### 🔎 쉽게 말하면
보안에선 그냥 구글링을 넘어 **전문 검색 도구**를 쓴다. **도구마다 찾는 게 다르다.**

### 🧰 도구별 쉬운 설명 (+ 내 답)
| 도구 | 쉽게 말하면 | 내 답 |
|------|-----------|-----|
| **Shodan** | "인터넷에 연결된 기계들의 구글". 서버·CCTV·공유기가 인터넷에 뭘 노출하는지 검색 | IP 185.243.115.47 → `tryhackme.thm` |
| **VirusTotal** | 파일/링크를 넣으면 백신 70여 개가 "악성이냐" 투표 | invoice_payment.exe → **`52`개** 벤더가 위험 판정 |
| **CVE** | 알려진 취약점의 "주민번호"(`CVE-2026-1337`). **CVSS**는 그 위험 점수(0~10) | CVE-2026-1337 → CVSS **`10`** (=Critical, 최악) |
| **man page** | 리눅스에서 `man 명령어` 치면 그 명령 설명서가 나옴 | `nc host.example.com 42` |
| **GitHub** | 연구자들이 취약점 시연 코드(PoC)를 올리는 곳 | 시연 스크립트 → `exploit.py` |

> 💬 **nc(netcat)** = 네트워크로 연결·통신하는 만능 도구.
> 💬 **PoC(Proof of Concept)** = 취약점이 진짜 된다는 걸 보여주는 시연 코드.

### 💡 그래서 핵심
1. **"어디서 찾을지"가 "무엇을 찾을지"만큼 중요** — Shodan(기계)·VirusTotal(악성)·CVE(취약점)·man(도구)·GitHub(PoC).
2. **CVSS 10점 = Critical** → 조직은 높은 점수부터 우선 조치.
3. GitHub PoC는 유용하지만 **악성일 수 있어 실행 전 검증** 필수.

## 📝 복습 셀프 체크 (답 가리고 맞혀보기)
1. 공격적 보안 vs 방어적 보안? → *공격=약점 찾기 / 방어=탐지·대응*
2. `dirb`는 뭘 하는 도구? → *숨은 웹 페이지 찾기*
3. 공격 당하면 제일 먼저 할 것? → *격리(차단)*
4. CVSS 10점의 의미? → *Critical(최고 위험)*
5. 파일이 악성인지 여러 백신으로 확인하는 사이트? → *VirusTotal*
6. 인터넷에 연결된 장치를 검색하는 엔진? → *Shodan*

> 위 6개를 막힘없이 답하면 이 모듈은 이해 완료. 막히면 해당 룸만 다시.

## 주요 용어 정리
| 용어 | 정의 |
|------|------|
| 공격적 보안(Offensive) | 공격자처럼 약점을 먼저 찾는 보안 |
| 방어적 보안(Defensive) | 탐지·조사·대응으로 시스템을 지키는 보안 |
| dirb | 숨은 웹 디렉터리·페이지 탐색 도구 |
| SOC 분석가 | 관제센터에서 알림을 감시·대응하는 사람 |
| 격리(Containment) | 공격 진행 중 피해 확산을 막는 즉각 조치 |
| 가시성(Visibility) | 로그·모니터링으로 무슨 일이 일어나는지 보는 능력 |
| Shodan | 인터넷 연결 장치 검색엔진 |
| VirusTotal | 다중 백신으로 파일·URL 악성 여부 확인 |
| CVE | 알려진 취약점 표준 식별자(CVE-연도-번호) |
| CVSS | 취약점 위험 점수(0~10, 10=Critical) |
| PoC | 취약점을 시연하는 코드 |
| man page | 리눅스 명령 매뉴얼 |
| nc (netcat) | TCP/UDP 연결·통신 도구 |
