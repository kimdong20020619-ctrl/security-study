## CIA 트라이어드 (The CIA Triad)

## 섹션 1 — CIA 트라이어드란?

**CIA 트라이어드 (CIA Triad)** = 사이버 보안이 보호해야 하는 세 가지 핵심 원칙을 묶은 보안 사고방식

> 이름이 비슷해 헷갈릴 수 있는데, 여기서 CIA는 미국 중앙정보국(Central Intelligence Agency)이 아니다. **Confidentiality, Integrity, Availability**의 앞 글자를 딴 보안 전문 용어다.

과거에는 정보가 종이 문서로 저장됐지만, 오늘날은 디지털 시스템에 보관되고 네트워크를 통해 전송된다. 적절한 보안 조치가 없으면 이 데이터는:

- **잘못된 사람에게 노출**되거나 (기밀성 침해)
- **허가 없이 수정**되거나 (무결성 침해)
- **가장 필요할 때 이용 불가능**해질 수 있다 (가용성 침해)

이 세 가지 위협에 대응하는 원칙이 바로 **CIA 트라이어드**다. 사이버 보안에서 접하는 거의 모든 개념은 이 세 가지를 중심으로 전개된다.

```
          기밀성
       Confidentiality
           △
          / \
         /   \
        /     \
  무결성 ———— 가용성
 Integrity  Availability
```

| 원칙 | 영문 | 핵심 질문 | 한 줄 정의 |
|------|------|----------|------------|
| **기밀성** | Confidentiality | 누가 볼 수 있는가? | 승인된 사람만 데이터에 접근할 수 있도록 보장 |
| **무결성** | Integrity | 누가 바꿀 수 있는가? | 권한 없는 사람이 데이터를 수정하지 못하도록 보장 |
| **가용성** | Availability | 언제 쓸 수 있는가? | 권한 있는 사용자가 필요할 때 데이터와 서비스를 이용할 수 있도록 보장 |

> **TryHackMe 번역 오류 주의**
> TryHackMe 한국어 번역에서 Integrity를 **"진실성"**, Availability를 **"유효성"** 으로 번역한다.
> 두 표현 모두 잘못된 번역이다. 보안 업계 표준 한국어 용어는 각각 **무결성**과 **가용성**이다.
> - "진실성(眞實性)" = 사실을 말한다는 의미 → 데이터 보안 문맥과 의미가 다름
> - "무결성(無缺性)" = 결함이나 변조가 없는 상태 → 보안 문맥에서 정확한 표현
> - "유효성(有效性)" = 효과가 있다는 의미 → 접근 가능성과 의미가 다름
> - "가용성(可用性)" = 사용할 수 있는 상태 → 보안 문맥에서 정확한 표현

## 섹션 2 — 기밀성 (Confidentiality)

**기밀성 (Confidentiality)** = 민감한 데이터가 **승인된 사람(authorized person)만** 접근할 수 있도록 보장하는 원칙

기밀성이 침해되면:
- 금전적 손실 (계좌 정보 유출, 사기 피해 등)
- 개인정보 침해 (신원 도용, 사생활 침해 등)
- 법적 문제 (개인정보 보호법 위반 등)

### 현실 세계 예시 — 엿듣기

사적인 대화를 나누는데 모르는 사람이 엿듣고 그 정보로 나중에 협박이나 조종에 사용한다면 기밀성이 침해된 것이다.

**예방법**: 안전한 장소에서 대화, 주변 상황 파악, 낯선 사람 앞에서 민감한 정보 언급 자제

### 디지털 세계 예시 — 공용 와이파이 도청

카페의 공용 와이파이에서 소셜 미디어에 로그인했다가 갑자기 로그아웃되고 계정에 접속이 불가능해진다. 이는 누군가가 같은 네트워크 내부에서 로그인 정보(아이디/비밀번호)를 가로챈 것이다. 데이터가 암호화되지 않은 채로 전송되면 중간에서 이를 훔쳐보는 것이 가능하다.

**예방법**:
- **암호화(Encryption)**: 데이터를 읽을 수 없는 형태로 변환해 가로채더라도 내용을 알 수 없게 함
- **접근 제어(Access Control)**: 승인된 사용자만 데이터에 접근할 수 있도록 권한을 관리

### 기밀성 판단 예시

| 상황 | 기밀성 달성 여부 | 이유 |
|------|----------------|------|
| Gmail 로그인 정보가 사무실 컴퓨터 모니터 포스트잇에 적혀 있다 | ❌ 침해 | 주변 모든 사람이 볼 수 있는 상태 |
| 회사 내부 문서는 업무상 필요한 직원들만 열람 가능하도록 권한 설정 | ✅ 달성 | 승인된 사람에게만 접근이 제한되어 있음 |
| 개인 문서가 링크 공개 설정으로 인터넷에 올라가 있다 | ❌ 침해 | URL을 아는 전 세계 누구나 접근 가능 |

## 섹션 3 — 무결성 (Integrity)

**무결성 (Integrity)** = **권한 없는 사람이 데이터를 수정하지 못하도록** 보장하는 원칙

"권한 없는"이 핵심이다. 승인된 사용자의 정상적인 수정은 무결성 침해가 아니다. **허가 없이 몰래 변경하는 것**이 문제다.

무결성이 침해되면:
- 데이터를 더 이상 신뢰할 수 없게 됨
- 변조된 데이터를 기반으로 잘못된 의사결정 발생
- 재무·의료·법률 기록 변조 시 심각한 결과 초래

### 현실 세계 예시 — 시험 점수 변조

교사가 채점한 시험 점수를 누군가가 기관에 제출하기 전에 몰래 바꿨다면 무결성이 침해된 것이다. 점수가 높아 보여도 그 데이터는 이미 신뢰할 수 없다.

**예방법**: 교사가 점수를 별도 용지에 따로 기록하고, 최종 제출 전에 원본과 비교 확인

### 디지털 세계 예시 — 은행 송금 계좌 변조

모바일 뱅킹으로 친구에게 돈을 보낼 때, 악성 프로그램이 거래 완료 전에 수취인 계좌 정보를 공격자의 계좌로 변경한다. 내가 입력한 계좌와 실제 송금된 계좌가 다르다. 데이터(계좌 정보)가 무단으로 수정된 것이다.

**예방법**:
- **해시(Hash)**: 데이터의 고유한 지문 값을 생성해, 변조 시 값이 달라지는 것으로 변조 탐지
- **디지털 서명(Digital Signature)**: 데이터가 특정 주체가 보낸 것임을 수학적으로 보증
- **체크섬(Checksum)**: 데이터 전송 전후 값을 비교해 변조 여부 확인

### 무결성 판단 예시

| 상황 | 무결성 달성 여부 | 이유 |
|------|----------------|------|
| 관리자 권한을 가진 승인된 사용자가 데이터베이스 레코드를 수정했다 | ✅ 달성 | 권한 있는 정상적인 변경 |
| 교사가 출석 기록을 잠금 상태로 저장했는데 학생 기록이 변경됐다 | ❌ 침해 | 권한 없는 무단 변경 |
| 주문이 결제 처리되는 사이에 주문 가격이 수정됐다 | ❌ 침해 | 권한 없는 무단 변경 |

## 섹션 4 — 가용성 (Availability)

**가용성 (Availability)** = **권한 있는 사용자가 필요할 때** 데이터와 서비스를 이용할 수 있도록 보장하는 원칙

현대 기업과 기관은 디지털 서비스에 크게 의존한다. 짧은 서비스 중단도 심각한 손실을 유발할 수 있다.

가용성이 침해되면:
- 사업 운영 불가능 (온라인 쇼핑몰 다운 → 판매 불가)
- 막대한 금전적 손실
- 의료·응급 통신 시스템 중단 시 생명에도 위협

### 현실 세계 예시 — 정전으로 인한 은행 접근 불가

은행에 돈이 안전하게 보관되어 있어도 정전으로 은행 문이 닫히면 돈을 인출할 수 없다. 데이터(돈)가 안전하게 존재하더라도 접근 자체가 불가능하면 가용성이 침해된 것이다.

**예방법**: 예비 발전기(UPS), 백업 시스템, 대체 접근 경로 마련

### 디지털 세계 예시 — DDoS 공격

**DDoS(Distributed Denial of Service, 분산 서비스 거부) 공격**은 수천~수만 개의 장치에서 대상 서버에 엄청난 양의 요청을 동시에 보내 서버를 마비시키는 공격이다. 서버는 정상 처리 용량을 초과하는 요청에 압도되어 응답을 멈춘다.

실제 사용자들은 데이터가 멀쩡히 존재해도 서비스에 접근할 수 없게 된다. 데이터 유출 없이도 서비스 마비만으로 심각한 피해가 발생하는 것이 가용성 공격의 특징이다.

**예방법**:
- **트래픽 부하 관리(Load Balancing)**: 요청을 여러 서버에 분산해 특정 서버에 과부하 방지
- **Rate Limiting**: 특정 IP가 임계값 이상의 요청을 보내면 자동 차단
- **CDN 활용**: 콘텐츠를 여러 지역 서버에 분산 배치해 공격 영향 최소화
- **백업 시스템**: 주 시스템 장애 시 즉시 전환 가능한 예비 시스템 운영

### 가용성 판단 예시

| 상황 | 가용성 달성 여부 | 이유 |
|------|----------------|------|
| 소프트웨어 설치 과정에서 필수 서비스가 일시 중단됐다 | ❌ 침해 | 권한 있는 사용자도 서비스 이용 불가 |
| 회사 웹사이트가 영업시간(09:00-18:00) 동안 접속 불가 상태다 | ❌ 침해 | 가장 필요한 시간에 서비스를 이용할 수 없음 |
| 모든 직원이 근무 시간 동안 필요한 시스템에 정상 접근 가능 | ✅ 달성 | 권한 있는 사용자가 필요한 시간에 이용 가능 |

## 섹션 5 — 보안 마인드셋 (Security Mindset)

CIA 트라이어드는 단순히 세 가지 정의의 집합이 아니다. **보안 전문가가 세상을 바라보는 사고방식(Security mindset)**이다.

보안 사고(Security Incident)가 발생했을 때, 보안 전문가들은 아래 세 가지 질문을 기준으로 영향 범위를 파악하고 대응 우선순위를 결정한다:

| 질문 | 판단하는 원칙 |
|------|-------------|
| 민감한 데이터가 권한 없는 사람에게 노출됐는가? | 기밀성 (Confidentiality) 침해 여부 |
| 허가 없이 데이터가 수정됐는가? | 무결성 (Integrity) 침해 여부 |
| 사용자들이 필요할 때 시스템이나 서비스를 이용할 수 없었는가? | 가용성 (Availability) 침해 여부 |

이 사고방식이 중요한 이유는, 하나의 보안 사고가 세 원칙 중 하나 또는 여러 개를 동시에 침해할 수 있기 때문이다. 어느 원칙이 침해됐는지 파악하면 어떤 복구 조치가 필요한지가 명확해진다.

### 실제 보안 사고 9가지 — CIA 원칙별 분류

아래는 실제로 발생하는 보안 사고 유형을 CIA 원칙에 따라 분류한 것이다.

**기밀성 침해 — 보면 안 되는 사람이 봤다**

| 사건 | 분류 |
|------|------|
| 백업 데이터 보안 설정 미흡으로 고객 민감 정보 유출 | 기밀성 |
| 직원이 로그인 자격증명(아이디/비밀번호)을 권한 없는 외부인과 공유 | 기밀성 |
| 직원 급여 파일이 잘못된 설정으로 인터넷에 공개 접근 가능 상태 | 기밀성 |

**무결성 침해 — 변경하면 안 되는 사람이 변경했다**

| 사건 | 분류 |
|------|------|
| 데이터베이스 레코드가 권한 없이 무단 수정됨 | 무결성 |
| 보고서 내용이 변조되어 재무 수치가 변경됨 | 무결성 |
| 공격자가 활동 흔적을 은폐하기 위해 시스템 로그를 삭제 | 무결성 |

**가용성 침해 — 필요한 사람이 필요한 때 이용할 수 없다**

| 사건 | 분류 |
|------|------|
| DoS(서비스 거부) 공격으로 이메일 서버가 오프라인 상태로 전환 | 가용성 |
| 중요 시스템 업데이트가 차단되어 서비스 정상 실행 불가 | 가용성 |
| 회사 웹사이트가 영업시간 중 접속 불가 상태 지속 | 가용성 |

> **주목할 포인트**: 시스템 로그 삭제가 무결성 침해로 분류된다. 로그는 "무슨 일이 있었는지"를 기록하는 데이터다. 공격자는 자신의 흔적을 지우기 위해 로그를 삭제하는 경우가 많은데, 이 행위 자체가 데이터(로그)를 무단으로 변조/삭제한 것이므로 무결성 원칙을 위반한다. 실제 침해 사고 분석(포렌식)에서 로그가 사라져 있다는 사실 자체가 공격의 흔적으로 간주된다.

## 주요 용어 정리

| 용어 | 정의 |
|------|------|
| **기밀성 (Confidentiality)** | 디지털 정보가 승인되지 않은 사람에게 노출되지 않도록 보장 |
| **무결성 (Integrity)** | 허가 없이 디지털 정보가 수정되지 않도록 보장 |
| **가용성 (Availability)** | 권한 있는 사용자가 필요할 때 디지털 정보를 이용할 수 있도록 보장 |
| **CIA 트라이어드 (CIA Triad)** | 위 세 가지 원칙을 통칭하는 보안 사고방식 |

| 이러한 모든 요소를 통칭하는 용어는? | `CIA Triad` |

**배운 점**

1. 기밀성과 무결성을 구분하는 핵심 기준은 **"볼 수 있는가(접근)"** 와 **"바꿀 수 있는가(수정)"** 다. 누군가가 데이터를 읽기만 했다면 기밀성 침해, 몰래 수정했다면 무결성 침해다. 같은 해킹 사고라도 어떤 행위가 발생했는지에 따라 침해된 원칙이 달라지기 때문에 구체적인 행위를 파악하는 것이 사고 분석의 첫 단계가 된다.
2. 무결성은 단순히 "데이터가 변경되지 않는 것"이 아니라 **"권한 없는 변경이 없는 것"** 이다. 관리자가 정상적으로 수정하는 것은 무결성 침해가 아니다. "데이터의 신뢰성이 떨어질 때 어떤 원칙이 침해됐는가?"에 Integrity가 답인 이유도 여기에 있다. 신뢰성이 떨어진다는 것은 누군가가 허가 없이 변조했을 가능성을 의미하기 때문이다.
3. CIA 트라이어드를 단순 암기하는 것보다 **각 원칙을 침해하는 공격 유형과 매핑하는 것**이 실전에 더 유용하다. Confidentiality → 도청/정보 유출, Integrity → 변조/위변조, Availability → DoS/DDoS. 이 매핑이 자연스럽게 되면 어떤 보안 사고 시나리오를 보더라도 빠르게 CIA 중 어떤 원칙이 침해됐는지 분류할 수 있다.

### 실습 2 — Task 3: 보안 사고 9개 분류 (드래그앤드롭)

**목표**: 9개 보안 사고 시나리오를 Confidentiality / Integrity / Availability 세 영역으로 분류

**결과**: 9/9 정답

| 사건 | 분류 |
|------|------|
| Sensitive customer data was leaked through an unsecured backup | Confidentiality |
| An employee shared the login credentials with an unauthorized person | Confidentiality |
| The employee salary files are publicly accessible to anyone on the Internet | Confidentiality |
| A database record was modified without authorization | Integrity |
| A report was altered, changing financial figures | Integrity |
| System logs are deleted to hide activity | Integrity |
| The email server was taken offline due to a denial of service attack | Availability |
| Critical system updates were blocked, preventing services from running | Availability |
| The company website is unavailable during business hours | Availability |

**플래그**: `THM{CIA_IS_ABOUT_BALANCE}`

**CIA 삼원칙은 단순 정의 집합이 아니라 하나의 사고방식, 이것은?** → `Security mindset`

**배운 점**

1. 시스템 로그 삭제(System logs are deleted to hide activity)가 **무결성 침해**로 분류된다는 것이 중요하다. 공격자는 침입 흔적을 지우기 위해 로그를 삭제하는 경우가 많다. 이 행위 자체가 로그라는 데이터를 무단으로 삭제/변조한 것이므로 무결성 위반이다. 실제 사고 분석(포렌식)에서 로그가 사라져 있다는 사실 자체가 "공격이 있었다"는 역설적인 증거가 되는 이유다.
2. DoS/DDoS 공격은 전형적인 **가용성 공격**이다. 데이터 유출이나 변조가 없더라도 서비스 자체를 이용할 수 없게 만드는 것만으로 심각한 보안 사고로 간주된다. 특히 의료 시스템, 금융 서비스처럼 가동 중단이 직접적인 피해로 이어지는 분야에서는 가용성이 기밀성만큼이나 중요한 원칙이다.
3. 플래그 `THM{CIA_IS_ABOUT_BALANCE}`에서 **"Balance(균형)"** 가 핵심 메시지다. 실제 보안 환경에서는 세 원칙 간에 트레이드오프가 발생한다. 기밀성을 극단적으로 강화하면(인증 절차 복잡화) 접근이 어려워져 가용성이 낮아진다. 반대로 가용성을 극단적으로 높이면(어디서나 쉽게 접근) 기밀성이 약해진다. 보안 전문가의 역할은 이 세 원칙 간의 균형을 상황과 조직의 필요에 맞게 최적화하는 것이다.

## 섹션 6 — 암호학이란?

**암호학 (Cryptography)** = 데이터를 읽을 수 없는 형태로 변환해 권한 없는 사람의 접근과 변조를 막는 기술

앞서 배운 CIA 트라이어드 중 **기밀성(Confidentiality)** 을 실제로 구현하는 핵심 수단이 암호학이다. 또한 전송 중 데이터가 변조되지 않았음을 확인하는 **무결성(Integrity)** 보호에도 활용된다.

**도입 질문**: 브라우저 주소 표시줄에 작은 자물쇠 아이콘이 보일 때, 인터넷을 통해 전송되는 내 데이터를 누군가가 읽거나 수정하는 것을 실제로 막는 것은 무엇일까?

→ 정답: **암호화(Encryption)**

**실제 시나리오**: 병원에서 환자의 이름, 질병, 치료 이력 등을 보험회사에 인터넷으로 전송할 때, 데이터가 수십 대의 라우터를 거치는 과정에서 보호 장치가 없으면 누구든지 읽거나 변경하거나 차단할 수 있다. 암호화가 이 문제를 해결한다.

**학습 목표**

- 암호화가 무엇이며 기밀성 보호에 왜 중요한지
- 평문(Plaintext)과 암호문(Ciphertext)의 차이
- 키(Key)와 알고리즘(Algorithm)이 무엇인지, 키의 비밀 유지가 왜 중요한지
- 대칭 암호화와 비대칭 암호화의 차이
- 실제 웹 브라우저에서 두 방식이 어떻게 결합되어 작동하는지

## 섹션 7 — 평문, 암호문, 그리고 시저 암호

### 평문(Plaintext) vs 암호문(Ciphertext)

| 용어 | 설명 | 예시 |
|------|------|------|
| **평문 (Plaintext)** | 암호화되지 않은, 누구나 읽을 수 있는 원본 데이터 | `HELLO` |
| **암호문 (Ciphertext)** | 암호화된 데이터. 키 없이는 읽을 수 없다 | `KHOOR` |

암호화의 목표: 암호문은 키를 모르는 사람에게 **무작위적인 의미 없는 문자열**처럼 보여야 한다.

앨리스가 `HELLO`를 밥에게 보내는 과정:
1. 앨리스가 알고리즘과 비밀 키로 `HELLO` → `KHOOR` 변환 후 전송
2. 밥이 `KHOOR` 수신 → 동일한 알고리즘과 키로 복호화 → 다시 `HELLO`

### 시저 암호 (Caesar Cipher)

시저 암호는 메시지의 각 글자를 알파벳에서 **고정된 숫자만큼 이동**시키는 암호화 방식이다. 기원전 50년경 율리우스 카이사르가 군사 메시지 보호에 사용했다고 전해진다.

- **알고리즘**: 각 글자를 알파벳에서 N칸 앞으로 이동 (완전히 공개된 정보)
- **키**: 이동할 숫자 N (비밀 정보)

**Key = 3 으로 HELLO 암호화 → KHOOR**

```
평문:   H  E  L  L  O
       +3 +3 +3 +3 +3
암호문: K  H  O  O  R
```

알파벳 순환 규칙 (key = 3):

| 평문 | 암호문 | | 평문 | 암호문 |
|------|--------|---|------|--------|
| A | D | | X | A (끝에서 처음으로) |
| B | E | | Y | B |
| C | F | | Z | C |

**복호화 (KHOOR → HELLO)**: 각 글자를 반대 방향으로 3칸 이동

```
암호문: K  H  O  O  R
       -3 -3 -3 -3 -3
평문:   H  E  L  L  O
```

**시저 암호의 핵심 원칙**:
- **알고리즘은 공개**: 어떻게 작동하는지는 누구나 알아도 됨
- **키만 비밀**: 앨리스와 밥만 이동 숫자를 알고 있음

### 시저 암호의 한계 — 왜 실제로는 쓰지 않는가?

시저 암호는 키가 1~25 사이의 숫자 하나뿐이다. 공격자가 암호문을 가로채면 25가지 경우를 모두 시도해보면 된다.

- 사람이 직접 시도: 지루하지만 가능
- **컴퓨터로 시도: 약 1밀리초(0.001초)** 만에 전부 해독 가능

시저 암호는 교육 목적으로만 사용하며, 실제 시스템에서는 절대 사용하지 않는다.

**실제 사용 알고리즘**: **AES (Advanced Encryption Standard, 고급 암호화 표준)**
- 훨씬 복잡한 수학 연산 사용
- 가능한 키 조합: `2^256`가지 (10^77) — 현존하는 모든 컴퓨터를 합쳐도 우주 나이보다 긴 시간 필요
- 기본 개념은 동일: **알고리즘 + 키 + 평문 → 암호문**

## 섹션 8 — 대칭 암호화 (Symmetric Encryption)

**대칭 암호화 (Symmetric Encryption)** = **동일한 키 하나**로 암호화와 복호화를 모두 수행하는 방식

시저 암호가 대칭 암호화의 가장 간단한 예다.

```
[앨리스]                                        [밥]
평문 HELLO
  → [암호화: 키 K 사용]
    → 암호문 KHOOR  ───전송───▶  암호문 KHOOR
                                  → [복호화: 동일한 키 K 사용]
                                    → 평문 HELLO
```

### 대칭 암호화의 장점

- **속도가 매우 빠름**: 대용량 데이터를 실시간으로 처리 가능
- **효율적**: 파일 암호화, 하드 드라이브 암호화, 네트워크 트래픽 암호화에 적합

### 대칭 암호화의 치명적 단점 — 키 분배 문제

앨리스와 밥이 처음 만날 때 어떻게 키를 안전하게 공유할까?

```
시도 1: 앨리스가 키를 인터넷으로 그냥 전송
  → 공격자가 가로채면 이후 모든 암호문을 복호화 가능 ❌

시도 2: "그럼 키를 암호화해서 전송하자!"
  → 그 키를 암호화하려면 또 다른 키가 필요
    → 그 키를 암호화하려면 또 다른 키가 필요
      → 끝없이 반복... ❌ (무한 퇴행)
```

이를 **키 분배 문제 (Key Distribution Problem)** 라 한다. 대칭 암호화를 단독으로 사용할 때의 근본적인 약점이다.

→ 해결책: 다음 섹션에서 배울 **비대칭 암호화**

---

## 섹션 9 — 비대칭 암호화 (Asymmetric Encryption)

**비대칭 암호화 (Asymmetric Encryption)** = **수학적으로 연관된 두 개의 키**를 사용하는 암호화 방식

| 키 종류 | 설명 |
|---------|------|
| **공개 키 (Public Key)** | 누구나 알고 사용할 수 있다. 인터넷에 공개해도 됨 |
| **개인 키 (Private Key)** | 오직 주인만이 비밀로 보관. 절대 공유하지 않음 |

**핵심 규칙**:
- 공개 키로 암호화한 내용 → **개인 키로만** 복호화 가능
- 두 키는 수학적으로 연결되어 있지만, 공개 키에서 개인 키를 계산해내는 데 **현대 컴퓨터로 수백~수천 년** 이 걸림 → 사실상 불가능

### 우체통 비유

```
┌─────────────────────────────┐
│          우체통             │
│  ┌───────────────────┐     │
│  │   우편 투입구     │ ← 공개 키 (Public Key)
│  │ (누구나 편지 투입 가능)  │   누구든 메시지를 넣을 수 있음
│  └───────────────────┘     │
│                             │
│  [잠긴 문] ← 개인 키 (Private Key)
│  (오직 우편함 주인만 열 수 있음)
└─────────────────────────────┘
```

### 앨리스가 밥에게 비밀 메시지를 보내는 과정

```
1. 밥이 공개 키(Public Key)와 개인 키(Private Key) 한 쌍 생성
   공개 키 → 웹사이트·이메일 등으로 전 세계에 공개
   개인 키 → 절대 공유하지 않고 본인만 보관

2. 앨리스가 밥의 공개 키를 입수 (공개된 정보라 가로채도 상관없음)

3. 앨리스가 밥의 공개 키로 메시지를 암호화 후 전송

4. 밥이 자신의 개인 키로 복호화 → 원본 메시지 확인
   공격자가 암호문을 가로채도 개인 키 없이는 해독 불가
```

### 키 배포 문제 해결

네트워크를 통해 **비밀 키를 전혀 교환하지 않아도 된다**. 공개적으로 전송한 것은 공개 키뿐이고, 공개 키는 애초에 비밀이 아니다.

## 섹션 10 — HTTPS와 인증서

### HTTPS — 비대칭과 대칭 암호화의 결합

비대칭 암호화가 가장 널리 사용되는 곳이 **HTTPS**다. 브라우저 주소창의 **자물쇠 아이콘** = 아래 과정이 완료됐다는 신호다.

`https://google.com` 접속 시 일어나는 일:

```
1. [브라우저] → 웹사이트에 공개 키 요청

2. [웹사이트] → 인증서(공개 키 + 신원 정보 포함)를 전송

3. [브라우저 + 웹사이트] → 비대칭 암호화를 이용해
                            공유 대칭 키(Shared Secret Key)를 안전하게 협의

4. 이후 실제 데이터 전송 → 속도가 빠른 대칭 암호화로 전환
```

이 방식을 **하이브리드 접근 방식 (Hybrid Approach)** 이라 한다:
- **비대칭 암호화**: 초기 키 교환 담당 → 키 분배 문제 해결
- **대칭 암호화**: 실제 데이터 처리 담당 → 속도 문제 해결

두 방식의 장점만 결합한 구조다.

### 인증서 (Certificate) — 공개 키의 신원 보증

**문제**: 앨리스가 받은 공개 키가 진짜 밥의 것인지, 밥인 척하는 공격자의 것인지 어떻게 알 수 있을까?

**해결책**: 인증서 (Certificate)

인증서는 아래 세 가지를 포함하는 디지털 문서다:
- 웹사이트의 공개 키
- 해당 키가 누구의 것인지 (예: `google.com`)
- **인증 기관(CA, Certificate Authority)** 의 디지털 서명

**CA (인증 기관)**: 신뢰할 수 있는 제3자 기관 (예: Let's Encrypt, DigiCert, GlobalSign)
- 브라우저와 운영체제에는 신뢰할 수 있는 CA 목록이 미리 내장되어 있음
- CA가 인증서에 "이 공개 키는 정말 이 사이트의 것이 맞다"는 서명을 함

**브라우저의 인증서 확인 과정**:

```
인증서 수신
  → CA 서명 확인 (신뢰 CA 목록과 대조)
  → 유효 기간 확인 (만료 여부)
  → 모두 정상 → 자물쇠 아이콘 ✅
  → 문제 발생 → 경고 메시지 + 연결 거부 가능 ⚠️
```

| 발행처 (Issued By) | `Let's Encrypt` (무료 공개 CA) |
| 발행일 | 2025년 11월 11일 |
| 만료일 | 2026년 2월 9일 |

> **Let's Encrypt**: 무료 SSL/TLS 인증서를 발급하는 비영리 CA 기관. 인증서는 90일마다 갱신이 필요하다. 소규모 사이트부터 대형 서비스까지 폭넓게 사용된다.

## 섹션 11 — 대칭 vs 비대칭 비교 및 실제 활용

### 두 암호화 방식 비교

| 항목 | 대칭 암호화 | 비대칭 암호화 |
|------|------------|--------------|
| **키의 수** | 1개 (암호화 = 복호화 키) | 2개 (공개 키 + 개인 키) |
| **키 공유** | 비밀리에 공유해야 함 ❌ | 공개 키는 공개적으로 공유 가능 ✅ |
| **속도** | 매우 빠름 ✅ | 상대적으로 느림 ❌ |
| **주용도** | 대용량 데이터 암호화 (파일, 네트워크 트래픽) | 키 교환, 디지털 서명 |
| **약점** | 키 분배 문제 | 느린 처리 속도 |
| **대표 알고리즘** | AES | RSA |

### 실제 시스템: 두 방식의 조합

현대 보안 시스템(HTTPS 등)은 두 방식을 결합해 각각의 약점을 서로 보완한다:

```
[1단계] 비대칭 암호화로 초기 키 교환
   → 인터넷을 통해 비밀을 공유하지 않아도 됨
   → 키 분배 문제 해결 ✅

[2단계] 대칭 암호화로 실제 데이터 전송
   → 빠른 속도
   → 대용량 데이터 처리에 효율적 ✅

결과: 자물쇠 아이콘 = 이 두 단계가 완료된 상태
      비밀번호, 은행 정보, 개인 메시지 등이 안전하게 보호됨
```

### 암호화의 한계 — 암호화는 만능이 아니다

암호화는 방어자의 가장 중요한 도구 중 하나지만, **보안 체계의 한 요소**일 뿐이다.

암호화만으로 해결되지 않는 것들:
- **약한 비밀번호**: 암호화가 되어 있어도 비밀번호가 `1234`라면 의미 없음
- **키 관리 실패**: 개인 키를 분실하거나 유출하면 암호화 자체가 무력화됨
- **사용자 실수**: 피싱에 속아 직접 정보를 입력하면 암호화로 막을 수 없음
- **소프트웨어 취약점**: 암호화 이외의 취약점으로 공격 가능
- **사고 대응 부재**: 침해 발생 시 탐지·대응 능력도 함께 필요

## 주요 용어 정리 (암호학)

| 용어 | 정의 |
|------|------|
| **평문 (Plaintext)** | 암호화되지 않은, 읽을 수 있는 원본 데이터 |
| **암호문 (Ciphertext)** | 암호화된 데이터. 키 없이는 의미 없는 문자열처럼 보임 |
| **키 (Key)** | 암호화와 복호화를 제어하는 비밀 값 |
| **알고리즘 (Algorithm)** | 키를 사용해 암호화·복호화하는 공개된 수학적 방법 |
| **대칭 암호화 (Symmetric Encryption)** | 암호화와 복호화에 동일한 키 하나를 사용하는 방식 |
| **비대칭 암호화 (Asymmetric Encryption)** | 공개 키와 개인 키 쌍을 사용하는 방식 |
| **공개 키 (Public Key)** | 누구나 사용할 수 있는 키. 공개해도 안전함 |
| **개인 키 (Private Key)** | 소유자만 비밀로 보관하는 키. 절대 공유하지 않음 |
| **키 분배 문제 (Key Distribution Problem)** | 대칭 암호화에서 초기에 키를 안전하게 공유하는 방법의 문제 |
| **인증서 (Certificate)** | 공개 키와 소유자 정보를 CA가 서명한 디지털 문서 |
| **CA (인증 기관)** | 인증서에 서명해 공개 키의 신원을 보증하는 신뢰된 제3자 기관 |
| **AES** | Advanced Encryption Standard. 현대 표준 대칭 암호화 알고리즘 |
| **HTTPS** | HTTP + TLS(비대칭+대칭 하이브리드). 자물쇠 아이콘으로 표시됨 |

## 실습 기록 — 암호학 개념

### 실습 1 — Task 2: 시저 암호 게임 (비밀 메시지 복구)

| 문제 | 정답 |
|------|------|
| 비밀 메시지 복구 게임 완료 플래그 | `THM{CAESAR_CIPHER_MASTER_2026}` |
| 키 5로 `CYBER` 를 암호화하면? | `HDGJW` |
| 다음 암호문을 해독하라: `FVZCYR PNRFNE PVCURE` | `SIMPLE CAESAR CIPHER` |

**CYBER + Key 5 = HDGJW 계산 과정 (A=0 기준)**:

```
C(2)  + 5 = 7  = H
Y(24) + 5 = 29 → 29-26 = 3 = D  ← 알파벳 끝을 넘으면 처음으로 순환
B(1)  + 5 = 6  = G
E(4)  + 5 = 9  = J
R(17) + 5 = 22 = W
→ HDGJW ✓
```

**FVZCYR PNRFNE PVCURE 해독 (Key = 13, ROT13)**:

```
F(5) -13 = -8+26 = 18 = S    P(15)-13 = 2  = C    P(15)-13 = 2  = C
V(21)-13 = 8  = I            N(13)-13 = 0  = A    V(21)-13 = 8  = I
Z(25)-13 = 12 = M            R(17)-13 = 4  = E    C(2) -13 =-11+26=15=P
C(2) -13 =-11+26=15=P        F(5) -13 =-8+26=18=S  U(20)-13 = 7  = H
Y(24)-13 = 11 = L            N(13)-13 = 0  = A    R(17)-13 = 4  = E
R(17)-13 = 4  = E            R(17)-13 = 4  = E    R(17)-13 = 4  = E

→ SIMPLE  CAESAR  CIPHER ✓
```

**배운 점**

1. 시저 암호에서 알파벳 끝(Z)을 넘어가면 다시 A로 돌아오는 순환 구조는 **모듈러 연산(Modular Arithmetic)** 이다. `(위치 + 키) % 26` 이라는 수식 하나가 시저 암호 전체를 표현한다. 현대 암호화(AES, RSA 등)도 훨씬 복잡하지만 같은 종류의 수학 원리를 기반으로 동작한다. 암호학이 결국 수학이라는 것을 보여주는 가장 간단한 예다.
2. `FVZCYR PNRFNE PVCURE` 를 해독한 키는 13(ROT13)이다. ROT13은 영어 알파벳 26개의 정중앙인 13을 이동값으로 사용하기 때문에 암호화와 복호화가 완전히 동일한 연산이 된다는 특성이 있다. 실제 보안 용도로는 전혀 사용하지 않지만, 인터넷에서 스포일러를 가리거나 포럼에서 민감한 내용을 숨기는 관습적 용도로 여전히 사용된다.
3. 시저 암호의 진짜 교훈은 **키 공간(Key Space)의 크기가 보안 강도를 결정한다**는 것이다. 시저 암호는 키가 25가지뿐이라 컴퓨터로 1ms 만에 전수 탐색 가능하다. 반면 AES-256은 키 공간이 `2^256`가지로, 지구상 모든 컴퓨터가 우주 나이보다 오래 계산해도 찾을 수 없다. 알고리즘이 공개되어 있어도 키 공간이 충분히 크면 안전하다는 원칙이 현대 암호학의 기반이다.

### 실습 2 — Task 3: 비대칭 암호화 개념 퀴즈

| 질문 | 정답 |
|------|------|
| 비대칭 암호화에서 절대 공개하면 안 되는 키는? | `private key` |
| 암호화와 복호화를 제어하는 비밀 정보를 무엇이라 하는가? | `key` |
| 대칭 암호화를 단독으로 사용할 때의 근본적인 문제는? | `key distribution` |
| 공개 키와 개인 키 쌍을 사용하는 암호화 유형은? | `asymmetric` |

**배운 점**

1. 비대칭 암호화에서 **공개 키를 공개해도 안전하다**는 것은 직관에 반하는 가장 중요한 개념이다. 암호화에 쓰는 키(공개 키)와 복호화에 쓰는 키(개인 키)를 수학적으로 분리했기 때문이다. 이 아이디어 덕분에 인터넷에서 처음 만나는 두 사람도 사전에 비밀을 공유하지 않고 안전하게 통신할 수 있다. 1976년 Diffie와 Hellman이 이 개념을 발표했고, 이것이 현대 인터넷 보안 전체의 토대가 됐다.
2. HTTPS에서 비대칭 암호화가 **모든 데이터를 직접 암호화하지 않는 이유는 속도 때문**이다. 비대칭 암호화는 복잡한 수학 연산 때문에 대칭 암호화보다 수십~수백 배 느리다. 그래서 비대칭 암호화는 "공유 키를 협의하는" 짧은 초기 단계에만 사용하고, 실제 데이터는 빠른 대칭 암호화로 처리한다. 보안과 성능을 동시에 달성하기 위한 역할 분담 설계다.
3. CA(인증 기관)가 없다면 비대칭 암호화도 **중간자 공격(Man-in-the-Middle Attack)** 에 취약하다. 공격자가 밥인 척하며 가짜 공개 키를 앨리스에게 보내면, 앨리스는 밥과 통신하고 있다고 믿지만 실제로는 공격자와 통신하는 셈이다. CA는 "이 공개 키가 진짜 이 도메인의 것"임을 보증하는 신뢰된 제3자 역할을 한다. 브라우저의 자물쇠 아이콘이 단순 암호화가 아닌 **신원 검증까지 완료**됐다는 의미인 이유다.

## 섹션 12 — 공격적 보안이란?

**공격적 보안 (Offensive Security)** = 실제 공격자가 시스템을 악용하기 전에 취약점을 먼저 찾아내기 위해, 선제적으로 시스템에 침입을 시도하는 보안 접근 방식

앞서 배운 CIA 트라이어드와 암호학이 **방어적** 관점이었다면, 공격적 보안은 **공격자의 관점**으로 먼저 약점을 찾아내는 데 중점을 둔다. 두 접근 방식은 서로 보완하며 완전한 보안 체계를 이룬다.

공격적 보안은 다음 세 가지 질문으로 시작한다:

| 질문 | 의미 |
|------|------|
| 무엇이 노출되어 있는가? | 외부에서 접근 가능한 포트, 서비스, 페이지는? |
| 무엇에 접근할 수 있는가? | 권한 없이 접근 가능한 리소스는? |
| 시스템은 어떤 가정을 하고 있는가? | 설계자가 놓친 전제 조건이나 취약한 가정은? |

해커는 이 질문들을 체계적으로 적용해 시스템이 예상치 못한 입력에 어떻게 반응하는지 관찰한다.

### 침투 테스트 (Penetration Testing) — "해킹"의 윤리적 형태

"해킹"이라는 단어는 부정적으로 들리지만, 보안 업계에서 허가된 해킹은 **침투 테스트(Penetration Testing)** 다.

> **TryHackMe 번역 오류 주의**: TryHackMe 한국어 번역에서 "Penetration Testing"을 **"심두 테스트"** 로 번역한다. 보안 업계 표준 한국어 용어는 **침투 테스트** 또는 **모의 해킹**이다. "심두"는 의미 없는 번역이다.

| 특성 | 설명 |
|------|------|
| **윤리적 (Ethical)** | 반드시 사전 허가(범위 합의)를 받고 수행 |
| **합법적 (Legal)** | 허가된 시스템 범위 안에서만 테스트 수행 |
| **체계적 (Systematic)** | 취약점을 식별하고 보고하는 방법론 사용 |

해커(침투 테스터)는 이 기술로 취약점을 먼저 발견해 조직에 보고하고, 보안 개선을 돕는다. **공격 기법을 이해하지 않으면 효과적으로 방어할 수 없다.**

### 학습 목표

- 공격적 보안이란 무엇이며 시스템 보안 강화를 위해 왜 사용되는지 설명
- 공격적 보안에 사용되는 일반적인 용어와 방법론을 이해
- 안전하고 허가된 환경에서 윤리적 해킹 기술을 직접 연습
- 공격적 보안 여정을 위한 다음 단계와 학습 경로 파악

## 섹션 14 — 디렉토리 열거: gobuster

### 시나리오

대상: 온라인 쇼핑몰 `http://www.onlineshop.thm`

웹사이트 메인 페이지에는 블루투스 스피커, 헤드폰 등 상품 목록이 보인다. 그런데 개발자가 일반 탐색 메뉴에는 연결하지 않았지만 실제로는 존재하는 페이지가 있을 수 있다. 예를 들어 어드민 패널, 백엔드 설정 페이지 같은 것들이다. 이런 숨겨진 경로를 찾아내는 것이 **열거(Enumeration)** 다.

### gobuster란?

**gobuster** = 웹 서버에서 숨겨진 디렉토리·파일을 단어 목록으로 빠르게 탐색하는 CLI 도구

작동 원리:

```
wordlist에서 단어 하나씩 꺼내기
  → http://대상/[단어] 로 HTTP 요청 전송
  → 서버 응답 확인:
      200 OK  → 경로 존재! ✅ 목록에 추가
      404 Not Found → 경로 없음 ❌ 건너뜀
  → 다음 단어로 반복
```

### 실행 명령어

```bash
gobuster dir -url http://www.onlineshop.thm/ -w /usr/share/wordlists/dirbuster/directory-list.txt
```

**명령어 분해**:

| 부분 | 의미 |
|------|------|

| `-w /usr/share/wordlists/dirbuster/directory-list.txt` | 사용할 단어 목록 파일 경로 |

## 섹션 15 — 해커처럼 생각하기

### 해커의 사고방식

해커는 시스템 설계자가 **의도하지 않은 방식**으로 시스템을 사용하는 방법을 찾는다.

```
[설계자의 가정]
/admin 페이지는 관리자만 안다
  → 관리자만 로그인할 것이다
  → 비밀번호는 안전할 것이다

[해커의 질문]
비밀번호가 짧거나 흔한 단어라면?
  → 사전 공격으로 추측할 수 있다!
```

### 윤리적 해킹의 방법론

실제 침투 테스터가 따르는 단계:

| 단계 | 내용 |
|------|------|
| **범위 확인** | 테스트 허가를 받은 시스템과 작업 범위를 먼저 명확히 정의 |
| **정보 수집 및 열거** | gobuster 같은 도구로 대상 시스템의 구조와 세부 정보 파악 |
| **취약점 확인 및 공격** | 발견한 정보를 바탕으로 실제 취약점을 확인하고 익스플로잇 시도 |
| **결과 보고** | 발견한 취약점과 공격 경로를 조직에 체계적으로 보고 |
| **책임감 있는 공개** | 취약점을 시스템 소유자에게만 보고하고 악용하지 않음 |

> **가장 중요한 원칙**: 허가(Scope) 없는 침투 테스트는 범죄다. 아무리 보안 연구 목적이라도 허가받지 않은 시스템에 이 기술을 사용하면 법적 처벌을 받는다.

## 섹션 16 — 브루트포스 공격: Hydra

### 사전 공격(Dictionary Attack)이란?

**사전 공격** = 미리 만들어 둔 비밀번호 목록(wordlist)을 하나씩 자동으로 시도해 올바른 비밀번호를 찾아내는 공격

| 방법 | 소요 시간 |
|------|----------|
| 사람이 직접 수동 시도 (100만 개 비밀번호) | 수십 년 |
| 컴퓨터로 자동화 (Hydra 사용) | 수 분 ~ 수 시간 |

비밀번호 목록(wordlist)에는 `123456`, `password`, `admin`, `bee` 처럼 사람들이 자주 쓰는 비밀번호 수천~수백만 개가 담겨 있다.

### Hydra란?

**Hydra** = HTTP 로그인 양식, SSH, FTP 등 다양한 프로토콜에 대해 사전 공격을 자동으로 수행하는 브루트포스 도구

작동 원리:

```
wordlist에서 비밀번호 하나씩 꺼내기
  → 로그인 양식에 ID=admin, PW=[비밀번호] 자동 입력 및 제출
  → 응답에 실패 메시지("incorrect") 있으면 → 다음 비밀번호
  → 실패 메시지 없으면 → 로그인 성공! 결과 출력
```

### 실행 명령어

```bash
hydra -l admin -P /pencilList.lst www.onlineshop.thm http-form-post "/login?username=^USER^&password=^PASS^:F=incorrect" -v
```

**명령어 분해**:

| 옵션 | 의미 |
|------|------|
| `-l admin` | 사용자 이름을 `admin` 으로 고정 |
| `-P /pencilList.lst` | 시도할 비밀번호가 담긴 단어 목록 파일 |
| `www.onlineshop.thm` | 공격 대상 호스트 |
| `http-form-post` | HTTP POST 방식의 로그인 양식 공격 모드 |
| `"/login?username=^USER^&password=^PASS^:F=incorrect"` | 로그인 URL 경로, `^USER^`와 `^PASS^`는 Hydra가 채우는 자리. `:F=incorrect` = 응답에 "incorrect"가 있으면 실패로 간주 |
| `-v` | 진행 상황을 상세히 출력 (verbose) |

### 실행 결과

Hydra가 `pencilList.lst` 의 단어들을 빠르게 시도한 결과, **17번째 시도**에서 올바른 비밀번호를 발견했다:

```
[ATTEMPT] target www.onlineshop.thm - login "admin" - pass "1234"     ← INCORRECT
[ATTEMPT] target www.onlineshop.thm - login "admin" - pass "guest"    ← INCORRECT
[ATTEMPT] target www.onlineshop.thm - login "admin" - pass "password" ← INCORRECT
...
[ATTEMPT] target www.onlineshop.thm - login "admin" - pass "bee"      ← SUCCESS!

[800][http-form-post] host: www.onlineshop.thm   login: admin   password: bee
1 of 1 target successfully completed, 1 valid password found
```

**발견된 자격 증명**:

| 항목 | 값 |
|------|-----|
| 사용자 이름 | `admin` |
| 비밀번호 | `bee` |
| 총 시도 횟수 | 17번 |

→ `http://www.onlineshop.thm/admin` 에 `admin / bee` 로 로그인 → 어드민 패널 접근 → 플래그 획득

## 주요 용어 정리 (공격적 보안)

| 용어 | 정의 |
|------|------|
| **공격적 보안 (Offensive Security)** | 선제적으로 취약점을 찾아 시스템을 보호하는 보안 접근 방식 |
| **침투 테스트 (Penetration Testing)** | 허가를 받고 시스템에 침입을 시도해 취약점을 찾는 체계적 활동. "모의 해킹"이라고도 함 |
| **범위 (Scope)** | 보안 테스트에서 허용되는 시스템과 작업의 경계. 범위 밖 시스템 테스트는 범죄 |
| **취약점 (Vulnerability)** | 공격자가 이용할 수 있는 시스템의 숨겨진 약점 |
| **익스플로잇 (Exploit)** | 취약점을 실제로 악용하는 방법 또는 공격 코드 |
| **열거 (Enumeration)** | 대상 시스템에 대한 세부 정보를 수집하는 과정 |
| **자격 증명 (Credentials)** | 접근 권한 확인에 사용되는 아이디와 비밀번호 등의 로그인 정보 |
| **인증 (Authentication)** | 사용자가 주장하는 신원과 실제 신원이 일치하는지 확인하는 과정 |
| **사전 공격 (Dictionary Attack)** | 단어 목록으로 비밀번호를 하나씩 자동으로 시도하는 공격 |
| **gobuster** | 웹 서버의 숨겨진 디렉토리와 파일을 탐색하는 열거 도구 |
| **Hydra** | 로그인 양식에 사전 공격을 자동으로 수행하는 브루트포스 도구 |
| **wordlist (단어 목록)** | 사전 공격에 사용하는 가능한 비밀번호 또는 경로의 목록 파일 |

## 실습 기록 — 공격적 보안

### 실습 1 — Task 2: gobuster로 숨겨진 디렉토리 열거

**대상**: `http://www.onlineshop.thm`

**사용 명령어**:
```bash
gobuster dir -url http://www.onlineshop.thm/ -w /usr/share/wordlists/dirbuster/directory-list.txt
```

**발견된 경로**:

| 경로 | HTTP 상태 | 내용 |
|------|----------|------|
| `/items` | 200 | 상품 목록 페이지 |
| `/sitemap` | 200 | 사이트맵 |
| `/login` | 200 | 로그인 페이지 |
| `/admin` | 200 | **관리자 패널 (숨겨진 핵심 경로!)** |

**배운 점**

1. gobuster가 `/admin` 을 발견한 것이 핵심이다. 이 경로는 웹사이트 어디에도 링크가 없지만 실제로 서버에 존재한다. **"숨겨놓는 것"은 보안이 아니다.** 공격자는 gobuster 같은 도구로 숨겨진 경로를 단 몇 분 안에 모두 찾아낼 수 있다. 이를 **"보안을 통한 모호성(Security through Obscurity)"** 이라 하며, 보안 업계에서는 이를 진짜 보안으로 인정하지 않는다.
2. gobuster는 단어 목록에 있는 수만 개의 경로를 수 분 이내에 전부 시도한다. 사람이 브라우저로 직접 추측하는 것과 비교하면 수만 배 빠르다. 이것이 **자동화 도구**가 공격자에게 강력한 무기인 이유다. 방어 측에서는 자동화된 스캐닝을 탐지하는 WAF(Web Application Firewall), Rate Limiting, 접근 로그 모니터링 등이 필요하다.
3. 침투 테스트에서 각 단계는 다음 단계의 기반이 된다. gobuster가 `/admin` 을 찾지 못했다면 Hydra 실습도 없었다. **열거(Enumeration) → 취약점 식별 → 익스플로잇 → 접근 획득**이 공격적 보안의 기본 흐름이다. 첫 단계인 열거를 철저히 할수록 이후 단계가 명확해진다.

### 실습 2 — Task 3: Hydra로 관리자 비밀번호 크랙

**목표**: `http://www.onlineshop.thm/admin` 로그인 페이지의 admin 계정 비밀번호를 사전 공격으로 알아내기

**사용 명령어**:
```bash
hydra -l admin -P /pencilList.lst www.onlineshop.thm http-form-post "/login?username=^USER^&password=^PASS^:F=incorrect" -v
```

**결과**:

| 항목 | 값 |
|------|-----|
| 발견된 사용자 이름 | `admin` |
| 발견된 비밀번호 | `bee` |
| 총 시도 횟수 | 17번 |

→ `admin / bee` 로 어드민 패널 로그인 성공 → 플래그 획득

**배운 점**

1. 비밀번호 `bee` 는 단 17번 시도 만에 발견됐다. **약한 비밀번호의 위험성**을 직접 확인한 것이다. 비밀번호가 짧고 단순한 단어라면 암호화되어 저장되어 있어도 사전 공격에 극도로 취약하다. 방어 관점에서 강력한 비밀번호 정책(길이 12자 이상, 특수문자 포함, 흔한 단어 금지)과 로그인 시도 횟수 제한(Rate Limiting, Account Lockout)이 반드시 필요한 이유다.
2. Hydra 명령어에서 `:F=incorrect` 부분이 핵심이다. "응답에 `incorrect`라는 문자열이 있으면 실패로 간주하라"는 의미다. Hydra는 응답 내용으로 성공/실패를 구분하기 때문에, 로그인 실패 메시지가 애매하거나 없으면 오작동한다. 반대로 방어 측에서는 **로그인 실패 메시지를 최대한 불특정하게** 만드는 것이 공격자가 자동화 도구를 사용하기 어렵게 만드는 방법이다.
3. 가장 중요한 교훈: **이 실습은 반드시 허가된 환경에서만 수행해야 한다.** TryHackMe는 학습용으로 격리된 가상 환경을 제공하며 모든 공격 대상은 사전 허가된 시스템이다. 동일한 기술을 실제 웹사이트에 사용하면 컴퓨터 부정사용 방지법 위반으로 형사 처벌을 받는다. **범위(Scope)와 허가(Permission)** 가 윤리적 해킹과 사이버 범죄를 구분하는 단 하나의 기준이다.

## 섹션 18 — 방어적 보안이란?

**방어적 보안 (Defensive Security)** = 보호해야 할 대상을 파악하고 잠재적 공격을 예방, 탐지 및 완화하기 위한 보안 조치를 구현하는 데 중점을 두는 보안 접근 방식

이전 섹션에서 배운 **공격적 보안(Offensive Security)** 이 공격자의 관점으로 취약점을 먼저 찾아내는 것이었다면, **방어적 보안**은 그 취약점으로부터 시스템을 지키는 것이다. 두 접근 방식은 서로 보완하며 완전한 보안 체계를 이룬다.

방어 담당자(수비수)의 목표:
- 시스템에 대한 **가시성(Visibility)** 확보 — 무엇이 있는지, 무슨 일이 일어나는지 파악
- 취약점 식별 및 **사전 예방** 조치 수행
- 시스템의 **기밀성·무결성·가용성(CIA 트라이어드)** 유지
- 사건 발생에 **대비**하고, 발생 시 **신속하게 대응**

> **TryHackMe 번역 오류**: 본문에서 CIA 트라이어드의 두 번째 요소를 **"진실성"** 으로 번역한다. 올바른 한국어 용어는 **무결성(Integrity)** 이다. "삼위일체" 역시 **CIA 트라이어드**로 불러야 한다.

**블루팀(Blue Team)** = 방어 전담 보안 팀
- 공격자의 사고방식, 공격 대상, 공격 전개 방식을 이해하고 방어책 적용
- 핵심 기반 시설을 파악하고 공격자의 목표를 예측하여 선제적으로 보호

### 학습 목표

- 수비수가 무엇을 보호하는지, 가시성이 왜 중요한지 이해
- 고객 환경의 핵심 요소(인프라 구성요소) 파악
- 일반적인 시스템과 서비스를 보호하는 기본적인 방법 학습
- 자신감을 갖고 방어적 보안 여정 시작

## 섹션 19 — 고객 인프라 이해: 도시 비유

방어자가 시스템을 보호하려면 먼저 **존재하는 모든 것과 그 연결 방식을 파악**해야 한다. TryHackMe는 이를 직관적으로 이해하기 위해 **도시 비유**를 사용한다.

- 고객의 인프라 = 번화한 도시
- 방어자 = 도시 경비원 (도시 구조를 알고, 문제를 감시하고, 대응 방법을 앎)

### 방어적 질문 → 도시 비유 → 보안 동등성

| 방어적 질문 | 도시 비유 | 보안 동등성 |
|-----------|---------|-----------|
| 무엇을 보호하는가? | 집, 건물, 사람들 | 클라이언트 서버, 데이터, 워크스테이션, 사용자 |
| 보호 대상이 보이는가? (가시성) | 카메라, 보고서, 순찰 | 로그, 네트워크 트래픽, 알림 |
| 의심스러운 행동의 기준은? | 잠긴 문 열기 시도, 차들이 주변을 맴돔 | 반복적인 로그인 시도, 비정상적인 IP 주소 |
| 위협을 어떻게 막는가? | 경찰, 도로 봉쇄, 통행금지 | 방화벽 규칙, IP 주소 차단 |

### 고객 인프라 구성 요소

방어자가 보호해야 할 범위: 조직 또는 고객에게 속한 기기, 서버, 네트워크

| 시스템/인프라 구성 요소 | 목적 | 도시 비유 |
|----------------------|------|---------|
| **직원용 기기 (Employee Devices)** | 사용자들이 업무를 보고 회사 리소스에 접근하는 곳 | 주택 |
| **웹 서버 (Web Server)** | 사용자가 접속하는 웹사이트 또는 애플리케이션 호스팅 | 상점/공공 건물 |
| **메일 서버 (Mail Server)** | 조직을 대신하여 이메일을 송수신 | 우체국 |
| **방화벽 (Firewall)** | 출입 가능한 트래픽을 제어 | 성문 |
| **인터넷 (Internet)** | 조직에서 관리하지 않는 외부 네트워크 | 도시 외곽의 모든 곳 |

```
[인프라 구조 다이어그램]

         인터넷 (도시 외부 = 모든 외부 위협의 출처)
                      ↓
               방화벽 (성문 = 외부와 내부의 경계)
                      ↓
    ┌─────────────────────────────────────┐
    │  직원용 기기    웹 서버    메일 서버  │
    │   (주택)     (상점/건물)   (우체국)  │
    └─────────────────────────────────────┘
                (조직 내부 인프라)
```

## 섹션 20 — 수비수의 5가지 행동 방식

방어자는 5가지 핵심 행동으로 시스템을 보호한다:

| 행동 | 영문 | 설명 | 예시 |
|------|------|------|------|
| **예방** | Prevention | 공격이 발생하기 전에 차단하는 보안 조치 마련 | 방화벽 설치, 바이러스 백신 운영, 정기적인 패치 적용 |
| **탐지** | Detection | 의심스럽거나 악의적인 활동을 식별하기 위해 모니터링 | 로그 분석, 경고 시스템, 보안 모니터링 도구 운영 |
| **완화 조치** | Mitigation | 사고 발생 시 피해를 최소화하기 위한 즉각적 행동 | 트래픽 차단, 피해 시스템 격리, 손상된 계정 비활성화 |
| **분석** | Analysis | 사건 경위와 영향을 파악하기 위해 로그와 증거 검토 | 어떤 시스템이 영향을 받았는지, 어떻게 침입했는지 조사 |
| **대응 및 개선** | Response & Improvement | 사건 복구 후 향후 유사 공격을 막기 위한 방어 체계 개선 | 보안 정책 업데이트, 추가 패치 적용, 직원 보안 교육 |

## 섹션 21 — 수비수 마인드셋

### 시스템을 사슬로 바라봐야 하는 이유

공격자는 드물게 단일 시스템만 목표로 삼는다. **하나의 자산을 침해한 후 다음 자산으로 이동**하며 최종 목표를 향해 나아간다. 방어자는 시스템을 개별 부분이 아닌 **서로 연결된 사슬**로 인식해야 한다.

**실제 공격 경로 예시**:

```
[1단계] 악성 이메일 첨부 파일 발송
              ↓ 직원이 클릭
[2단계] 직원 워크스테이션(Employee Devices) 악성코드 감염
              ↓ 사용자 이름 + 비밀번호 탈취
[3단계] 메일 서버(Mail Server) 무단 접근
              ↓ 내부 네트워크 이동 (Lateral Movement)
[4단계] 데이터베이스 서버 접근
              ↓
[최종] 민감한 데이터 유출 또는 랜섬웨어 피해
```

각 단계는 이전 단계를 기반으로 구축된다. 방어자는 이 사슬의 **각 단계마다** 보안 조치를 적용해 공격을 차단한다. 사슬의 어느 한 고리를 끊으면 공격이 멈춘다.

## 섹션 22 — 수비의 핵심 원칙 4가지

| 원칙 | 영문 | 내용 |
|------|------|------|
| **위협 예측** | Threat Prediction | 보호하려는 시스템을 검토하고 "만약에?" 질문으로 공격자가 취할 수 있는 현실적인 경로를 상상 |
| **공격 패턴 파악** | Attack Pattern Recognition | 공격은 일반적으로 예측 가능한 단계를 거침. 일반적인 공격 경로와 구조 연구가 방어자에게 유용 |
| **위험 우선순위 설정** | Risk Prioritization | 시스템의 모든 부분이 동일한 위험을 내포하지 않음. 가치가 높은 시스템과 목표물을 먼저 식별하고 집중 보호 |
| **지속적인 적응** | Continuous Adaptation | 방어는 일회성 설정이 아님. 위협과 공격자는 진화하고, 기술은 변화하며, 새로운 취약점이 지속적으로 나타남 |

> **Risk Prioritization (위험 우선순위 설정)** 이 Task 3 퀴즈 정답이다. "보안 노력과 집중 방향을 제시하기 위해 가장 중요한 시스템을 식별하는 원칙"이 무엇인지 물어보는 문제의 정답이다.

## 섹션 23 — 인프라 구성요소별 방어 수단

인프라 각 구성 요소별로 발생할 수 있는 위협과 그에 대응하는 방어 수단:

| 시스템/인프라 구성 요소 | 발생 가능한 위협 | 사용할 방어 수단 |
|----------------------|---------------|---------------|
| **직원용 기기** | 악성 링크 클릭, 안전하지 않은 소프트웨어 다운로드 | **바이러스 백신 (Antivirus)** + **정기적인 소프트웨어 업데이트** |
| **웹 서버** | 공격자의 웹사이트 침입 시도 | **신뢰된 트래픽만 허용 (Allow Trusted Traffic)** + **안전한 통신(HTTPS) 사용** |
| **메일 서버** | 악의적이거나 기만적인 이메일 (피싱, Phishing) | **스팸 필터 (Spam Filter)** + **첨부파일 검사 (Scan Attachments)** |
| **방화벽** | 인터넷에서 침입을 시도하는 외부 공격자 | **접근 제어 규칙** + **알려진 악성 IP 주소 차단 (Block Known Bad IPs)** |
| **외부 인터넷** | 외부에서 오는 모든 위협의 출발점 | **유입 트래픽 제한** + **의심스러운 활동 모니터링 (Monitor Traffic)** |

**다층 방어 (Defense in Depth)**: 단 하나의 도구로 모든 공격자를 막을 수 없다. 여러 겹의 보안 시스템을 구축하면 공격자가 성공하기 훨씬 어려워진다. 방어의 핵심은 여러 보안 조치를 **겹겹이 쌓는 것**이다.

## 섹션 24 — 방어적 보안 커리어 경로

| 직무 | 설명 |
|------|------|
| **보안 운영 센터 (SOC, Security Operations Center)** | 네트워크와 시스템을 모니터링하여 의심스러운 활동이나 보안 경고를 감지하고 조사. 24/7 실시간 대응 |
| **위협 인텔리전스 (Threat Intelligence)** | 현재의 위협, 공격자 및 그 동향을 조사하여 잠재적인 공격에 대비하고 방어 지원 |
| **디지털 포렌식 및 사고 대응 (DFIR, Digital Forensics & Incident Response)** | 보안 사고를 조사하여 공격 발생 경위를 파악하고, 위협을 차단하여 영향을 받은 시스템을 복구 |

## 주요 용어 정리 (방어적 보안)

| 용어 | 정의 |
|------|------|
| **블루팀 (Blue Team)** | 시스템을 보호하고 위협에 대응하는 임무를 맡은 사이버 보안 방어 집단 |
| **클라이언트 인프라 (Client Infrastructure)** | 보호가 필요한 조직 소유의 네트워크, 서버, 장치 및 애플리케이션 전체 |
| **가시성 (Visibility)** | 잠재적인 문제를 발견하기 위해 시스템 전반의 활동을 확인하고 모니터링할 수 있는 능력 |
| **위협 (Threat)** | 해커나 악성 소프트웨어처럼 시스템이나 데이터에 피해를 줄 수 있는 잠재적 위험 |
| **예방 (Prevention)** | 위협이 피해를 입히기 전에 공격 기회를 차단, 제한 또는 늦추기 위한 조치 |
| **탐지 (Detection)** | 네트워크 및 시스템에서 위협의 의심스러운 활동을 식별하는 과정 |
| **완화 (Mitigation)** | 위협이 식별된 후 그 영향을 줄이거나 막기 위해 취하는 조치 |
| **위험 (Risk)** | 위협이 조직에 실제로 피해를 입힐 가능성과 그 잠재적 영향 |

### 실습 1 — Task 2: 도시 지도 만들기 (클라이언트 인프라 매핑)

**과제**: "Defending Your City" — 클라이언트 인프라를 나타내는 도시 지도에서 각 구성 요소를 올바른 위치에 드래그하여 배치

**배치 결과** (5/5 전체 정답, ✓):

| 구성 요소 | 배치 위치 | 도시 비유 |
|----------|---------|---------|
| Internet | 도시 최외각 (최상단) | 도시 외곽 |
| Employee Devices | 내부 좌측 | 주택가 |
| Web Server | 내부 좌하 | 상점/공공 건물 |
| Mail Server | 내부 우측 | 우체국 |
| Firewall | 입구 중앙 하단 | 성문 |

**퀴즈 정답**:

| 질문 | 정답 |
|------|------|
| 방어자가 피해 발생 전에 위협을 차단하기 위해 보안 제어를 구축하는 목적은? | `Prevention` |
| 사건 발생 경위와 영향을 파악하기 위해 로그와 증거를 검토하는 과정은? | `Analysis` |
| 도시 기반 시설 지도를 성공적으로 작성한 후 받은 깃발은? | `THM{mapping_infrastructure!}` |

**배운 점**

1. 인프라 구성 요소의 **위치 관계**가 핵심이다. 인터넷 → 방화벽 → 내부 자산 순서가 곧 **트래픽의 흐름**이며, 방화벽이 외부와 내부 사이 경계에서 문지기 역할을 한다. 이 구조를 이해하면 "방화벽 하나만 있으면 안전하지 않나?"라는 질문의 답도 나온다 — 내부 자산마다 각각의 추가 보호가 필요하기 때문이다.
2. **가시성(Visibility)이 방어의 출발점**이다. 무엇이 존재하는지조차 모르면 보호할 수 없다. 도시 지도 만들기 실습이 바로 이 가시성 확보 과정, 즉 **자산 목록(Asset Inventory) 파악**을 시뮬레이션한 것이다. 실제 보안 업무에서 자산 목록 파악이 항상 첫 번째 단계인 이유가 여기에 있다.
3. 각 구성 요소의 **목적과 역할**을 이해해야 적절한 보호 조치를 선택할 수 있다. 웹 서버는 외부 사용자와 직접 통신하므로 가장 노출된 자산이고, 메일 서버는 피싱 공격의 주요 진입점이며, 직원 기기는 내부 네트워크로 연결되는 고리다. 역할을 알면 어떤 위협이 들어올지 예측할 수 있다.

### 실습 2 — Task 3: 도시 방어하기 (보안 제어 드래그앤드롭)

**과제**: "Defending Your City" — 도시의 각 구성 요소에 적절한 보안 제어를 드래그하여 배치

**정답 매핑** (5/5 전체 정답, ✓):

| 구성 요소 | 배치한 보안 제어 |
|----------|--------------|
| Employee Devices | **Antivirus and Software Updates** (바이러스 백신 + 소프트웨어 업데이트) |
| Web Server | **Allow Trusted Traffic and Use Secure Communication** (신뢰된 트래픽 허용 + 보안 통신) |
| Mail Server | **Spam Filters and Scan Attachments** (스팸 필터 + 첨부파일 검사) |
| Firewall | **Block Known Bad IP Addresses** (알려진 악성 IP 차단) |
| Internet | **Monitor Traffic for Suspicious Activity** (의심스러운 활동 트래픽 모니터링) |

**퀴즈 정답**:

| 질문 | 정답 |
|------|------|
| 가장 중요한 시스템을 식별하는 데 중점을 두는 방어 원칙은? | `Risk Prioritization` |
| 도시 기반 시설을 성공적으로 방어한 후 받은 깃발은? | `THM{defensive_techniques!}` |

**배운 점**

1. 각 보안 제어가 **왜** 그 구성 요소에 배치되는지 논리를 이해하는 것이 중요하다. 직원 기기에 바이러스 백신이 필요한 이유는 악성 파일이 가장 많이 실행되는 곳이기 때문이고, 메일 서버에 스팸 필터가 필요한 이유는 피싱 이메일의 주요 진입점이기 때문이다. 단순 암기가 아니라 **위협 → 방어 수단**의 논리적 연결을 이해해야 한다.
2. **방화벽의 역할은 "알려진 나쁜 것"을 차단**하는 것이다. 방화벽은 모든 악성 트래픽을 자동으로 막는 마법의 도구가 아니라, 규칙에 기반해 특정 IP나 포트를 차단하는 필터다. 따라서 악성 IP 목록 업데이트 등 **지속적인 관리**가 필요하다. 섹션 22의 "지속적인 적응(Continuous Adaptation)" 원칙이 바로 이것을 설명한다.
3. **Risk Prioritization(위험 우선순위 설정)이 핵심 원칙인 이유**: 보안 자원은 유한하다. 모든 시스템을 동일하게 보호할 수 없으므로 가장 중요하고 위험한 시스템을 먼저 집중 보호해야 한다. 예를 들어 외부 인터넷과 직접 연결된 웹 서버와 방화벽이 내부 프린터보다 훨씬 높은 우선순위를 갖는다. 이 원칙 없이는 낮은 위험의 시스템에 자원을 낭비하고 정작 중요한 시스템이 무방비 상태가 될 수 있다.
