## CIA 트라이어드 (The CIA Triad)

---

## 섹션 1 — CIA 트라이어드란?

**CIA 트라이어드 (CIA Triad)** = 사이버 보안이 보호해야 하는 세 가지 핵심 원칙을 묶은 보안 사고방식

> 이름이 비슷해 헷갈릴 수 있는데, 여기서 CIA는 미국 중앙정보국(Central Intelligence Agency)이 아니다. **Confidentiality, Integrity, Availability**의 앞 글자를 딴 보안 전문 용어다.

과거에는 정보가 종이 문서로 저장됐지만, 오늘날은 디지털 시스템에 보관되고 네트워크를 통해 전송된다. 적절한 보안 조치가 없으면 이 데이터는:

- **잘못된 사람에게 노출**되거나 (기밀성 침해)
- **허가 없이 수정**되거나 (무결성 침해)
- **가장 필요할 때 이용 불가능**해질 수 있다 (가용성 침해)

이 세 가지 위협에 대응하는 원칙이 바로 **CIA 트라이어드**다. 사이버 보안에서 접하는 거의 모든 개념은 이 세 가지를 중심으로 전개된다.

```
          기밀성
       Confidentiality
           △
          / \
         /   \
        /     \
  무결성 ———— 가용성
 Integrity  Availability
```

| 원칙 | 영문 | 핵심 질문 | 한 줄 정의 |
|------|------|----------|------------|
| **기밀성** | Confidentiality | 누가 볼 수 있는가? | 승인된 사람만 데이터에 접근할 수 있도록 보장 |
| **무결성** | Integrity | 누가 바꿀 수 있는가? | 권한 없는 사람이 데이터를 수정하지 못하도록 보장 |
| **가용성** | Availability | 언제 쓸 수 있는가? | 권한 있는 사용자가 필요할 때 데이터와 서비스를 이용할 수 있도록 보장 |

> **TryHackMe 번역 오류 주의**
> TryHackMe 한국어 번역에서 Integrity를 **"진실성"**, Availability를 **"유효성"** 으로 번역한다.
> 두 표현 모두 잘못된 번역이다. 보안 업계 표준 한국어 용어는 각각 **무결성**과 **가용성**이다.
> - "진실성(眞實性)" = 사실을 말한다는 의미 → 데이터 보안 문맥과 의미가 다름
> - "무결성(無缺性)" = 결함이나 변조가 없는 상태 → 보안 문맥에서 정확한 표현
> - "유효성(有效性)" = 효과가 있다는 의미 → 접근 가능성과 의미가 다름
> - "가용성(可用性)" = 사용할 수 있는 상태 → 보안 문맥에서 정확한 표현

## 섹션 2 — 기밀성 (Confidentiality)

**기밀성 (Confidentiality)** = 민감한 데이터가 **승인된 사람(authorized person)만** 접근할 수 있도록 보장하는 원칙

기밀성이 침해되면:
- 금전적 손실 (계좌 정보 유출, 사기 피해 등)
- 개인정보 침해 (신원 도용, 사생활 침해 등)
- 법적 문제 (개인정보 보호법 위반 등)

### 현실 세계 예시 — 엿듣기

사적인 대화를 나누는데 모르는 사람이 엿듣고 그 정보로 나중에 협박이나 조종에 사용한다면 기밀성이 침해된 것이다.

**예방법**: 안전한 장소에서 대화, 주변 상황 파악, 낯선 사람 앞에서 민감한 정보 언급 자제

### 디지털 세계 예시 — 공용 와이파이 도청

카페의 공용 와이파이에서 소셜 미디어에 로그인했다가 갑자기 로그아웃되고 계정에 접속이 불가능해진다. 이는 누군가가 같은 네트워크 내부에서 로그인 정보(아이디/비밀번호)를 가로챈 것이다. 데이터가 암호화되지 않은 채로 전송되면 중간에서 이를 훔쳐보는 것이 가능하다.

**예방법**:
- **암호화(Encryption)**: 데이터를 읽을 수 없는 형태로 변환해 가로채더라도 내용을 알 수 없게 함
- **접근 제어(Access Control)**: 승인된 사용자만 데이터에 접근할 수 있도록 권한을 관리

### 기밀성 판단 예시

| 상황 | 기밀성 달성 여부 | 이유 |
|------|----------------|------|
| Gmail 로그인 정보가 사무실 컴퓨터 모니터 포스트잇에 적혀 있다 | ❌ 침해 | 주변 모든 사람이 볼 수 있는 상태 |
| 회사 내부 문서는 업무상 필요한 직원들만 열람 가능하도록 권한 설정 | ✅ 달성 | 승인된 사람에게만 접근이 제한되어 있음 |
| 개인 문서가 링크 공개 설정으로 인터넷에 올라가 있다 | ❌ 침해 | URL을 아는 전 세계 누구나 접근 가능 |

## 섹션 3 — 무결성 (Integrity)

**무결성 (Integrity)** = **권한 없는 사람이 데이터를 수정하지 못하도록** 보장하는 원칙

"권한 없는"이 핵심이다. 승인된 사용자의 정상적인 수정은 무결성 침해가 아니다. **허가 없이 몰래 변경하는 것**이 문제다.

무결성이 침해되면:
- 데이터를 더 이상 신뢰할 수 없게 됨
- 변조된 데이터를 기반으로 잘못된 의사결정 발생
- 재무·의료·법률 기록 변조 시 심각한 결과 초래

### 현실 세계 예시 — 시험 점수 변조

교사가 채점한 시험 점수를 누군가가 기관에 제출하기 전에 몰래 바꿨다면 무결성이 침해된 것이다. 점수가 높아 보여도 그 데이터는 이미 신뢰할 수 없다.

**예방법**: 교사가 점수를 별도 용지에 따로 기록하고, 최종 제출 전에 원본과 비교 확인

### 디지털 세계 예시 — 은행 송금 계좌 변조

모바일 뱅킹으로 친구에게 돈을 보낼 때, 악성 프로그램이 거래 완료 전에 수취인 계좌 정보를 공격자의 계좌로 변경한다. 내가 입력한 계좌와 실제 송금된 계좌가 다르다. 데이터(계좌 정보)가 무단으로 수정된 것이다.

**예방법**:
- **해시(Hash)**: 데이터의 고유한 지문 값을 생성해, 변조 시 값이 달라지는 것으로 변조 탐지
- **디지털 서명(Digital Signature)**: 데이터가 특정 주체가 보낸 것임을 수학적으로 보증
- **체크섬(Checksum)**: 데이터 전송 전후 값을 비교해 변조 여부 확인

### 무결성 판단 예시

| 상황 | 무결성 달성 여부 | 이유 |
|------|----------------|------|
| 관리자 권한을 가진 승인된 사용자가 데이터베이스 레코드를 수정했다 | ✅ 달성 | 권한 있는 정상적인 변경 |
| 교사가 출석 기록을 잠금 상태로 저장했는데 학생 기록이 변경됐다 | ❌ 침해 | 권한 없는 무단 변경 |
| 주문이 결제 처리되는 사이에 주문 가격이 수정됐다 | ❌ 침해 | 권한 없는 무단 변경 |

## 섹션 4 — 가용성 (Availability)

**가용성 (Availability)** = **권한 있는 사용자가 필요할 때** 데이터와 서비스를 이용할 수 있도록 보장하는 원칙

현대 기업과 기관은 디지털 서비스에 크게 의존한다. 짧은 서비스 중단도 심각한 손실을 유발할 수 있다.

가용성이 침해되면:
- 사업 운영 불가능 (온라인 쇼핑몰 다운 → 판매 불가)
- 막대한 금전적 손실
- 의료·응급 통신 시스템 중단 시 생명에도 위협

### 현실 세계 예시 — 정전으로 인한 은행 접근 불가

은행에 돈이 안전하게 보관되어 있어도 정전으로 은행 문이 닫히면 돈을 인출할 수 없다. 데이터(돈)가 안전하게 존재하더라도 접근 자체가 불가능하면 가용성이 침해된 것이다.

**예방법**: 예비 발전기(UPS), 백업 시스템, 대체 접근 경로 마련

### 디지털 세계 예시 — DDoS 공격

**DDoS(Distributed Denial of Service, 분산 서비스 거부) 공격**은 수천~수만 개의 장치에서 대상 서버에 엄청난 양의 요청을 동시에 보내 서버를 마비시키는 공격이다. 서버는 정상 처리 용량을 초과하는 요청에 압도되어 응답을 멈춘다.

실제 사용자들은 데이터가 멀쩡히 존재해도 서비스에 접근할 수 없게 된다. 데이터 유출 없이도 서비스 마비만으로 심각한 피해가 발생하는 것이 가용성 공격의 특징이다.

**예방법**:
- **트래픽 부하 관리(Load Balancing)**: 요청을 여러 서버에 분산해 특정 서버에 과부하 방지
- **Rate Limiting**: 특정 IP가 임계값 이상의 요청을 보내면 자동 차단
- **CDN 활용**: 콘텐츠를 여러 지역 서버에 분산 배치해 공격 영향 최소화
- **백업 시스템**: 주 시스템 장애 시 즉시 전환 가능한 예비 시스템 운영

### 가용성 판단 예시

| 상황 | 가용성 달성 여부 | 이유 |
|------|----------------|------|
| 소프트웨어 설치 과정에서 필수 서비스가 일시 중단됐다 | ❌ 침해 | 권한 있는 사용자도 서비스 이용 불가 |
| 회사 웹사이트가 영업시간(09:00-18:00) 동안 접속 불가 상태다 | ❌ 침해 | 가장 필요한 시간에 서비스를 이용할 수 없음 |
| 모든 직원이 근무 시간 동안 필요한 시스템에 정상 접근 가능 | ✅ 달성 | 권한 있는 사용자가 필요한 시간에 이용 가능 |

## 섹션 5 — 보안 마인드셋 (Security Mindset)

CIA 트라이어드는 단순히 세 가지 정의의 집합이 아니다. **보안 전문가가 세상을 바라보는 사고방식(Security mindset)**이다.

보안 사고(Security Incident)가 발생했을 때, 보안 전문가들은 아래 세 가지 질문을 기준으로 영향 범위를 파악하고 대응 우선순위를 결정한다:

| 질문 | 판단하는 원칙 |
|------|-------------|
| 민감한 데이터가 권한 없는 사람에게 노출됐는가? | 기밀성 (Confidentiality) 침해 여부 |
| 허가 없이 데이터가 수정됐는가? | 무결성 (Integrity) 침해 여부 |
| 사용자들이 필요할 때 시스템이나 서비스를 이용할 수 없었는가? | 가용성 (Availability) 침해 여부 |

이 사고방식이 중요한 이유는, 하나의 보안 사고가 세 원칙 중 하나 또는 여러 개를 동시에 침해할 수 있기 때문이다. 어느 원칙이 침해됐는지 파악하면 어떤 복구 조치가 필요한지가 명확해진다.

### 실제 보안 사고 9가지 — CIA 원칙별 분류

아래는 실제로 발생하는 보안 사고 유형을 CIA 원칙에 따라 분류한 것이다.

**기밀성 침해 — 보면 안 되는 사람이 봤다**

| 사건 | 분류 |
|------|------|
| 백업 데이터 보안 설정 미흡으로 고객 민감 정보 유출 | 기밀성 |
| 직원이 로그인 자격증명(아이디/비밀번호)을 권한 없는 외부인과 공유 | 기밀성 |
| 직원 급여 파일이 잘못된 설정으로 인터넷에 공개 접근 가능 상태 | 기밀성 |

**무결성 침해 — 변경하면 안 되는 사람이 변경했다**

| 사건 | 분류 |
|------|------|
| 데이터베이스 레코드가 권한 없이 무단 수정됨 | 무결성 |
| 보고서 내용이 변조되어 재무 수치가 변경됨 | 무결성 |
| 공격자가 활동 흔적을 은폐하기 위해 시스템 로그를 삭제 | 무결성 |

**가용성 침해 — 필요한 사람이 필요한 때 이용할 수 없다**

| 사건 | 분류 |
|------|------|
| DoS(서비스 거부) 공격으로 이메일 서버가 오프라인 상태로 전환 | 가용성 |
| 중요 시스템 업데이트가 차단되어 서비스 정상 실행 불가 | 가용성 |
| 회사 웹사이트가 영업시간 중 접속 불가 상태 지속 | 가용성 |

> **주목할 포인트**: 시스템 로그 삭제가 무결성 침해로 분류된다. 로그는 "무슨 일이 있었는지"를 기록하는 데이터다. 공격자는 자신의 흔적을 지우기 위해 로그를 삭제하는 경우가 많은데, 이 행위 자체가 데이터(로그)를 무단으로 변조/삭제한 것이므로 무결성 원칙을 위반한다. 실제 침해 사고 분석(포렌식)에서 로그가 사라져 있다는 사실 자체가 공격의 흔적으로 간주된다.

## 주요 용어 정리

| 용어 | 정의 |
|------|------|
| **기밀성 (Confidentiality)** | 디지털 정보가 승인되지 않은 사람에게 노출되지 않도록 보장 |
| **무결성 (Integrity)** | 허가 없이 디지털 정보가 수정되지 않도록 보장 |
| **가용성 (Availability)** | 권한 있는 사용자가 필요할 때 디지털 정보를 이용할 수 있도록 보장 |
| **CIA 트라이어드 (CIA Triad)** | 위 세 가지 원칙을 통칭하는 보안 사고방식 |

| 이러한 모든 요소를 통칭하는 용어는? | `CIA Triad` |

**배운 점**

1. 기밀성과 무결성을 구분하는 핵심 기준은 **"볼 수 있는가(접근)"** 와 **"바꿀 수 있는가(수정)"** 다. 누군가가 데이터를 읽기만 했다면 기밀성 침해, 몰래 수정했다면 무결성 침해다. 같은 해킹 사고라도 어떤 행위가 발생했는지에 따라 침해된 원칙이 달라지기 때문에 구체적인 행위를 파악하는 것이 사고 분석의 첫 단계가 된다.
2. 무결성은 단순히 "데이터가 변경되지 않는 것"이 아니라 **"권한 없는 변경이 없는 것"** 이다. 관리자가 정상적으로 수정하는 것은 무결성 침해가 아니다. "데이터의 신뢰성이 떨어질 때 어떤 원칙이 침해됐는가?"에 Integrity가 답인 이유도 여기에 있다. 신뢰성이 떨어진다는 것은 누군가가 허가 없이 변조했을 가능성을 의미하기 때문이다.
3. CIA 트라이어드를 단순 암기하는 것보다 **각 원칙을 침해하는 공격 유형과 매핑하는 것**이 실전에 더 유용하다. Confidentiality → 도청/정보 유출, Integrity → 변조/위변조, Availability → DoS/DDoS. 이 매핑이 자연스럽게 되면 어떤 보안 사고 시나리오를 보더라도 빠르게 CIA 중 어떤 원칙이 침해됐는지 분류할 수 있다.

### 실습 2 — Task 3: 보안 사고 9개 분류 (드래그앤드롭)

**목표**: 9개 보안 사고 시나리오를 Confidentiality / Integrity / Availability 세 영역으로 분류

**결과**: 9/9 정답

| 사건 | 분류 |
|------|------|
| Sensitive customer data was leaked through an unsecured backup | Confidentiality |
| An employee shared the login credentials with an unauthorized person | Confidentiality |
| The employee salary files are publicly accessible to anyone on the Internet | Confidentiality |
| A database record was modified without authorization | Integrity |
| A report was altered, changing financial figures | Integrity |
| System logs are deleted to hide activity | Integrity |
| The email server was taken offline due to a denial of service attack | Availability |
| Critical system updates were blocked, preventing services from running | Availability |
| The company website is unavailable during business hours | Availability |

**플래그**: `THM{CIA_IS_ABOUT_BALANCE}`

**CIA 삼원칙은 단순 정의 집합이 아니라 하나의 사고방식, 이것은?** → `Security mindset`

**배운 점**

1. 시스템 로그 삭제(System logs are deleted to hide activity)가 **무결성 침해**로 분류된다는 것이 중요하다. 공격자는 침입 흔적을 지우기 위해 로그를 삭제하는 경우가 많다. 이 행위 자체가 로그라는 데이터를 무단으로 삭제/변조한 것이므로 무결성 위반이다. 실제 사고 분석(포렌식)에서 로그가 사라져 있다는 사실 자체가 "공격이 있었다"는 역설적인 증거가 되는 이유다.
2. DoS/DDoS 공격은 전형적인 **가용성 공격**이다. 데이터 유출이나 변조가 없더라도 서비스 자체를 이용할 수 없게 만드는 것만으로 심각한 보안 사고로 간주된다. 특히 의료 시스템, 금융 서비스처럼 가동 중단이 직접적인 피해로 이어지는 분야에서는 가용성이 기밀성만큼이나 중요한 원칙이다.
3. 플래그 `THM{CIA_IS_ABOUT_BALANCE}`에서 **"Balance(균형)"** 가 핵심 메시지다. 실제 보안 환경에서는 세 원칙 간에 트레이드오프가 발생한다. 기밀성을 극단적으로 강화하면(인증 절차 복잡화) 접근이 어려워져 가용성이 낮아진다. 반대로 가용성을 극단적으로 높이면(어디서나 쉽게 접근) 기밀성이 약해진다. 보안 전문가의 역할은 이 세 원칙 간의 균형을 상황과 조직의 필요에 맞게 최적화하는 것이다.

## 섹션 6 — 암호학이란?

**암호학 (Cryptography)** = 데이터를 읽을 수 없는 형태로 변환해 권한 없는 사람의 접근과 변조를 막는 기술

앞서 배운 CIA 트라이어드 중 **기밀성(Confidentiality)** 을 실제로 구현하는 핵심 수단이 암호학이다. 또한 전송 중 데이터가 변조되지 않았음을 확인하는 **무결성(Integrity)** 보호에도 활용된다.

**도입 질문**: 브라우저 주소 표시줄에 작은 자물쇠 아이콘이 보일 때, 인터넷을 통해 전송되는 내 데이터를 누군가가 읽거나 수정하는 것을 실제로 막는 것은 무엇일까?

→ 정답: **암호화(Encryption)**

**실제 시나리오**: 병원에서 환자의 이름, 질병, 치료 이력 등을 보험회사에 인터넷으로 전송할 때, 데이터가 수십 대의 라우터를 거치는 과정에서 보호 장치가 없으면 누구든지 읽거나 변경하거나 차단할 수 있다. 암호화가 이 문제를 해결한다.

**학습 목표**

- 암호화가 무엇이며 기밀성 보호에 왜 중요한지
- 평문(Plaintext)과 암호문(Ciphertext)의 차이
- 키(Key)와 알고리즘(Algorithm)이 무엇인지, 키의 비밀 유지가 왜 중요한지
- 대칭 암호화와 비대칭 암호화의 차이
- 실제 웹 브라우저에서 두 방식이 어떻게 결합되어 작동하는지

## 섹션 7 — 평문, 암호문, 그리고 시저 암호

### 평문(Plaintext) vs 암호문(Ciphertext)

| 용어 | 설명 | 예시 |
|------|------|------|
| **평문 (Plaintext)** | 암호화되지 않은, 누구나 읽을 수 있는 원본 데이터 | `HELLO` |
| **암호문 (Ciphertext)** | 암호화된 데이터. 키 없이는 읽을 수 없다 | `KHOOR` |

암호화의 목표: 암호문은 키를 모르는 사람에게 **무작위적인 의미 없는 문자열**처럼 보여야 한다.

앨리스가 `HELLO`를 밥에게 보내는 과정:
1. 앨리스가 알고리즘과 비밀 키로 `HELLO` → `KHOOR` 변환 후 전송
2. 밥이 `KHOOR` 수신 → 동일한 알고리즘과 키로 복호화 → 다시 `HELLO`

### 시저 암호 (Caesar Cipher)

시저 암호는 메시지의 각 글자를 알파벳에서 **고정된 숫자만큼 이동**시키는 암호화 방식이다. 기원전 50년경 율리우스 카이사르가 군사 메시지 보호에 사용했다고 전해진다.

- **알고리즘**: 각 글자를 알파벳에서 N칸 앞으로 이동 (완전히 공개된 정보)
- **키**: 이동할 숫자 N (비밀 정보)

**Key = 3 으로 HELLO 암호화 → KHOOR**

```
평문:   H  E  L  L  O
       +3 +3 +3 +3 +3
암호문: K  H  O  O  R
```

알파벳 순환 규칙 (key = 3):

| 평문 | 암호문 | | 평문 | 암호문 |
|------|--------|---|------|--------|
| A | D | | X | A (끝에서 처음으로) |
| B | E | | Y | B |
| C | F | | Z | C |

**복호화 (KHOOR → HELLO)**: 각 글자를 반대 방향으로 3칸 이동

```
암호문: K  H  O  O  R
       -3 -3 -3 -3 -3
평문:   H  E  L  L  O
```

**시저 암호의 핵심 원칙**:
- **알고리즘은 공개**: 어떻게 작동하는지는 누구나 알아도 됨
- **키만 비밀**: 앨리스와 밥만 이동 숫자를 알고 있음

### 시저 암호의 한계 — 왜 실제로는 쓰지 않는가?

시저 암호는 키가 1~25 사이의 숫자 하나뿐이다. 공격자가 암호문을 가로채면 25가지 경우를 모두 시도해보면 된다.

- 사람이 직접 시도: 지루하지만 가능
- **컴퓨터로 시도: 약 1밀리초(0.001초)** 만에 전부 해독 가능

시저 암호는 교육 목적으로만 사용하며, 실제 시스템에서는 절대 사용하지 않는다.

**실제 사용 알고리즘**: **AES (Advanced Encryption Standard, 고급 암호화 표준)**
- 훨씬 복잡한 수학 연산 사용
- 가능한 키 조합: `2^256`가지 (10^77) — 현존하는 모든 컴퓨터를 합쳐도 우주 나이보다 긴 시간 필요
- 기본 개념은 동일: **알고리즘 + 키 + 평문 → 암호문**

## 섹션 8 — 대칭 암호화 (Symmetric Encryption)

**대칭 암호화 (Symmetric Encryption)** = **동일한 키 하나**로 암호화와 복호화를 모두 수행하는 방식

시저 암호가 대칭 암호화의 가장 간단한 예다.

```
[앨리스]                                        [밥]
평문 HELLO
  → [암호화: 키 K 사용]
    → 암호문 KHOOR  ───전송───▶  암호문 KHOOR
                                  → [복호화: 동일한 키 K 사용]
                                    → 평문 HELLO
```

### 대칭 암호화의 장점

- **속도가 매우 빠름**: 대용량 데이터를 실시간으로 처리 가능
- **효율적**: 파일 암호화, 하드 드라이브 암호화, 네트워크 트래픽 암호화에 적합

### 대칭 암호화의 치명적 단점 — 키 분배 문제

앨리스와 밥이 처음 만날 때 어떻게 키를 안전하게 공유할까?

```
시도 1: 앨리스가 키를 인터넷으로 그냥 전송
  → 공격자가 가로채면 이후 모든 암호문을 복호화 가능 ❌

시도 2: "그럼 키를 암호화해서 전송하자!"
  → 그 키를 암호화하려면 또 다른 키가 필요
    → 그 키를 암호화하려면 또 다른 키가 필요
      → 끝없이 반복... ❌ (무한 퇴행)
```

이를 **키 분배 문제 (Key Distribution Problem)** 라 한다. 대칭 암호화를 단독으로 사용할 때의 근본적인 약점이다.

→ 해결책: 다음 섹션에서 배울 **비대칭 암호화**

---

## 섹션 9 — 비대칭 암호화 (Asymmetric Encryption)

**비대칭 암호화 (Asymmetric Encryption)** = **수학적으로 연관된 두 개의 키**를 사용하는 암호화 방식

| 키 종류 | 설명 |
|---------|------|
| **공개 키 (Public Key)** | 누구나 알고 사용할 수 있다. 인터넷에 공개해도 됨 |
| **개인 키 (Private Key)** | 오직 주인만이 비밀로 보관. 절대 공유하지 않음 |

**핵심 규칙**:
- 공개 키로 암호화한 내용 → **개인 키로만** 복호화 가능
- 두 키는 수학적으로 연결되어 있지만, 공개 키에서 개인 키를 계산해내는 데 **현대 컴퓨터로 수백~수천 년** 이 걸림 → 사실상 불가능

### 우체통 비유

```
┌─────────────────────────────┐
│          우체통             │
│  ┌───────────────────┐     │
│  │   우편 투입구     │ ← 공개 키 (Public Key)
│  │ (누구나 편지 투입 가능)  │   누구든 메시지를 넣을 수 있음
│  └───────────────────┘     │
│                             │
│  [잠긴 문] ← 개인 키 (Private Key)
│  (오직 우편함 주인만 열 수 있음)
└─────────────────────────────┘
```

### 앨리스가 밥에게 비밀 메시지를 보내는 과정

```
1. 밥이 공개 키(Public Key)와 개인 키(Private Key) 한 쌍 생성
   공개 키 → 웹사이트·이메일 등으로 전 세계에 공개
   개인 키 → 절대 공유하지 않고 본인만 보관

2. 앨리스가 밥의 공개 키를 입수 (공개된 정보라 가로채도 상관없음)

3. 앨리스가 밥의 공개 키로 메시지를 암호화 후 전송

4. 밥이 자신의 개인 키로 복호화 → 원본 메시지 확인
   공격자가 암호문을 가로채도 개인 키 없이는 해독 불가
```

### 키 배포 문제 해결

네트워크를 통해 **비밀 키를 전혀 교환하지 않아도 된다**. 공개적으로 전송한 것은 공개 키뿐이고, 공개 키는 애초에 비밀이 아니다.

## 섹션 10 — HTTPS와 인증서

### HTTPS — 비대칭과 대칭 암호화의 결합

비대칭 암호화가 가장 널리 사용되는 곳이 **HTTPS**다. 브라우저 주소창의 **자물쇠 아이콘** = 아래 과정이 완료됐다는 신호다.

`https://google.com` 접속 시 일어나는 일:

```
1. [브라우저] → 웹사이트에 공개 키 요청

2. [웹사이트] → 인증서(공개 키 + 신원 정보 포함)를 전송

3. [브라우저 + 웹사이트] → 비대칭 암호화를 이용해
                            공유 대칭 키(Shared Secret Key)를 안전하게 협의

4. 이후 실제 데이터 전송 → 속도가 빠른 대칭 암호화로 전환
```

이 방식을 **하이브리드 접근 방식 (Hybrid Approach)** 이라 한다:
- **비대칭 암호화**: 초기 키 교환 담당 → 키 분배 문제 해결
- **대칭 암호화**: 실제 데이터 처리 담당 → 속도 문제 해결

두 방식의 장점만 결합한 구조다.

### 인증서 (Certificate) — 공개 키의 신원 보증

**문제**: 앨리스가 받은 공개 키가 진짜 밥의 것인지, 밥인 척하는 공격자의 것인지 어떻게 알 수 있을까?

**해결책**: 인증서 (Certificate)

인증서는 아래 세 가지를 포함하는 디지털 문서다:
- 웹사이트의 공개 키
- 해당 키가 누구의 것인지 (예: `google.com`)
- **인증 기관(CA, Certificate Authority)** 의 디지털 서명

**CA (인증 기관)**: 신뢰할 수 있는 제3자 기관 (예: Let's Encrypt, DigiCert, GlobalSign)
- 브라우저와 운영체제에는 신뢰할 수 있는 CA 목록이 미리 내장되어 있음
- CA가 인증서에 "이 공개 키는 정말 이 사이트의 것이 맞다"는 서명을 함

**브라우저의 인증서 확인 과정**:

```
인증서 수신
  → CA 서명 확인 (신뢰 CA 목록과 대조)
  → 유효 기간 확인 (만료 여부)
  → 모두 정상 → 자물쇠 아이콘 ✅
  → 문제 발생 → 경고 메시지 + 연결 거부 가능 ⚠️
```

| 항목 | 내용 |

| 발행처 (Issued By) | `Let's Encrypt` (무료 공개 CA) |
| 발행일 | 2025년 11월 11일 |
| 만료일 | 2026년 2월 9일 |

> **Let's Encrypt**: 무료 SSL/TLS 인증서를 발급하는 비영리 CA 기관. 인증서는 90일마다 갱신이 필요하다. 소규모 사이트부터 대형 서비스까지 폭넓게 사용된다.

## 섹션 11 — 대칭 vs 비대칭 비교 및 실제 활용

### 두 암호화 방식 비교

| 항목 | 대칭 암호화 | 비대칭 암호화 |
|------|------------|--------------|
| **키의 수** | 1개 (암호화 = 복호화 키) | 2개 (공개 키 + 개인 키) |
| **키 공유** | 비밀리에 공유해야 함 ❌ | 공개 키는 공개적으로 공유 가능 ✅ |
| **속도** | 매우 빠름 ✅ | 상대적으로 느림 ❌ |
| **주용도** | 대용량 데이터 암호화 (파일, 네트워크 트래픽) | 키 교환, 디지털 서명 |
| **약점** | 키 분배 문제 | 느린 처리 속도 |
| **대표 알고리즘** | AES | RSA |

### 실제 시스템: 두 방식의 조합

현대 보안 시스템(HTTPS 등)은 두 방식을 결합해 각각의 약점을 서로 보완한다:

```
[1단계] 비대칭 암호화로 초기 키 교환
   → 인터넷을 통해 비밀을 공유하지 않아도 됨
   → 키 분배 문제 해결 ✅

[2단계] 대칭 암호화로 실제 데이터 전송
   → 빠른 속도
   → 대용량 데이터 처리에 효율적 ✅

결과: 자물쇠 아이콘 = 이 두 단계가 완료된 상태
      비밀번호, 은행 정보, 개인 메시지 등이 안전하게 보호됨
```

### 암호화의 한계 — 암호화는 만능이 아니다

암호화는 방어자의 가장 중요한 도구 중 하나지만, **보안 체계의 한 요소**일 뿐이다.

암호화만으로 해결되지 않는 것들:
- **약한 비밀번호**: 암호화가 되어 있어도 비밀번호가 `1234`라면 의미 없음
- **키 관리 실패**: 개인 키를 분실하거나 유출하면 암호화 자체가 무력화됨
- **사용자 실수**: 피싱에 속아 직접 정보를 입력하면 암호화로 막을 수 없음
- **소프트웨어 취약점**: 암호화 이외의 취약점으로 공격 가능
- **사고 대응 부재**: 침해 발생 시 탐지·대응 능력도 함께 필요

## 주요 용어 정리 (암호학)

| 용어 | 정의 |
|------|------|
| **평문 (Plaintext)** | 암호화되지 않은, 읽을 수 있는 원본 데이터 |
| **암호문 (Ciphertext)** | 암호화된 데이터. 키 없이는 의미 없는 문자열처럼 보임 |
| **키 (Key)** | 암호화와 복호화를 제어하는 비밀 값 |
| **알고리즘 (Algorithm)** | 키를 사용해 암호화·복호화하는 공개된 수학적 방법 |
| **대칭 암호화 (Symmetric Encryption)** | 암호화와 복호화에 동일한 키 하나를 사용하는 방식 |
| **비대칭 암호화 (Asymmetric Encryption)** | 공개 키와 개인 키 쌍을 사용하는 방식 |
| **공개 키 (Public Key)** | 누구나 사용할 수 있는 키. 공개해도 안전함 |
| **개인 키 (Private Key)** | 소유자만 비밀로 보관하는 키. 절대 공유하지 않음 |
| **키 분배 문제 (Key Distribution Problem)** | 대칭 암호화에서 초기에 키를 안전하게 공유하는 방법의 문제 |
| **인증서 (Certificate)** | 공개 키와 소유자 정보를 CA가 서명한 디지털 문서 |
| **CA (인증 기관)** | 인증서에 서명해 공개 키의 신원을 보증하는 신뢰된 제3자 기관 |
| **AES** | Advanced Encryption Standard. 현대 표준 대칭 암호화 알고리즘 |
| **HTTPS** | HTTP + TLS(비대칭+대칭 하이브리드). 자물쇠 아이콘으로 표시됨 |

## 실습 기록 — 암호학 개념

### 실습 1 — Task 2: 시저 암호 게임 (비밀 메시지 복구)

| 문제 | 정답 |
|------|------|
| 비밀 메시지 복구 게임 완료 플래그 | `THM{CAESAR_CIPHER_MASTER_2026}` |
| 키 5로 `CYBER` 를 암호화하면? | `HDGJW` |
| 다음 암호문을 해독하라: `FVZCYR PNRFNE PVCURE` | `SIMPLE CAESAR CIPHER` |

**CYBER + Key 5 = HDGJW 계산 과정 (A=0 기준)**:

```
C(2)  + 5 = 7  = H
Y(24) + 5 = 29 → 29-26 = 3 = D  ← 알파벳 끝을 넘으면 처음으로 순환
B(1)  + 5 = 6  = G
E(4)  + 5 = 9  = J
R(17) + 5 = 22 = W
→ HDGJW ✓
```

**FVZCYR PNRFNE PVCURE 해독 (Key = 13, ROT13)**:

```
F(5) -13 = -8+26 = 18 = S    P(15)-13 = 2  = C    P(15)-13 = 2  = C
V(21)-13 = 8  = I            N(13)-13 = 0  = A    V(21)-13 = 8  = I
Z(25)-13 = 12 = M            R(17)-13 = 4  = E    C(2) -13 =-11+26=15=P
C(2) -13 =-11+26=15=P        F(5) -13 =-8+26=18=S  U(20)-13 = 7  = H
Y(24)-13 = 11 = L            N(13)-13 = 0  = A    R(17)-13 = 4  = E
R(17)-13 = 4  = E            R(17)-13 = 4  = E    R(17)-13 = 4  = E

→ SIMPLE  CAESAR  CIPHER ✓
```

**배운 점**

1. 시저 암호에서 알파벳 끝(Z)을 넘어가면 다시 A로 돌아오는 순환 구조는 **모듈러 연산(Modular Arithmetic)** 이다. `(위치 + 키) % 26` 이라는 수식 하나가 시저 암호 전체를 표현한다. 현대 암호화(AES, RSA 등)도 훨씬 복잡하지만 같은 종류의 수학 원리를 기반으로 동작한다. 암호학이 결국 수학이라는 것을 보여주는 가장 간단한 예다.
2. `FVZCYR PNRFNE PVCURE` 를 해독한 키는 13(ROT13)이다. ROT13은 영어 알파벳 26개의 정중앙인 13을 이동값으로 사용하기 때문에 암호화와 복호화가 완전히 동일한 연산이 된다는 특성이 있다. 실제 보안 용도로는 전혀 사용하지 않지만, 인터넷에서 스포일러를 가리거나 포럼에서 민감한 내용을 숨기는 관습적 용도로 여전히 사용된다.
3. 시저 암호의 진짜 교훈은 **키 공간(Key Space)의 크기가 보안 강도를 결정한다**는 것이다. 시저 암호는 키가 25가지뿐이라 컴퓨터로 1ms 만에 전수 탐색 가능하다. 반면 AES-256은 키 공간이 `2^256`가지로, 지구상 모든 컴퓨터가 우주 나이보다 오래 계산해도 찾을 수 없다. 알고리즘이 공개되어 있어도 키 공간이 충분히 크면 안전하다는 원칙이 현대 암호학의 기반이다.

### 실습 2 — Task 3: 비대칭 암호화 개념 퀴즈

| 질문 | 정답 |
|------|------|
| 비대칭 암호화에서 절대 공개하면 안 되는 키는? | `private key` |
| 암호화와 복호화를 제어하는 비밀 정보를 무엇이라 하는가? | `key` |
| 대칭 암호화를 단독으로 사용할 때의 근본적인 문제는? | `key distribution` |
| 공개 키와 개인 키 쌍을 사용하는 암호화 유형은? | `asymmetric` |

**배운 점**

1. 비대칭 암호화에서 **공개 키를 공개해도 안전하다**는 것은 직관에 반하는 가장 중요한 개념이다. 암호화에 쓰는 키(공개 키)와 복호화에 쓰는 키(개인 키)를 수학적으로 분리했기 때문이다. 이 아이디어 덕분에 인터넷에서 처음 만나는 두 사람도 사전에 비밀을 공유하지 않고 안전하게 통신할 수 있다. 1976년 Diffie와 Hellman이 이 개념을 발표했고, 이것이 현대 인터넷 보안 전체의 토대가 됐다.
2. HTTPS에서 비대칭 암호화가 **모든 데이터를 직접 암호화하지 않는 이유는 속도 때문**이다. 비대칭 암호화는 복잡한 수학 연산 때문에 대칭 암호화보다 수십~수백 배 느리다. 그래서 비대칭 암호화는 "공유 키를 협의하는" 짧은 초기 단계에만 사용하고, 실제 데이터는 빠른 대칭 암호화로 처리한다. 보안과 성능을 동시에 달성하기 위한 역할 분담 설계다.
3. CA(인증 기관)가 없다면 비대칭 암호화도 **중간자 공격(Man-in-the-Middle Attack)** 에 취약하다. 공격자가 밥인 척하며 가짜 공개 키를 앨리스에게 보내면, 앨리스는 밥과 통신하고 있다고 믿지만 실제로는 공격자와 통신하는 셈이다. CA는 "이 공개 키가 진짜 이 도메인의 것"임을 보증하는 신뢰된 제3자 역할을 한다. 브라우저의 자물쇠 아이콘이 단순 암호화가 아닌 **신원 검증까지 완료**됐다는 의미인 이유다.
