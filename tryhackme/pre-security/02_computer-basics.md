## 섹션 1 — 컴퓨터 하드웨어 구성

| 부품 | 인체 비유 | 역할 | 연결 방식 |
|------|---------|------|---------|
| **메인보드 (Motherboard)** | 골격 + 신경계 | 모든 부품을 자리에 고정하고 서로 연결하는 중심 회로판 | 모든 부품이 직접 꽂히거나 메인보드를 통해 연결됨 |
| **CPU (Central Processing Unit)** | 뇌 | 프로세서(Processor)라고도 불린다. 모든 명령을 처리하며, 최근에는 여러 코어로 병렬 처리 | CPU 소켓 |
| **RAM (Random Access Memory)** | 단기·작업 기억 | CPU가 빠르게 접근할 데이터를 임시 보관. 전원 차단 시 데이터 소멸 (휘발성) | 메인보드 DIMM 슬롯 (DDR5, DDR6 등 세대 규격) |
| **SSD / HDD (Storage)** | 장기 기억 | 데이터를 장기간 보관. HDD: 저렴·대용량·속도 낮음 / SSD: 빠름·비쌈 | SATA 케이블, PCI Express 슬롯 |
| **네트워크 어댑터** | — | 다른 시스템과의 유선·무선 통신을 담당 | 메인보드 내장 또는 PCI Express 확장 카드 |
| **PSU (Power Supply)** | 심장 | 콘센트에서 전력을 받아 각 부품에 맞는 전압으로 분배. 공급 능력 초과 시 시스템 불안정 | 메인보드 주 커넥터, Molex 커넥터 등 |
| **그래픽 카드 (GPU)** | 시각 처리 | OS·프로그램의 시각 데이터를 처리해 모니터로 출력 | 메인보드 PCI Express 슬롯 |
| **입출력 장치 (I/O)** | 감각 기관 | 입력: 키보드·마우스·마이크·스캐너 / 출력: 모니터·프린터·스피커 | USB, HDMI, DisplayPort 등 |

## 섹션 2 — 부팅 과정 (Boot Process)

### 1단계 — 전원 버튼 누르기
컴퓨터 시스템의 전원 버튼을 누르면 전원 공급 장치(PSU)로 신호가 전송되어 전원이 공급된다. 우리가 잠자는 동안 우리 몸의 전원이 꺼져 있다고 상상해 보면, 잠에서 깨어나 산소를 공급받으면 우리 몸은 혈액을 펌핑하고 활동을 시작한다.

### 2단계 — 펌웨어 시작
신체가 시동되면 핵심 구성 요소들은 작동을 시작하지만 뇌는 아직 의식이 있는 상태가 아니다. 컴퓨터 시스템에도 모든 구성 요소가 시동될 수 있도록 하는 펌웨어가 있다. 이를 관리하는 중앙 시스템을 **통합 확장 가능 펌웨어 인터페이스 (UEFI)** 라고 한다. (구버전은 BIOS라고 불렸으며, 현재는 UEFI로 대체됨)

### 3단계 — 전원 켜짐 자가 진단 (POST)
UEFI는 **전원 켜짐 자체 테스트 (Power-On Self Test)** 를 실행하여 필요한 모든 구성 요소가 있는지, 올바르게 구성되었는지, 그리고 제대로 작동하는지 테스트한다.

### 4단계 — 부팅 장치 선택
UEFI는 운영 체제 부팅 루틴을 찾기 위해 어떤 장치를 먼저 찾아야 하는지 우선순위를 정한 목록을 유지한다.

### 5단계 — 부트로더 초기화
선택된 부팅 장치에서 **부트로더**가 초기화된다. 이 부트로더는 운영 체제(OS)를 선택된 부팅 장치에서 RAM으로 전송한다. OS가 이전되면 UEFI는 다양한 구성 요소에 대한 제어 권한을 OS에 넘긴다.

## 섹션 3 — 컴퓨터 종류

### 전통적인 컴퓨터 (별도 화면·키보드 필요)

| 종류 | 특징 |
|------|------|
| **랩탑** | 휴대 가능한 일상용 컴퓨팅 |
| **데스크톱** | 고정된 위치에서 지속적인 성능 |
| **워크스테이션** | 전문적인 업무에 필요한 정확성과 신뢰성 |
| **서버** | 화면·키보드 불필요 / 네트워크를 통해 다수의 사용자에게 서비스를 제공 |

### 소형·특수 목적 컴퓨터

| 종류 | 특징 | 예시 |
|------|------|------|
| **스마트폰** | 배터리 수명과 연결성에 최적화된 주머니 크기의 컴퓨터 | 아이폰, 안드로이드 폰 |
| **태블릿** | 터치 우선 방식의 대형 화면 컴퓨터 | 아이패드, 드로잉 태블릿 |
| **IoT 장치** | 단일 목적을 가진 네트워크 연결 장치 | 온도 조절기, 스마트 초인종, 피트니스 트래커 |
| **임베디드 컴퓨터** | 다른 장치에 내장된 컴퓨터 | 커피메이커 컨트롤러, 자동문 센서, 램프 디머 칩 |

### IoT vs 임베디드 비교

- 둘 다 소형이며, 단일 용도로 사용된다.
- 차이점은 **연결성**이다.
- **IoT 기기**: 데이터를 보고하거나 명령을 수신하기 위해 네트워크에 연결
- **임베디드 컴퓨터**: 네트워크에 연결하지 않을 수 있다. 기계 내부에서 작업을 수행하며, 종종 아무도 그 존재를 알지 못하는 경우도 있다.

### 요약

컴퓨터는 항상 가장 빠르거나 화려한 것은 아니다. 때로는 문을 열고 닫고, 비행기를 날게 하고, 커피 머신을 작동시키는 데 필수적인, 조용히 제 역할을 다하는 칩들이 바로 그 역할을 한다.

## 섹션 4 — 클라이언트/서버 기초

### 1. 서비스, 클라이언트, 서버

- **브라우저** = 웹페이지를 요청하는 클라이언트이다.
- **서버** = 웹페이지를 제공하는 시스템이다.

### 2. 요청과 응답

- 브라우저(클라이언트)를 사용하여 서버에 웹페이지를 요청했고
- 서버는 그 웹페이지를 클라이언트로 전송

### 3. 규약 (프로토콜)

프로토콜은 클라이언트가 서버와 통신하는 방법을 정의한다.

| 항목 | 내용 | 예시 |
|------|------|------|
| **명령어 (Commands)** | 클라이언트와 서버가 이해할 수 있는 명령어를 정의한다. | GET, POST, DELETE 등 |
| **요청 구조 (Request Structure)** | 요청이 어떤 순서와 형식으로 구성되는지 정의한다. | 명령 → 지시사항 순서 |
| **문법 (Syntax)** | 데이터를 표현하는 언어·형식을 정의한다. | HTML, JSON 등 |
| **정상 응답 (Normal Response)** | 올바른 요청에 어떤 응답을 반환해야 하는지 정의한다. | 200 OK, 201 Created 등 |
| **오류 응답 (Error Response)** | 잘못된 요청에 어떤 오류 응답을 반환해야 하는지 정의한다. | 404 Not Found, 500 Internal Server Error 등 |

### 4. 포트

- 포트는 시스템에서 실행 중인 특정 서비스를 식별하는 데 사용
- 클라이언트가 서버의 서비스에 액세스하려면 올바른 포트를 사용하여 연결해야 한다.
- 포트 번호 범위: 0~65535. 0~1023은 **잘 알려진 포트 (Well-Known Ports)** 로 서비스가 고정되어 있다.

**자주 쓰이는 대표 포트 번호**

| 포트 | 프로토콜 | 설명 |
|------|---------|------|
| **21** | FTP | 파일 전송 (File Transfer Protocol) |
| **22** | SSH | 암호화된 원격 접속 (Secure Shell) |
| **23** | Telnet | 암호화 없는 원격 접속 (현재는 SSH로 대체) |
| **25** | SMTP | 이메일 발송 (Simple Mail Transfer Protocol) |
| **53** | DNS | 도메인 이름 조회 (Domain Name System) |
| **80** | HTTP | 일반 웹 트래픽 (암호화 없음) |
| **443** | HTTPS | 암호화된 웹 트래픽 (SSL/TLS) |
| **3306** | MySQL | 데이터베이스 접근 |
| **3389** | RDP | Windows 원격 데스크톱 (Remote Desktop Protocol) |

> **보안 참고**: 포트 번호를 알면 어떤 서비스가 공격 대상인지 파악할 수 있다. 공격자는 포트 스캔(nmap 등)으로 열린 포트를 먼저 확인한다.

### 5. DNS (Domain Name System)

- Domain Name System의 약자로 GPS와 유사하게 작동한다.
- 웹사이트 이름을 입력하면, DNS가 이를 서버의 위치로 해석
- 위치 좌표를 **인터넷 프로토콜(IP) 주소**라고 한다.
- 이 IP 주소를 집 주소(도로명, 집 번호, 우편번호, 도시, 국가)라고 생각하면 컴퓨터 시스템을 위한 주소라고 이해할 수 있다.

**DNS가 필요한 이유**

사람은 `www.google.com` 같은 이름은 외울 수 있지만, `142.250.196.100` 같은 IP 주소는 외우기 어렵다. DNS는 이 이름을 IP 주소로 자동 변환해 주는 인터넷의 전화번호부 역할을 한다.

**DNS 동작 흐름**

```
① 브라우저에 www.google.com 입력
        ↓
② 내 컴퓨터가 DNS 서버에 "www.google.com의 IP가 뭐야?" 요청
        ↓
③ DNS 서버가 IP 주소(예: 142.250.196.100) 응답
        ↓
④ 브라우저가 해당 IP 주소의 서버에 접속하여 웹 페이지 요청
```

> **보안 참고**: DNS 스푸핑(DNS Spoofing) 공격은 이 과정에서 가짜 IP를 응답해 사용자를 피싱 사이트로 유도한다.

### HTTP 메서드 (HTTP Methods)

| 메서드 | 설명 | 주요 용도 |
|--------|------|---------|
| **GET** | 서버에서 리소스를 조회한다. | 웹 페이지 로드, 데이터 조회 |
| **POST** | 서버에 데이터를 전송하여 새 리소스를 생성한다. | 로그인, 폼 제출, 파일 업로드 |
| **PUT** | 지정된 리소스를 완전히 교체(전체 업데이트)한다. | 사용자 정보 전체 수정 |
| **DELETE** | 서버에서 리소스를 삭제한다. | 게시글 삭제, 계정 삭제 |
| **PATCH** | 리소스의 일부만 수정한다. | 비밀번호만 변경, 이름만 수정 |
| **HEAD** | GET과 같지만 응답 본문(body) 없이 헤더만 반환한다. | 파일 크기·존재 여부 확인 |
| **OPTIONS** | 서버가 지원하는 HTTP 메서드 목록을 조회한다. | CORS 사전 요청 (Preflight) — **CORS(Cross-Origin Resource Sharing)** 란 브라우저가 다른 도메인의 서버에 데이터를 요청할 때, 실제 요청 전에 해당 서버가 이를 허용하는지 먼저 확인하는 보안 메커니즘이다. |
| **CONNECT** | 목적지 서버로의 터널을 설정한다. | HTTPS 프록시 연결 |
| **TRACE** | 요청 경로를 추적하여 중간 서버를 확인한다. | 네트워크 디버깅 (보안상 대개 비활성화) |

> **보안 참고**: DELETE·PATCH·TRACE는 서버에서 비활성화하거나 인증을 요구하는 경우가 많다.

### HTTP 요청 구성 요소

URL 구조 예시:

```
https://www.example.com/page.html
  ↑           ↑              ↑
스키마        호스트          파일 이름(경로)
```

| 구성 요소 | 설명 | 예시 |
|----------|------|------|
| **스키마 (Scheme)** | 어떤 프로토콜(통신 규약)을 사용해 리소스에 접근할지 지정한다. `http://` = 암호화 없음, `https://` = SSL/TLS 암호화 적용. **SSL/TLS**는 인터넷 통신을 암호화하는 기술로, 전송 중 데이터를 가로채도 내용을 알 수 없게 만든다. SSL은 구버전이고 현재는 TLS가 사용되지만 통상 SSL/TLS로 함께 표기한다. | `https://` |
| **호스트 (Host)** | 리소스를 제공하는 서버의 도메인 이름이다. 브라우저가 요청을 보낼 때 DNS가 이 이름을 실제 IP 주소로 변환한다. | `www.example.com` |
| **파일 이름 (Path)** | 서버에서 요청하는 파일 또는 경로다. `/`만 있으면 보통 `index.html`을 의미한다. | `/page.html` |
| **IP 주소** | 웹사이트가 실제로 호스팅되는 서버의 IP 주소다. 같은 기기에서 호스팅하면 루프백 주소 `127.0.0.1`이 표시된다. | `93.184.216.34` |
| **상태코드 (Status Code)** | 서버가 요청을 처리한 결과를 숫자로 알려준다. `2xx` = 성공, `3xx` = 리다이렉트, `4xx` = 클라이언트 오류, `5xx` = 서버 오류 | `200 OK` / `404 Not Found` / `500 Internal Server Error` |

## 섹션 5 — 가상화 기초 (Virtualization)

### 가상화의 탄생 배경

가상화는 각각의 물리적 서버를 사용할 때 발생하는 비용 문제와 효율성 문제를 해결하기 위해 탄생했다.

기업이 웹사이트, 데이터베이스, 이메일 서비스, 내부 애플리케이션을 운영하려면 각각에 대해 별도의 물리적 서버가 필요했는데, 이로 인해 다음과 같은 문제가 발생했다.

### 기존 물리 서버 방식의 문제점

| 문제 | 설명 |
|------|------|
| **높은 비용** | 여러 대의 물리적 서버를 구매하는 것은 하드웨어 비용뿐 아니라 전기료, 냉각비, 유지보수비, 데이터 센터 공간까지 포함하면 상당한 비용이 필요하다. |
| **낮은 활용률** | 대부분의 애플리케이션은 서버의 최대 용량을 사용하지 않고 5~20%의 사용률에 머물러 용량을 낭비하고 있다. |
| **느린 배포 속도** | 새로운 물리적 서버를 설정하는 데 며칠에서 몇 주가 걸릴 수 있다. |
| **낮은 확장성** | 애플리케이션에 더 많은 리소스가 필요하게 되면 서버를 하나 더 구매해야 하는 경우가 발생할 수 있다. |

### 가상 컴퓨터 (VM, Virtual Machine)

가상 컴퓨터는 랩 머신(Lab Machine)이라고도 한다. VM은 물리적 하드웨어를 동일하게 공유하지만, 자체 운영 체제, 앱 및 설정을 갖춘 독립적인 시스템처럼 작동한다.

### 하이퍼바이저 (Hypervisor)

가상화의 핵심 기술로, 가상 머신을 생성하고 관리하는 소프트웨어다.

**주요 기능**
1. 물리적 컴퓨터를 여러 개의 가상 컴퓨터로 나눈다.
2. 각 가상 머신에 적절한 자원을 할당한다.
3. 모든 것을 격리하고 안전하게 유지한다.
4. 가상 머신의 수명 주기를 관리한다.

**구현 방식**

| 구분 | 유형 1 (Type 1) | 유형 2 (Type 2) |
|------|----------------|----------------|
| 실행 위치 | 물리적 하드웨어에서 직접 실행 | 기존 운영 체제 위에서 실행 |
| 특징 | 빠르고 효율적 | 설치가 쉽고 접근성이 높음 |
| 적합한 환경 | 서버 및 전문 환경 | 학습, 테스트, 소규모 환경 |
| 대표 제품 | VMware ESXi, Microsoft Hyper-V, Xen | VirtualBox, VMware Workstation, Parallels |

**사용 사례별 유형 추천**

| 사용 사례 | 유형 1 | 유형 2 |
|----------|:------:|:------:|
| 악성 파일 테스트 | | ✓ |
| 프로덕션 서버 | ✓ | |
| 데이터베이스 서버 | ✓ | |
| 소프트웨어 테스트 | | ✓ |
| 칼리 리눅스 | | ✓ |
| 데이터 센터 | ✓ | |

### 실험실 VM (Lab Machine)

하이퍼바이저가 생성한 가상 컴퓨터로, 실제 컴퓨터처럼 동작한다.

- 자체적인 가상 공간을 가지고 있음
- 다른 VM과 완전히 격리되어 있음
- Oracle VirtualBox나 VMware Workstation 같은 도구를 사용하면 자신의 컴퓨터에 VM을 배포할 수 있다. 이러한 소프트웨어는 2형 하이퍼바이저 역할을 하며 Windows, macOS, Linux 등 여러 운영 체제를 실행할 수 있도록 해준다.

### 컨테이너 (Container)

단일 애플리케이션과 이를 지원하는 데 필요한 모든 구성 요소를 실행하는 경량의 격리된 환경이다.

별도의 운영 체제 전체를 설치하는 대신, 컨테이너는 기존 시스템의 핵심 부분을 빌려와 커널에서 실행된다. **커널**은 운영 체제의 일부로서 하드웨어와 통신하고 메모리 및 실행 중인 프로그램과 같은 리소스를 관리한다.

**컨테이너가 작고 독립적인 공간처럼 작동하는 이유**

- 애플리케이션과 그 종속 요소를 패키징한다.
- 호스트의 운영 체제를 공유하므로 거의 즉시 시작된다.
- 서로 격리된 상태로 남아 있어, 하나의 컨테이너에 문제가 생겨도 다른 것들에 영향을 미치지 않는다.
- 어떤 기기에서든 안정적으로 실행되어 개발, 테스트, 확장 가능한 배포에 적합하다.

### VM vs 컨테이너 핵심 차이

| 비교 항목 | 가상 머신 (VM) | 컨테이너 (Container) |
|----------|--------------|-------------------|
| **OS 포함 여부** | 자체 OS 전체를 포함 | 호스트 OS의 커널을 공유, 별도 OS 없음 |
| **용량** | 수 GB (OS 포함) | 수십~수백 MB (앱과 종속성만 포함) |
| **시작 시간** | 수십 초~수 분 | 거의 즉시 (초 단위) |
| **격리 수준** | 완전한 격리 (OS 수준) | 프로세스 수준 격리 |
| **주요 용도** | 전혀 다른 OS 실행, 보안 테스트 환경 | 동일 OS에서 앱을 빠르게 배포·확장 |
| **대표 도구** | VirtualBox, VMware | Docker |

> 한 줄 요약: VM은 컴퓨터 안에 컴퓨터를 통째로 만드는 것이고, 컨테이너는 앱만 담아 빠르게 실행하는 가벼운 상자다.

### 주요 용어 정리

| 용어 | 설명 |
|------|------|
| **가상화** | 하나의 물리적 컴퓨터가 여러 개의 독립적인 컴퓨터처럼 작동할 수 있도록 해주는 기술 |
| **하이퍼바이저** | 가상 컴퓨터를 생성하고 실행하는 관리 소프트웨어 |
| **실험실 머신 (VM)** | 실제 컴퓨터 안에 자체 시스템을 갖춘 완전한 가상 컴퓨터 |
| **컨테이너** | 호스트와 동일한 운영 체제를 공유하는 하나의 앱을 위한 작고 독립적인 실행 환경 |
| **컨테이너 이미지** | 컨테이너를 만드는 데 사용되는 미리 패키징된 레시피/템플릿 |
| **네트워크 포트** | 애플리케이션이 네트워크를 통해 통신하는 데 사용하는 번호가 매겨진 진입점 |

### 가상화의 주요 이점

1. 비용 절감
2. 자원 활용 개선
3. 사이버 보안을 위한 안전한 테스트 환경 제공
4. 더 빠른 배포
5. 유연성
6. 이식성 (Portability)
7. 확장성
8. 중앙 집중식 관리

## 섹션 6 — 클라우드 컴퓨팅 기초 (Cloud Computing)

### 1. 클라우드 컴퓨팅이란

클라우드 컴퓨팅은 서버 사용 및 관리 방식의 오랜 변화에 따른 결과물이다.

클라우드는 용량 부족, 높은 비용, 느린 성장 속도와 같은 일반적인 문제들을 해결하기 위해 설계되었다.

### 2. 클라우드 장점

| 장점 | 설명 |
|------|------|
| **확장성** | 애플리케이션 요구 사항 변화에 따라 손쉽게 규모를 확장하거나 축소할 수 있다. |
| **온디맨드 셀프 서비스** | 하드웨어를 기다릴 필요 없이 서버와 스토리지를 즉시 생성하거나 제거할 수 있다. |
| **사용한 만큼만 지불** | 선불금이 아닌 사용량에 따라 요금이 부과된다. |
| **보안** | 클라우드 제공업체는 강력한 보안 조치를 통해 인프라를 보호한다. |
| **고가용성** | 시스템의 일부에 장애가 발생하더라도 애플리케이션은 계속 실행된다. |
| **글로벌 접근성** | 전 세계 어디에서든 사용자가 애플리케이션에 접근할 수 있다. |

### 3. 클라우드 종류

클라우드 컴퓨팅이 제공하는 유연성 덕분에 사용자의 필요와 제어 수준에 따라 애플리케이션을 다양한 방식으로 실행할 수 있다. 클라우드 제공업체는 각기 다른 시나리오에 적합한 다양한 배포 및 사용 모델을 제공한다.

**배포 유형**

| 유형 | 주요 사용처 | 특징 |
|------|------------|------|
| **퍼블릭 클라우드** | 스타트업, 웹사이트, 글로벌 앱 | 가격이 저렴하고 확장이 용이하며 인프라 관리가 필요 없다. 거의 모든 사용 사례에 적합하다. |
| **프라이빗 클라우드** | 은행, 의료기관, 정부기관 | 민감한 데이터에 대한 더 나은 제어, 맞춤 설정 및 규정 준수를 제공한다. |
| **하이브리드 클라우드** | 전자상거래 플랫폼 등 | 민감한 데이터를 비공개로 유지하면서도 수요가 많을 때 퍼블릭 클라우드로 확장할 수 있다. |

**주요 클라우드 서비스 모델**

| 모델 | 전체 명칭 | 설명 |
|------|----------|------|
| **IaaS** | 서비스형 인프라 | 가상 서버, 스토리지, 네트워킹과 같은 기본 컴퓨팅 리소스를 임대하는 방식. 사용자는 운영 체제와 애플리케이션을 관리하고, 공급업체는 물리적 하드웨어를 관리한다. |
| **PaaS** | 서비스형 플랫폼 | 클라우드 제공업체가 인프라와 운영 체제를 관리한다. 사용자는 서버에 대한 걱정 없이 애플리케이션 구축, 배포, 실행에 집중할 수 있다. |
| **SaaS** | 서비스형 소프트웨어 | 인터넷을 통해 완전한 애플리케이션을 사용한다. 서비스 제공업체가 모든 것을 관리하고, 사용자는 브라우저나 앱(예: Gmail, Zoom)을 통해 소프트웨어에 접속한다. |

**주요 클라우드 공급업체**

| 공급업체 | 특징 |
|---------|------|
| **AWS (Amazon Web Services)** | 가장 광범위한 서비스와 글로벌 네트워크를 보유한 업계 선두 기업 |
| **Microsoft Azure** | 기업 및 하이브리드 클라우드 환경에서 강력한 경쟁력을 보유 |
| **Google Cloud Platform (GCP)** | 강력한 데이터 분석 기능과 머신러닝 도구들로 잘 알려져 있음 |
| **Alibaba Cloud** | 아시아의 주요 기업으로, 전 세계적으로 경쟁력 있는 클라우드 서비스를 제공 |
| **IBM Cloud** | 하이브리드 클라우드와 기업을 위한 맞춤형 솔루션에 중점 |
| **Oracle Cloud** | 엔터프라이즈 애플리케이션 및 데이터베이스에 특화 |

클라우드를 사용하는 이유: 손쉬운 규모 확장, 비용 절감, 안정성 유지, 하드웨어 관리 없이 제품 개선에 집중할 수 있기 때문이다.

### 4. AWS EC2 기초 개념

클라우드 리소스를 쉽게 생성하고 관리하는 방법을 이해하기 위한 기초 개념이다.

**AWS 기본 개념**

| 용어 | 설명 |
|------|------|
| **EC2 (가상 컴퓨터/서버)** | 클라우드 상의 가상 컴퓨터. 실제 컴퓨터처럼 CPU와 RAM을 갖추고 있으며 애플리케이션을 실행할 수 있다. EC2 인스턴스를 추가할 때마다 환경에 컴퓨터가 추가되는 것이다. |
| **인스턴스 유형 (예: t2, t3, m5)** | 가상 컴퓨터의 성능을 나타낸다. CPU와 RAM 용량이 더 큰 인스턴스는 가격이 더 높다. |

- 인스턴스 크기가 클수록 더 많은 성능과 더 높은 비용이 발생한다.
- 소규모인 경우 = 전력 소모 감소 + 비용 절감

### 5. 주요 용어 정리

| 용어 | 설명 |
|------|------|
| **퍼블릭 클라우드** | 여러 사람과 기업이 공유하는 인터넷을 통해 접속하는 클라우드 서비스 |
| **프라이빗 클라우드** | 특정 기업만을 위해 구축된 클라우드로, 더 많은 제어 권한과 보안을 확보할 수 있다. |
| **하이브리드 클라우드** | 퍼블릭 클라우드와 프라이빗 클라우드가 혼합되어 서로 연동하고 데이터를 공유할 수 있는 클라우드 |
| **IaaS** | 클라우드에서 서버 및 스토리지와 같은 기본 컴퓨팅 리소스를 임대하는 서비스 |
| **PaaS** | 서버 관리 없이 바로 앱을 구축하고 실행할 수 있는 환경을 제공하는 서비스 |
| **SaaS** | Gmail이나 Zoom처럼 아무것도 설치하지 않고 온라인에서 사용하는 소프트웨어 |
| **EC2** | AWS의 클라우드 가상 컴퓨터로, 필요할 때마다 신속하게 생성·사용·크기 조정이 가능하다. |

## 실습 기록

### 실습 1 — VM 애플리케이션 배포 체험

**내용**: 가상 머신 환경에서 애플리케이션 배포 과정 체험

가상화가 애플리케이션 배포를 간소화하고 가속화하여, 애플리케이션을 일관되게 빠르고 안전하게 실행할 수 있는 방법을 제공하는 방식을 확인함.

**배운 점**
1. 하나의 물리 서버에서 VM 여러 개를 동시에 실행할 수 있다. 각 VM은 완전히 독립적으로 작동한다.
2. VM 위에 애플리케이션을 올리는 방식은 물리 서버를 새로 설정하는 것보다 배포 속도가 훨씬 빠르다.
3. VM이 망가져도 다른 VM이나 물리 서버에는 영향을 주지 않아 안전한 테스트 환경으로 적합하다.

### 실습 2 — AWS EC2 인스턴스 개념 실습

**내용**: TryHackMe 실습 환경에서 AWS EC2 클라우드 인스턴스 기본 개념 확인

- 인스턴스 유형(t2, t3, m5 등)별 성능과 비용 차이 이해
- 클라우드 리소스 생성·관리 방식 체험

**배운 점**
1. EC2 인스턴스는 클라우드 위의 가상 컴퓨터다. 물리 서버를 주문하면 며칠 걸리지만 EC2는 수분 내에 생성된다.
2. 인스턴스 유형(t2, t3, m5 등)에 따라 CPU·RAM 성능이 달라지고 비용도 달라진다. 작은 유형일수록 비용이 적게 든다.
3. 필요에 따라 인스턴스 크기를 바꾸거나 즉시 삭제할 수 있다. 물리 서버는 이게 불가능하다.
