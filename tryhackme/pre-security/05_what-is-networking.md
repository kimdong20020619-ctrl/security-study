## 섹션 1 — What is Networking? (네트워킹이란?)

### Task 1 — 네트워킹이란 무엇인가?

네트워크란 간단히 말해 **서로 연결된 것들**을 의미한다. 꼭 컴퓨터가 아니어도 된다. 일상 속에도 네트워크는 이미 존재한다.

**현실 속 네트워크 예시**

| 예시 | 설명 |
|------|------|
| 도시 대중교통 시스템 | 버스·지하철 노선이 도시 곳곳을 연결하는 네트워크 |
| 국가 전력망 | 발전소에서 각 가정까지 전기를 공급하는 인프라 네트워크 |
| 이웃 관계 | 사람들이 서로 연결되어 인사하고 소통하는 사회적 네트워크 |
| 우편 시스템 | 편지·소포를 주고받는 물류 네트워크 |

컴퓨터 분야에서 네트워킹도 동일한 개념이다. 다만 사람 대신 **기술 기기들이 연결**된다는 점이 다르다. 스마트폰을 예로 들면, 우리가 스마트폰을 사용하는 이유 중 하나는 다양한 정보에 접근하기 위해서인데, 이는 스마트폰이 다른 기기와 네트워크로 연결되어 있기 때문이다.

컴퓨터 네트워크의 규모는 굉장히 유연하다. **최소 2개** 장치로도 네트워크가 성립하고, 수십억 개의 장치가 연결된 거대한 네트워크도 존재한다. 노트북, 휴대폰, 보안 카메라, 신호등, 심지어 농업 장비까지 모두 네트워크에 포함될 수 있다.

이처럼 네트워크는 현대 사회 깊숙이 자리잡고 있으며, 사이버 보안 분야에서도 네트워크는 핵심 개념이다.

### Task 2 — 인터넷이란 무엇인가?

인터넷은 **작은 네트워크들이 모여 이뤄진 거대한 네트워크**다.

앞서 Alice, Bob, Jim이 서로 연결된 작은 네트워크를 형성했다고 했다. 여기서 Alice의 친구 관계가 넓어지면, 이전에 연락하던 친구 네트워크에 새로운 친구 네트워크가 추가된다. 이렇게 작은 네트워크들이 서로 연결되면서 점점 커지고, 그 결과물이 바로 인터넷이다.

**인터넷의 역사**

| 연도 | 사건 | 의미 |
|------|------|------|
| 1960년대 | **ARPANET** 프로젝트 시작 | 미국 학술·군사 목적으로 처음 만들어진 인터넷의 전신 |
| 1989년 | **Tim Berners-Lee**가 **WWW (World Wide Web)** 개발 | 일반인도 인터넷을 쉽게 사용할 수 있는 웹 탄생 |
| 현재 | 전 세계 정보 저장·공유 플랫폼 | 수십억 명이 사용하는 거대한 글로벌 네트워크 |

인터넷에서 여러 작은 네트워크들이 어떻게 연결되는지 구조를 이해하면, 네트워크는 크게 두 가지 유형으로 나눌 수 있다.

**네트워크 두 가지 유형**

| 유형 | 설명 | 예시 |
|------|------|------|
| **사설 네트워크 (Private Network)** | 내부 기기들끼리만 연결된 폐쇄적 네트워크 | 집 Wi-Fi, 회사 내부 네트워크 |
| **공용 네트워크 (Public Network)** | 인터넷을 통해 외부와 연결되는 개방적 네트워크 | 인터넷 자체 |

인터넷의 구조는 여러 개의 작은 사설 네트워크들이 공용 인터넷을 통해 서로 연결된 형태다. 내 집의 사설 네트워크와 친구 집의 사설 네트워크가 인터넷이라는 공용 통로를 거쳐 연결된다고 생각하면 된다.

### Task 3 — 네트워크상의 장치 식별

네트워크 상에서 기기들이 서로 통신하려면, **누가 누구인지 식별**할 수 있어야 한다. 누구와 대화하는지 모르면 소통이 불가능하기 때문이다.

장치를 식별하는 방식은 인간과 매우 유사하다. 인간에게 이름과 지문이 있는 것처럼, 네트워크 장치에도 두 가지 식별 수단이 있다.

| 인간 | 장치 | 특징 |
|------|------|------|
| 이름 | **IP 주소** | 변경 가능 |
| 지문 | **MAC 주소** | 고유값 (단, 스푸핑으로 위조 가능) |

---

#### IP 주소 (Internet Protocol Address)

IP 주소는 **네트워크 상에서 장치를 일정 기간 동안 식별**하는 주소다. 중요한 점은 "일정 기간 동안"이라는 것인데, IP 주소는 상황에 따라 변경될 수 있고 다른 장치에 재할당될 수도 있다.

IP 주소는 **4개의 옥텟(Octet)** 으로 구성된다. 옥텟이란 0부터 255까지의 숫자 하나를 의미하며, 이 4개가 점(`.`)으로 구분되어 하나의 IP 주소를 이룬다.

```
192  .  168  .  1  .  1
옥텟#1   옥텟#2  옥텟#3  옥텟#4
(0~255) (0~255) (0~255) (0~255)
```

같은 네트워크 내에서 두 장치가 동시에 동일한 IP 주소를 사용할 수 없다. IP 주소는 프로토콜이라고 하는 표준 규칙을 따르며, 이 덕분에 수많은 장치들이 동일한 언어로 통신할 수 있다.

**공용 IP vs 사설 IP**

장치가 어디에 있느냐에 따라 공용 IP 또는 사설 IP 중 하나를 가진다.

| 구분 | 역할 | 특징 |
|------|------|------|
| **사설 IP (Private)** | 사설 네트워크 내부에서 장치를 식별 | 같은 네트워크 내 기기들끼리 사용 |
| **공용 IP (Public)** | 인터넷에서 장치를 식별 | ISP(인터넷 서비스 제공업체)가 제공, 월 요금에 포함 |

같은 사설 네트워크에 있는 두 기기는 서로 다른 사설 IP를 가지지만, 인터넷으로 나갈 때는 동일한 공용 IP를 사용한다. 아래 예시처럼 집에 있는 두 PC가 각자 다른 사설 IP를 가지면서도, 외부에서 볼 때는 같은 공용 IP로 보이는 것이다.

| 장치 이름 | IP 주소 | 유형 |
|----------|---------|------|
| DESKTOP-KJE57FD | 192.168.1.77 | 사설 |
| DESKTOP-KJE57FD | 86.157.52.21 | 공용 |
| CMNatic-PC | 192.168.1.74 | 사설 |
| CMNatic-PC | 86.157.52.21 | 공용 |

**IPv4 vs IPv6**

IP 주소에는 버전이 있다. 현재 주로 쓰이는 **IPv4**는 주소를 2³² = 약 42억 9천만 개밖에 만들 수 없다. 문제는 인터넷에 연결되는 기기가 폭발적으로 늘어나면서 이 주소가 고갈되기 시작했다는 것이다. 시스코(Cisco)는 2021년 말까지 인터넷 연결 기기가 약 500억 개에 달할 것으로 추산했다. 42억 개의 주소로는 턱없이 부족한 숫자다.

이 문제를 해결하기 위해 등장한 것이 **IPv6**다. IPv6는 2¹²⁸ = 340조 개 이상의 주소를 지원하며, 더 효율적인 방법론을 도입해 성능도 개선되었다.

| 버전 | 주소 수 | 형식 예시 | 특징 |
|------|--------|-----------|------|
| **IPv4** | 2³² ≈ 42억 9천만 개 | `86.157.52.21` | 현재 주로 사용, 고갈 문제 발생 |
| **IPv6** | 2¹²⁸ ≈ 340조 개 이상 | `2a00:22c6:a531:c509:470f:ccce:c36b:f64d` | IPv4 부족 문제 해결, 효율성 향상 |

#### MAC 주소 (Media Access Control Address)

MAC 주소는 장치의 마더보드 안에 있는 **네트워크 인터페이스(마이크로칩)에 제조 공장에서 직접 부여되는 고유 주소**다. IP 주소가 나중에 할당되는 것과 달리, MAC 주소는 하드웨어 자체에 새겨진다고 볼 수 있다.

MAC 주소는 **12자리 16진수**로 구성되며, 두 자리씩 콜론(`:`)으로 구분한다.

```
a4 : c3 : f0 : 85 : ac : 2d
|___앞 6자리___|  |___뒤 6자리___|
  제조사 식별        고유 번호
```

| 구분 | 자리 | 의미 |
|------|------|------|
| 앞 6자리 | `a4:c3:f0` | 네트워크 인터페이스를 만든 제조사 식별 |
| 뒤 6자리 | `85:ac:2d` | 해당 장치의 고유 번호 |

**MAC 스푸핑 (MAC Spoofing)**

MAC 주소는 하드웨어에 새겨져 있지만, 소프트웨어를 이용해 **위조(스푸핑)** 하는 것이 가능하다. 스푸핑이란 자신의 MAC 주소를 다른 장치의 MAC 주소처럼 속이는 행위다.

예를 들어, 방화벽이 "관리자의 MAC 주소에서 오는 통신만 허용"하도록 설정되어 있다면, 공격자가 그 MAC 주소를 스푸핑해서 방화벽을 속일 수 있다.

실제 사례로, 카페나 호텔의 유료 Wi-Fi를 생각해보자.

- Alice는 Wi-Fi 요금을 냈다 → Alice의 MAC은 허용
- Bob은 요금을 내지 않았다 → Bob의 MAC은 차단
- Bob이 자신의 MAC 주소를 Alice의 것으로 스푸핑하면 → 방화벽이 Alice로 착각해 통과

> **보안 시사점**: MAC 주소만으로 신뢰를 판단하는 보안 설계는 취약하다. MAC은 위조될 수 있다.

| 퀴즈 | 정답 |
|------|------|
| "IP"는 무슨 약자인가? | `Internet Protocol` |
| IP 주소의 각 부분은 무엇이라고 불리나? | `Octet` |
| IPv4 주소는 몇 자리 숫자로 구성되어 있나? | `4` |
| "MAC"은 무슨 약자인가? | `Media Access Control` |
| MAC 스푸핑 실습 플래그 | `THM{YOU_GOT_ON_TRYHACKME}` |

### Task 4 — 핑 (Ping / ICMP)

Ping은 우리가 사용할 수 있는 **가장 기본적인 네트워크 도구** 중 하나다. 네트워크 상에서 특정 장치나 웹사이트에 연결이 되는지, 그리고 연결 속도가 얼마나 되는지 확인할 때 사용한다.

Ping은 **ICMP(Internet Control Message Protocol, 인터넷 제어 메시지 프로토콜)** 패킷을 사용한다. 동작 원리는 간단하다. 내가 ICMP 에코(echo) 패킷을 대상 장치로 보내면, 그 장치가 ICMP 에코 응답을 돌려보낸다. 이 왕복 시간을 측정해서 연결 상태를 확인하는 것이다.

Ping은 Windows, Linux, macOS에 기본으로 내장되어 있어서 별도 설치 없이 바로 사용할 수 있다.

**명령어 형식**

```bash
ping <IP 주소 또는 웹사이트 URL>
```

**실제 실행 예시**

아래는 `192.168.1.254`로 핑을 보낸 결과다.

```
cmnatic@CMNatic-THM-LAPTOP:~$ ping 192.168.1.254
PING 192.168.1.254 (192.168.1.254) 56(84) bytes of data.
64 bytes from 192.168.1.254: icmp_seq=1 ttl=63 time=2.18 ms
64 bytes from 192.168.1.254: icmp_seq=2 ttl=63 time=2.53 ms
64 bytes from 192.168.1.254: icmp_seq=3 ttl=63 time=2.15 ms
64 bytes from 192.168.1.254: icmp_seq=4 ttl=63 time=3.34 ms
64 bytes from 192.168.1.254: icmp_seq=5 ttl=63 time=10.3 ms
64 bytes from 192.168.1.254: icmp_seq=6 ttl=63 time=4.45 ms
^C
--- 192.168.1.254 ping statistics ---
6 packets transmitted, 6 received, 0% packet loss, time 5008ms
rtt min/avg/max/mdev = 2.152/4.160/10.313/2.864 ms
```

**결과 읽는 방법**

| 항목 | 값 | 의미 |
|------|-----|------|
| `6 packets transmitted` | 6개 전송 | 총 6개의 ICMP 패킷을 보냄 |
| `6 received` | 6개 수신 | 6개 모두 응답이 돌아옴 |
| `0% packet loss` | 손실 없음 | 연결 상태 정상 |
| `time=2.18 ms` | 응답 시간 | 이 패킷이 왕복하는 데 걸린 시간 |
| `avg = 4.160 ms` | 평균 응답 시간 | 전체 평균 약 4.16ms |

> 패킷 손실이 있거나 응답 시간이 매우 길다면 네트워크 연결에 문제가 있다는 신호다.

| 퀴즈 | 정답 |
|------|------|
| ping은 어떤 프로토콜을 사용하나? | `ICMP` |
| 10.10.10.10으로 핑을 보내는 명령은? | `ping 10.10.10.10` |
| 8.8.8.8로 핑을 보내면 어떤 플래그가 표시되나? | `THM{I_PINGED_THE_SERVER}` |

## 섹션 2 — Intro to LAN (LAN 소개)

### Task 1 — LAN 토폴로지 소개

**토폴로지(Topology)** 란 네트워크의 **실제 설계 또는 구조**를 의미하는 용어다. 수년에 걸쳐 다양한 네트워크 설계 방식이 실험되고 구현되어 왔다. 각 토폴로지마다 장단점이 다르기 때문에, 상황에 맞는 설계를 선택하는 것이 중요하다.

---

#### 스타 토폴로지 (Star Topology)

스타 토폴로지는 모든 장치가 **중앙의 스위치나 허브**를 통해 개별적으로 연결되는 구조다. 장치 간 직접 연결이 아니라, 반드시 중앙 장치를 거쳐서 데이터가 전달된다. 오늘날 가장 널리 사용되는 방식이다.

**장점**

- **확장성이 뛰어나다**: 새 장치를 추가할 때 중앙 장치에 포트 하나만 연결하면 되므로, 네트워크 수요가 늘어나도 쉽게 대응할 수 있다.
- **안정성이 높다**: 한 장치에 문제가 생겨도 다른 장치들은 영향을 받지 않는다.

**단점**

- **비용이 많이 든다**: 더 많은 케이블과 전용 네트워킹 장비(스위치/허브)가 필요하다.
- **유지보수가 복잡하다**: 네트워크 규모가 커질수록 유지보수 작업량도 늘어나고, 장애 발생 시 문제 파악이 어려워진다.
- **중앙 장치 의존성**: 중앙 스위치/허브가 고장나면 연결된 모든 장치가 통신 불능 상태가 된다. 다만 이런 중앙 장치는 일반적으로 내구성이 뛰어나 고장 가능성은 낮다.

---

#### 버스 토폴로지 (Bus Topology)

버스 토폴로지는 **단일 백본 케이블(Backbone Cable)** 에 모든 장치가 가지처럼 연결되는 구조다. 마치 나무의 줄기에서 잎이 뻗어나오는 모양과 비슷하다.

**장점**

- **비용이 저렴하다**: 별도의 전용 네트워킹 장비 없이 하나의 케이블만으로 구성할 수 있어 설정이 쉽고 비용 효율적이다.

**단점**

- **병목 현상 발생**: 모든 데이터가 같은 케이블을 통해 전송되기 때문에, 여러 장치가 동시에 데이터를 요청하면 속도가 급격히 느려진다.
- **문제 해결 어려움**: 모든 데이터가 같은 경로를 지나가므로, 어느 장치에서 문제가 발생했는지 파악하기가 매우 까다롭다.
- **이중화 없음**: 백본 케이블이 단선되면 해당 케이블에 연결된 모든 장치가 통신 불능이 된다. 장애를 대비한 백업 경로가 없다.

---

#### 링 토폴로지 (Ring Topology / Token Topology)

링 토폴로지는 장치들이 **루프(고리) 형태로 서로 직접 연결**되는 구조다. 토큰 토폴로지라고도 불린다. 스타 토폴로지처럼 중앙 장치가 필요 없어서 케이블 사용량이 적고, 전용 하드웨어 의존도도 낮다.

데이터는 루프를 따라 한 방향으로만 이동하며, 중간에 있는 장치들이 데이터를 릴레이처럼 전달하는 역할을 한다. 흥미로운 점은, 장치가 자신이 보낼 데이터가 있으면 다른 장치의 데이터를 전달하기 전에 자기 데이터를 먼저 보낸다는 것이다.

**장점**

- **병목 현상이 적다**: 버스 토폴로지와 달리 동시에 많은 트래픽이 몰려도 비교적 잘 처리한다.
- **오류 파악이 쉽다**: 데이터 이동 방향이 하나뿐이라 어디서 문제가 생겼는지 추적하기 비교적 쉽다.

**단점**

- **네트워크 효율 저하**: 데이터가 목적지에 도달하기 전에 여러 장치를 거쳐야 하므로 전송 속도가 느릴 수 있다.
- **전체 마비 위험**: 케이블 단선이나 장치 고장이 발생하면 루프 전체가 끊기므로 네트워크 전체가 마비될 수 있다.

---

**토폴로지 3종 비교**

| 항목 | 스타 (Star) | 버스 (Bus) | 링 (Ring) |
|------|------------|-----------|----------|
| **구조** | 중앙 장치에 개별 연결 | 단일 백본 케이블 공유 | 루프 형태 직접 연결 |
| **비용** | 높음 | 낮음 | 중간 |
| **확장성** | 매우 좋음 | 나쁨 | 보통 |
| **병목 현상** | 낮음 | 높음 | 낮음 |
| **단선 시 영향** | 해당 장치만 영향 | 전체 불능 | 전체 마비 |
| **유지보수** | 복잡 | 어려움 | 보통 |
| **현재 사용** | 가장 널리 사용 | 거의 사용 안 함 | 드물게 사용 |

---

#### 스위치 (Switch)

스위치는 **로컬 네트워크 내에서 여러 장치를 연결하는 전용 장치**다. 컴퓨터, 프린터, 카메라 등 다양한 장치를 스위치의 포트에 꽂아 연결한다.

스위치는 일반적으로 기업, 학교처럼 연결해야 할 장치가 많은 환경에서 사용되며, 포트 수에 따라 4개, 8개, 16개, 24개, 32개, 64개 등 다양한 제품이 있다.

스위치가 허브(Hub)보다 효율적인 이유는 **패킷을 전달하는 방식**이 다르기 때문이다.

| 장치 | 패킷 전달 방식 | 특징 |
|------|--------------|------|
| **허브 (Hub)** | 연결된 모든 포트로 패킷을 전송 | 불필요한 트래픽 발생, 비효율적 |
| **스위치 (Switch)** | 어떤 장치가 어떤 포트에 있는지 기억하고, 의도한 대상 포트로만 전송 | 트래픽 절감, 효율적 |

또한 스위치와 라우터를 서로 연결하면 **네트워크 이중화(Redundancy)** 를 구현할 수 있다. 이중화란 데이터가 이동할 수 있는 경로를 여러 개 만들어 두는 것으로, 한 경로에 장애가 발생해도 다른 경로로 통신을 유지할 수 있다. 속도가 약간 느려지는 단점이 있지만, 서비스 중단을 막는다는 점에서 충분히 감수할 만하다.

---

#### 라우터 (Router)

라우터의 역할은 **서로 다른 네트워크들을 연결하고, 네트워크 간에 데이터를 전달**하는 것이다. 라우터는 **라우팅(Routing)** 이라는 기술을 사용해서 이 일을 수행한다. (그래서 이름이 "라우터"다.)

라우팅이란 데이터가 출발지에서 목적지까지 네트워크를 통해 이동하는 과정이다. 여기서 중요한 점은 라우터가 데이터가 지나갈 **최적 경로를 생성**한다는 것이다. 장치가 여러 경로로 연결되어 있을 때, 라우터는 어떤 경로로 보내는 것이 가장 효율적인지 판단한다.

| 항목 | 스위치 | 라우터 |
|------|--------|--------|
| **역할** | 같은 네트워크 내 장치 연결 | 서로 다른 네트워크 간 데이터 전달 |
| **작동 범위** | 로컬 네트워크 (LAN) 내부 | 네트워크와 네트워크 사이 |
| **핵심 기능** | 포트별 패킷 전달 | 라우팅 (최적 경로 탐색) |

| 퀴즈 | 정답 |
|------|------|
| LAN은 무슨 약자인가요? | `Local Area Network` |
| 라우터가 수행하는 작업을 나타내는 동사는? | `Routing` |
| 로컬 네트워크에서 여러 장치를 중앙에서 연결하고 올바른 위치로 전송하는 장치는? | `Switch` |
| 설정 비용이 가장 효율적인 토폴로지는? | `Bus Topology` |
| 설치 및 유지 관리에 비용이 가장 많이 드는 토폴로지는? | `Star Topology` |
| 대화형 실습 플래그 | `THM{TOPOLOGY_FLAWS}` |

### Task 2 — 서브넷팅 입문 (Subnetting)

서브넷팅이란 **하나의 네트워크를 더 작은 네트워크로 분할하는 기술**이다. 케이크에 비유하면 이해하기 쉽다. 케이크(네트워크) 한 판이 있을 때, 여러 명에게 나눠주기 위해 조각을 자르는 행위가 바로 서브넷팅이다. 누가 어떤 조각을 가져갈지 결정하고, 그 조각을 따로 떼어 놓는 것이다.

예를 들어 한 기업에 회계팀, 재무팀, 인사팀이 있다고 하자. 현실에서도 각 부서마다 보내야 할 서류와 정보가 다르듯, 네트워크에서도 부서별로 트래픽을 분리해서 관리하는 것이 필요하다. 이를 가능하게 해주는 것이 서브넷팅이다.

서브넷팅은 **서브넷 마스크(Subnet Mask)** 라는 숫자로 네트워크를 분할한다. 서브넷 마스크도 IP 주소처럼 **4바이트(32비트)** , 즉 4개의 옥텟으로 이루어져 있으며, 각 옥텟은 0~255 범위의 값을 가진다.

**서브넷이 IP 주소를 사용하는 3가지 방식**

서브넷 안에서 IP 주소는 세 가지 역할로 나뉜다.

| 유형 | 목적 | 상세 설명 | 예시 |
|------|------|-----------|------|
| **네트워크 주소** | 네트워크의 시작점 표시 | 이 주소 자체는 장치에 할당되지 않고, 네트워크가 존재함을 나타내는 데 사용된다. IP가 192.168.1.100인 장치는 192.168.1.0 네트워크에 속한다. | 192.168.1.0 |
| **호스트 주소** | 서브넷 내 장치 식별 | 서브넷 안의 각 장치에 고유하게 할당되는 주소. 실제 기기들이 사용하는 주소다. | 192.168.1.100 |
| **기본 게이트웨이** | 다른 네트워크로 데이터 전송 | 이 네트워크(192.168.1.x)에 없는 곳으로 데이터를 보낼 때 거쳐가는 장치의 주소. 보통 첫 번째(.1) 또는 마지막(.254) 호스트 주소를 사용한다. | 192.168.1.254 |

가정의 소규모 네트워크에서는 동시에 254개 이상의 장치를 연결할 일이 거의 없으므로 서브넷이 하나면 충분하다. 하지만 기업이나 사무실처럼 PC, 프린터, 카메라, 센서 등 수많은 장치가 있는 환경에서는 서브넷팅이 필수다.

**서브넷팅의 이점**

| 이점 | 설명 |
|------|------|
| **효율성 (Efficiency)** | 부서별로 트래픽을 분리해 불필요한 데이터가 다른 부서로 넘어가지 않도록 최적화 |
| **보안 (Security)** | 서로 다른 네트워크 구간을 분리해 민감한 정보가 다른 구간으로 새어나가지 않도록 보호 |
| **완전한 제어 (Full Control)** | 네트워크 구조를 세밀하게 설계하고 관리할 수 있음 |

> **실제 예시**: 카페에 두 개의 네트워크가 있다고 하자. ① 직원·계산대·시설 기기용 네트워크 ② 손님이 사용하는 핫스팟 네트워크. 서브넷팅을 사용하면 이 둘을 분리하면서도 같은 인터넷 회선을 공유할 수 있다. 손님이 직원 네트워크에 접근하는 것을 막을 수 있다.

| 퀴즈 | 정답 |
|------|------|
| 네트워크를 더 작은 부분으로 나누는 것은? | `Subnetting` |
| 서브넷 마스크는 몇 비트로 구성되어 있나? | `32` |
| 서브넷 마스크 각 옥텟의 범위는? | `0-255` |
| 네트워크의 시작점을 나타내는 주소는? | `Network Address` |
| 네트워크 내 장치를 식별하는 주소는? | `Host Address` |
| 다른 네트워크로 데이터를 전송하는 장치를 지칭하는 용어는? | `Default Gateway` |

### Task 3 — ARP (Address Resolution Protocol)

앞서 장치는 **IP 주소**와 **MAC 주소** 두 가지 식별자를 가질 수 있다는 것을 배웠다. **ARP(주소 확인 프로토콜)** 는 이 두 식별자를 서로 연결하는 방법을 설명하는 프로토콜이다.

간단히 말해, ARP는 네트워크 상에서 장치들이 **"이 IP 주소를 가진 장치의 MAC 주소가 뭐야?"** 라고 물어보고 답을 얻는 방법이다.

**ARP 캐시**

네트워크 내의 각 장치는 **ARP 캐시(Cache)** 라고 불리는 저장소를 가지고 있다. 여기에는 네트워크 상의 다른 장치들의 IP 주소와 MAC 주소 매핑 정보가 저장된다. 한 번 확인한 정보를 캐시에 저장해 두면, 다음에 같은 장치에 데이터를 보낼 때 다시 물어볼 필요가 없어서 효율적이다.

**ARP 동작 과정**

두 장치가 처음으로 통신하려 할 때, 서로의 MAC 주소를 모르는 상태다. 이때 ARP가 2단계로 동작한다.

| 단계 | 메시지 유형 | 방향 | 내용 |
|------|------------|------|------|
| **1단계** | **ARP Request** (요청) | 요청 장치 → 네트워크 전체 브로드캐스트 | "192.168.1.10이라는 IP 주소를 가진 장치의 MAC 주소는 무엇입니까?" 라고 네트워크의 모든 장치에게 물음 |
| **2단계** | **ARP Reply** (응답) | 해당 장치 → 요청 장치 (유니캐스트) | 해당 IP를 가진 장치만 응답함. "저입니다. 제 MAC 주소는 1A:AC:33:12:8B:29입니다" |
| **저장** | ARP 캐시 업데이트 | 요청 장치 내부 | IP↔MAC 매핑을 ARP 캐시에 저장, 이후 통신 시 재사용 |

실제 패킷을 보면 아래와 같다.

```
① ARP Request (브로드캐스트)
   SRC MAC: 01:00:AB:78:99:33      ← 요청 장치의 MAC
   DST MAC: FF:FF:FF:FF:FF:FF      ← 브로드캐스트 (모든 장치에게)
   MSG: Who has IP Address 192.168.1.10?

② ARP Reply (유니캐스트)
   SRC MAC: 1A:AC:33:12:8B:29      ← 응답 장치의 MAC
   DST MAC: 01:00:AB:78:99:33      ← 요청했던 장치에게만
   MSG: I have IP Address 192.168.1.10
```

**식별자 정리**

| 식별자 | 종류 | 설명 |
|--------|------|------|
| **MAC 주소** | 물리적(Physical) 식별자 | 장치 하드웨어에 새겨진 고유 주소 |
| **IP 주소** | 논리적(Logical) 식별자 | 네트워크 상에서 소프트웨어적으로 할당되는 주소 |

| 퀴즈 | 정답 |
|------|------|
| ARP는 무슨 약자인가? | `Address Resolution Protocol` |
| IP 주소 소유 여부를 묻는 ARP 패킷 유형은? | `Request` |
| 네트워크 상에서 장치를 물리적으로 식별하는 주소는? | `MAC Address` |
| 네트워크 상에서 장치를 논리적으로 식별하는 주소는? | `IP Address` |

### Task 4 — DHCP (Dynamic Host Configuration Protocol)

장치에 IP 주소를 부여하는 방법은 두 가지다.

| 방법 | 설명 |
|------|------|
| **수동 할당 (Static)** | 관리자가 장치에 직접 IP 주소를 입력해 설정 |
| **자동 할당 (Dynamic)** | DHCP 서버가 자동으로 IP 주소를 할당 |

가장 일반적인 방법은 **DHCP(동적 호스트 구성 프로토콜)** 를 이용한 자동 할당이다. 우리가 카페에서 Wi-Fi에 연결할 때 IP 주소를 직접 입력하지 않아도 자동으로 연결되는 이유가 DHCP 덕분이다.

**DHCP 동작 4단계**

장치가 네트워크에 처음 연결되면, 아직 IP 주소가 없는 상태다. 이때 DHCP가 아래 4단계 과정으로 자동으로 IP 주소를 배정해준다.

| 단계 | 패킷 유형 | 방향 | 내용 |
|------|----------|------|------|
| **1단계** | **DHCP Discover** | 장치 → 네트워크 전체 브로드캐스트 | "안녕하세요! 저 새로 들어온 장치인데, IP 주소를 줄 수 있는 DHCP 서버 있나요?" |
| **2단계** | **DHCP Offer** | DHCP 서버 → 장치 | "있어요! 이 IP 주소(예: 192.168.1.10)를 사용할 수 있어요" |
| **3단계** | **DHCP Request** | 장치 → DHCP 서버 | "좋아요, 그 IP 주소 192.168.1.10 사용하겠습니다!" |
| **4단계** | **DHCP ACK** | DHCP 서버 → 장치 | "확인했습니다. 앞으로 24시간 동안 그 IP 주소를 사용하세요" |

이 과정을 거치면 장치는 네트워크에서 사용할 IP 주소를 갖게 된다.

> **IP 임대(Lease)**: DHCP가 부여하는 IP 주소는 영구적이지 않다. 일정 기간(임대 기간) 동안만 사용할 수 있으며, 기간이 지나면 갱신 과정을 거치거나 다른 IP 주소를 받게 된다. 위 예시에서 "24시간"이 임대 기간이다.

| 퀴즈 | 정답 |
|------|------|
| IP 주소를 가져오기 위해 사용하는 DHCP 패킷 유형은? | `DHCP Discover` |
| IP 주소 할당받으면 장치가 전송하는 패킷 유형은? | `DHCP Request` |
| DHCP 서버에서 기기로 전송되는 마지막 패킷은? | `DHCP ACK` |

## 섹션 3 — OSI Model (OSI 모델)

### Task 1 — OSI 모델이란?

**OSI 모델(Open Systems Interconnection Model, 개방형 시스템 상호 연결 모델)** 은 네트워킹에서 가장 핵심적인 개념 중 하나다. 이 모델은 네트워크에 연결된 모든 장치가 데이터를 어떻게 송수신하고 해석하는지를 규정하는 **표준 프레임워크**를 제공한다.

OSI 모델이 중요한 이유는 **서로 다른 제조사, 서로 다른 운영체제, 서로 다른 설계를 가진 장치들이 문제없이 통신할 수 있도록** 공통 규칙을 정해주기 때문이다. OSI 모델을 따르는 데이터는 어떤 장치에서 보내든 다른 장치에서 이해할 수 있다.

OSI 모델은 **7개의 계층(Layer)** 으로 구성되며, 아래와 같이 번호가 7에서 1로 내려가는 순서로 배열된다.

| 계층 번호 | 계층 이름 | 영문명 |
|----------|----------|--------|
| 7 | 응용 계층 | Application |
| 6 | 표현 계층 | Presentation |
| 5 | 세션 계층 | Session |
| 4 | 전송 계층 | Transport |
| 3 | 네트워크 계층 | Network |
| 2 | 데이터 링크 계층 | Data Link |
| 1 | 물리 계층 | Physical |

데이터가 각 계층을 통과할 때마다 특정 처리가 수행되고 정보가 추가된다. 이 과정을 **캡슐화(Encapsulation)** 라고 한다. 데이터를 보낼 때는 7계층 → 1계층 순으로 내려가며 각 계층이 헤더 정보를 추가하고, 받을 때는 반대로 1계층 → 7계층 순으로 올라가며 각 헤더를 해석하고 벗겨낸다.

| 퀴즈 | 정답 |
|------|------|
| "OSI 모델"에서 "OSI"의 뜻은? | `Open Systems Interconnection` |
| OSI 모델은 몇 개의 계층으로 구성되어 있나? | `7` |
| 데이터에 새로운 정보가 추가될 때 사용하는 용어는? | `Encapsulation` |

---

### Task 2 — Layer 1: Physical (물리 계층)

물리 계층은 OSI 7계층 중 **가장 아래에 위치하는 최하위 계층**이다. 이 계층은 이름 그대로 **네트워킹에 사용되는 하드웨어의 물리적 구성 요소**를 다룬다.

장치들은 물리 계층에서 **전기 신호(electrical signals)** 를 사용하여 데이터를 주고받는다. 이 전기 신호는 결국 **이진수 체계(Binary)**, 즉 0과 1로 이루어진 데이터다. 우리가 파일을 전송하거나 웹 페이지를 불러올 때, 그 데이터는 결국 수많은 0과 1의 전기 신호가 물리적인 매체를 타고 흐르는 것이다.

물리 계층에서 가장 대표적인 물리적 매체는 **이더넷 케이블(Ethernet Cables)** 이다. 우리가 공유기나 스위치에 꽂는 LAN 케이블이 바로 여기에 해당한다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Physical` |
| 0과 1을 사용하는 숫자 체계의 이름은? | `Binary` |
| 기기들을 연결하는 데 사용되는 케이블의 이름은? | `Ethernet Cables` |

---

### Task 3 — Layer 2: Data Link (데이터 링크 계층)

데이터 링크 계층은 **전송의 물리적 주소 지정(Physical Addressing)** 에 초점을 맞추는 계층이다. 위의 네트워크 계층(3계층)에서 패킷(원격 컴퓨터의 IP 주소가 담긴 데이터 조각)을 받아서, 그 위에 **수신 엔드포인트의 물리적 MAC 주소**를 추가하는 역할을 한다.

네트워크에 연결된 모든 컴퓨터 내부에는 **NIC(Network Interface Card, 네트워크 인터페이스 카드)** 가 탑재되어 있으며, 각 NIC에는 고유한 **MAC 주소**가 부여되어 있다. MAC 주소는 제조업체에서 카드에 문자 그대로 새겨 넣는 것이기 때문에 기본적으로 변경이 불가능하다. 다만 소프트웨어적으로 **위조(스푸핑)는 가능**하다 — 이는 섹션 1의 Task 3에서 배운 내용과 같다.

데이터 링크 계층이 중요한 이유는, 네트워크를 통해 정보가 전송될 때 "정확히 어느 물리적 장치에게 보내야 하는가"를 결정하는 것이 바로 이 MAC 주소이기 때문이다. 또한 데이터 링크 계층은 데이터를 **전송에 적합한 형식으로 포장**하는 역할도 담당한다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Data Link` |
| 네트워크에 연결된 모든 장치에 기본으로 탑재되는 하드웨어는? | `Network Interface Card` |

---

### Task 4 — Layer 3: Network (네트워크 계층)

네트워크 계층은 **라우팅(Routing)** 과 **데이터 재조립**이 이루어지는 계층이다. 라우팅이란, 분리된 작은 데이터 조각들이 목적지까지 이동하는 **최적의 경로(Optimal Path)** 를 결정하는 과정이다.

네트워크 계층에서 사용하는 주요 라우팅 프로토콜은 다음 두 가지다.

| 프로토콜 | 전체 이름 | 역할 |
|---------|----------|------|
| **OSPF** | Open Shortest Path First | 네트워크 내 최단 경로를 계산하는 프로토콜 |
| **RIP** | Routing Information Protocol | 경유하는 장치(홉) 수를 기준으로 경로를 결정하는 프로토콜 |

어떤 경로가 최적인지는 아래 세 가지 요소를 고려해서 결정한다.

- **가장 짧은 경로**: 패킷이 거쳐야 하는 장치의 수가 가장 적은 경로
- **가장 신뢰할 수 있는 경로**: 과거에 패킷 손실이 발생한 적 없는 경로
- **물리적 연결 속도가 빠른 경로**: 예를 들어 광섬유(fiber) 연결이 구리선(copper) 연결보다 훨씬 빠르다

이 계층에서 모든 것은 `192.168.1.100`과 같은 **IP 주소**를 통해 처리된다. IP 주소를 사용해 패킷을 전달할 수 있는 **라우터**는 3계층에서 작동하기 때문에 **3계층 장치(Layer 3 Device)** 라고 부른다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Network` |
| 패킷이 네트워크를 통해 최적의 경로를 따라 이동하나? (예/아니오) | `Y` |
| "OSPF"의 뜻은? | `Open Shortest Path First` |
| "RIP"의 뜻은? | `Routing Information Protocol` |
| 이 계층에서 처리되는 주소 유형은? | `IP Addresses` |

### Task 5 — Layer 4: Transport (전송 계층)

전송 계층은 장치 간 데이터 전송 방식을 결정하는 계층이다. 데이터가 전송될 때, 상황에 따라 아래 두 프로토콜 중 하나를 선택해서 사용한다.

- **TCP (Transmission Control Protocol, 전송 제어 프로토콜)**
- **UDP (User Datagram Protocol, 사용자 데이터그램 프로토콜)**

#### TCP — 신뢰성 우선

TCP는 **신뢰성과 안정성**을 최우선으로 설계된 프로토콜이다. 데이터를 송수신하는 동안 두 장치 사이에 **지속적인 연결을 유지**하며, **오류 검사(Error Checking)** 기능이 내장되어 있다. 덕분에 세션 계층(5계층)을 통해 작은 조각으로 나뉘어 보내진 데이터가 목적지에서 **같은 순서로 정확하게 재조립**되었는지 확인할 수 있다.

예를 들어, 고양이 사진을 웹 서버에서 다운받는다면, TCP는 사진을 Packet #1, #2, #3으로 나누어 보내고, 받는 쪽에서 이 조각들을 올바른 순서로 합쳐 완전한 사진으로 복원한다.

TCP는 **파일 공유, 웹 브라우징, 이메일 전송**처럼 데이터의 정확성과 완전성이 필수인 상황에서 사용된다.

#### UDP — 속도 우선

UDP는 TCP보다 발전된 기능이 적은 대신 **훨씬 빠른** 프로토콜이다. 오류 검사나 수신 확인 없이, 데이터가 상대방에게 도착하든 안 하든 관계없이 **그냥 보내버린다**. 두 장치 간의 동기화나 수신 보장이 없다.

예를 들어, 위와 같은 고양이 사진을 UDP로 보내면 Packet #2가 유실되어도 그냥 무시하고 나머지를 전달한다. 이미지 일부가 깨져 보이더라도 괜찮은 상황, 또는 빠른 전달이 더 중요한 상황에서 UDP가 유리하다.

UDP는 **비디오 스트리밍, ARP·DHCP 같은 소량 데이터 프로토콜, 소규모 파일 전송** 등에 쓰인다.

**TCP vs UDP 비교**

| 항목 | TCP | UDP |
|------|-----|-----|
| **속도** | 느림 | 빠름 |
| **신뢰성** | 높음 (오류 검사, 수신 확인) | 낮음 (전달 보장 없음) |
| **연결 방식** | 연결 기반 (Connection-oriented) | 비연결 (Connectionless) |
| **데이터 순서 보장** | 있음 | 없음 |
| **주요 용도** | 파일 다운로드, 이메일, 웹 브라우징 | 동영상 스트리밍, DNS, 실시간 게임 |
| **패킷 손실 시** | 재전송 요청 | 그냥 무시 |

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Transport` |
| TCP의 뜻은? | `Transmission Control Protocol` |
| UDP의 뜻은? | `User Datagram Protocol` |
| 데이터의 정확성을 보장하는 프로토콜은? | `TCP` |
| 데이터 수신 여부를 신경 쓰지 않는 프로토콜은? | `UDP` |
| 이메일 클라이언트는 어떤 프로토콜을 사용하나? | `TCP` |
| 파일 다운로드 애플리케이션은 어떤 프로토콜을 사용하나? | `TCP` |
| 동영상 스트리밍 애플리케이션은 어떤 프로토콜을 사용하나? | `UDP` |

### Task 6 — Layer 5: Session (세션 계층)

세션 계층은 **다른 컴퓨터와의 연결을 생성하고 유지하며 종료**하는 역할을 담당한다. 표현 계층(6계층)에서 데이터 형식이 올바르게 변환되고 나면, 세션 계층이 상대방 컴퓨터와 **연결(Connection)을 설정**하기 시작한다. 이 연결이 성공적으로 맺어지면 **세션(Session)** 이 생성되며, 연결이 유지되는 동안 세션도 활성 상태를 유지한다.

세션 계층의 또 다른 중요한 역할은, 일정 시간 동안 사용되지 않거나 연결이 끊기면 **자동으로 세션을 종료**하는 것이다. 이를 통해 불필요한 자원 낭비를 방지한다.

또한 세션에는 **체크포인트(Checkpoint)** 라는 개념이 포함될 수 있다. 예를 들어 100MB짜리 파일을 전송하는 도중에 연결이 끊겼다고 가정해보자. 체크포인트가 없다면 처음부터 다시 전송해야 하지만, 체크포인트가 있으면 마지막으로 성공한 지점부터 이어서 전송할 수 있다. 이렇게 하면 **대역폭을 절약**할 수 있다.

주목할 점은, **세션은 고유하다**는 것이다. 즉, 하나의 세션 내에서 전송 중인 데이터는 다른 세션으로 이동할 수 없다. 예를 들어 브라우저 탭 두 개를 열어 각각 다운로드하면, 각 다운로드는 서로 독립된 세션을 통해 처리된다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Session` |
| 연결이 성공적으로 설정되었을 때 사용하는 기술 용어는? | `Session` |

### Task 7 — Layer 6: Presentation (표현 계층)

표현 계층은 OSI 모델에서 **표준화가 시작되는 계층**이다. 이 계층의 핵심 역할은 **번역기(Translator)** 다. 서로 다른 소프트웨어나 시스템이 데이터를 각기 다른 형식으로 처리하더라도, 표현 계층이 이를 **공통적으로 이해할 수 있는 형식으로 변환**해준다.

예를 들어, A가 Gmail을 사용하고 B가 Outlook을 사용한다고 해도, 이메일 내용은 동일하게 표시된다. 각 클라이언트가 데이터를 받았을 때, 표현 계층이 해당 소프트웨어에 맞는 형식으로 변환하기 때문이다.

또한 표현 계층은 **보안 기능도 처리**한다. 대표적으로 **데이터 암호화(Encryption)** 가 이 계층에서 이루어진다. 웹 브라우저에서 `https://`로 시작하는 사이트에 접속할 때 사용되는 **HTTPS 암호화**가 바로 이 계층에서 처리된다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Presentation` |
| 이 계층의 주요 목적(역할)은? | `Translator` |

### Task 8 — Layer 7: Application (응용 계층)

응용 계층은 OSI 7계층 중 **가장 상위에 위치하며, 사용자와 가장 가까운 계층**이다. 우리가 매일 사용하는 이메일 클라이언트, 웹 브라우저, 파일 전송 소프트웨어 같은 프로그램들이 바로 이 계층에서 동작한다.

응용 계층은 **사용자가 데이터와 상호작용하는 방식**을 결정하는 프로토콜과 규칙이 존재하는 계층이다. 이메일 클라이언트나 브라우저 같은 소프트웨어는 **GUI(Graphical User Interface, 그래픽 사용자 인터페이스)** 를 통해 사용자가 데이터를 쉽게 송수신할 수 있도록 해준다.

응용 계층에서 작동하는 프로토콜의 대표적인 예로 **DNS(Domain Name System, 도메인 이름 시스템)** 가 있다. DNS는 우리가 브라우저에 `www.google.com`처럼 도메인 이름을 입력하면, 이를 컴퓨터가 실제로 통신에 사용하는 **IP 주소로 변환**해주는 역할을 한다. 이 덕분에 사람들은 숫자로 이루어진 IP 주소를 외울 필요 없이 기억하기 쉬운 이름으로 웹사이트에 접근할 수 있다.

| 퀴즈 | 정답 |
|------|------|
| 이 계층의 이름은? | `Application` |
| 사용자가 상호작용하는 소프트웨어를 나타내는 기술 용어는? | `Graphical User Interface` |

## 섹션 4 — Packets & Frames (패킷과 프레임)

### Task 1 — 패킷과 프레임이란?

**패킷(Packet)** 과 **프레임(Frame)** 은 둘 다 큰 데이터를 잘게 나눈 작은 조각이다. 하지만 OSI 모델에서는 이 두 개념을 명확히 구분한다.

| 구분 | OSI 계층 | 포함 정보 | 특징 |
|------|---------|----------|------|
| **패킷(Packet)** | 3계층 — Network | IP 헤더, 페이로드(Payload) | IP 주소 정보가 포함된 데이터 단위 |
| **프레임(Frame)** | 2계층 — Data Link | 패킷 + MAC 주소 등 추가 정보 | 패킷을 한 번 더 캡슐화한 단위 |

이 관계를 우편으로 이해하면 쉽다. **봉투(프레임)** 는 내용물을 다른 곳으로 옮기는 틀 역할을 하고, 봉투 안에 든 **편지(패킷)** 가 실제 전달할 내용이다. 수신자가 봉투를 열면 편지를 어떻게 처리해야 할지 알 수 있다. 이 포장 과정이 바로 **캡슐화(Encapsulation)** 이고, 반대로 포장을 벗겨내는 과정은 **역캡슐화(Decapsulation)** 이다.

데이터를 한 번에 통째로 전송하지 않고 패킷으로 쪼개서 보내는 이유는 **네트워크 병목 현상(Bottleneck)을 줄이기 위해서**다. 예를 들어 웹사이트에서 이미지를 불러올 때, 이미지 전체가 한꺼번에 오는 게 아니라 여러 패킷으로 나뉘어 전송된 뒤 수신 측에서 재조립된다.

**IP 패킷 주요 헤더**

패킷에는 데이터 본문 외에 전송에 필요한 여러 헤더(Header) 정보가 포함된다.

| 헤더 | 영문 명칭 | 역할 |
|------|----------|------|
| **생존 시간** | TTL (Time to Live) | 패킷이 목적지에 도달하지 못했을 때 네트워크를 마비시키지 않도록 만료 타이머를 설정 |
| **체크섬** | Checksum | 데이터 무결성(Integrity) 확인. 수신 측에서 계산한 값이 다르면 데이터 손상으로 판단 |
| **소스 주소** | Source Address | 패킷을 보낸 장치의 IP 주소. 응답 데이터가 돌아올 주소 |
| **목적지 주소** | Destination Address | 패킷을 받을 장치의 IP 주소. 데이터가 다음으로 이동해야 할 곳 |

| 퀴즈 | 정답 |
|------|------|
| IP 주소 정보가 포함된 데이터를 무엇이라고 부르나? | `Packet` |
| IP 주소 정보가 없는 데이터를 무엇이라고 부르나? | `Frame` |

### Task 2 — TCP/IP (3-Way Handshake)

TCP/IP 모델은 OSI 모델을 간략하게 압축한 4계층 구조다. OSI의 7계층처럼, 각 계층을 데이터가 통과할 때마다 캡슐화/역캡슐화가 이루어진다.

| TCP/IP 계층 | 대응하는 OSI 계층 |
|------------|----------------|
| Application (애플리케이션) | 5·6·7계층 |
| Transport (전송) | 4계층 |
| Internet (인터넷) | 3계층 |
| Network Interface (네트워크 인터페이스) | 1·2계층 |

TCP는 **연결 기반(Connection-based)** 프로토콜이다. 즉, 데이터를 전송하기 전에 반드시 클라이언트(Client)와 서버(Server) 사이에 연결을 먼저 맺어야 한다. 이 연결 수립 과정을 **3방향 핸드셰이크(3-Way Handshake)** 라고 한다.

**TCP 패킷 주요 헤더**

| 헤더 | 영문 명칭 | 설명 |
|------|----------|------|
| **소스 포트** | Source Port | 발신자가 데이터 전송을 위해 열어둔 포트. 0~65535 중 사용하지 않는 포트로 무작위 선택 |
| **목적지 포트** | Destination Port | 수신 측에서 서비스나 앱이 실행되는 포트 번호 (예: 웹 서버는 80번). 무작위가 아님 |
| **소스 IP** | Source IP | 패킷을 보내는 장치의 IP 주소 |
| **목적지 IP** | Destination IP | 패킷을 받을 장치의 IP 주소 |
| **순번** | Sequence Number | 연결 시 첫 번째 데이터에 부여되는 임의의 번호 |
| **확인 응답 번호** | Acknowledgement Number | 다음 데이터에 부여할 번호 (이전 순번 + 1) |
| **체크섬** | Checksum | 수학적 계산으로 데이터 무결성(Integrity) 확인 |
| **데이터** | Data | 전송 중인 파일의 실제 바이트 |
| **플래그** | Flags | 패킷 처리 방식을 결정. 핸드셰이크 단계별로 특정 플래그가 사용됨 |

**TCP 플래그(Flags) 종류**

| 단계 | 플래그 | 설명 |
|------|--------|------|
| 1 | **SYN** | 클라이언트가 보내는 첫 패킷. 연결 시작 및 동기화 요청 |
| 2 | **SYN/ACK** | 서버가 클라이언트의 SYN을 수신했음을 확인하고 자신의 동기화 번호도 함께 전송 |
| 3 | **ACK** | 메시지/패킷을 성공적으로 수신했음을 확인하는 응답 |
| 4 | **DATA** | 연결 설정 완료 후 실제 데이터 전송 |
| 5 | **FIN** | 작업 완료 후 연결을 정상적으로 종료 (Finish의 약자) |
| # | **RST** | 모든 통신을 즉시 강제 종료. 오류 발생 시 최후의 수단 (Reset의 약자) |

**3방향 핸드셰이크 과정**

연결 수립 전 두 장치는 무작위 초기 시퀀스 번호(ISN, Initial Sequence Number)를 서로 교환하고 합의한다. 예시(클라이언트 ISN=0, 서버 ISN=5000):

```
1. SYN     클라이언트 → 서버  : "내 ISN은 0입니다. 연결하고 싶습니다."
2. SYN/ACK 서버 → 클라이언트  : "ISN 0 확인했습니다. 내 ISN은 5000입니다."
3. ACK     클라이언트 → 서버  : "ISN 5000 확인했습니다. 데이터 전송 시작합니다."
```

이후 데이터는 순번(Sequence Number)이 매겨지며 전송되고, 수신 측은 순번+1로 확인 응답한다.

| 클라이언트 | 초기 순번(ISN) | 최종 시퀀스 |
|-----------|--------------|-----------|
| 첫 번째 전송 | 0 | 0 + 1 = 1 |
| 두 번째 전송 | 1 | 1 + 1 = 2 |
| 세 번째 전송 | 2 | 2 + 1 = 3 |

**TCP 연결 종료**

한 장치가 상대방이 모든 데이터를 수신했음을 확인하면 연결 종료를 시작한다. TCP는 시스템 자원을 예약하기 때문에 사용이 끝나면 가능한 빨리 연결을 종료하는 것이 좋다.

```
1. FIN     Alice → Bob  : "전송 완료. 연결 종료합니다."
2. ACK     Bob → Alice  : "FIN 수신 확인."
3. FIN     Bob → Alice  : "저도 종료합니다."
4. ACK     Alice → Bob  : "확인했습니다. 연결 종료."
```

| 퀴즈 | 정답 |
|------|------|
| TCP 패킷에서 데이터 무결성을 보장하는 헤더는? | `checksum` |
| 3방향 핸드셰이크 순서는? (쉼표 구분) | `SYN,SYN/ACK,ACK` |

### Task 4 — UDP/IP

UDP(User Datagram Protocol)는 TCP와 달리 **무상태(Stateless) 프로토콜**이다. 연결을 사전에 수립하지 않고, 3방향 핸드셰이크도 없으며, 두 장치 간의 동기화도 없다. 데이터를 보내고 나면 상대방이 받았는지 여부를 신경 쓰지 않는다.

UDP 연결 흐름은 아래와 같다. 확인 응답(ACK) 없이 그냥 데이터를 계속 보낸다:

```
Alice → Bob : REQUEST
Bob  → Alice: RESPONSE
Bob  → Alice: RESPONSE
Bob  → Alice: RESPONSE
(수신 확인 없이 일방적 전송)
```

**UDP 패킷 주요 헤더**

| 헤더 | 영문 명칭 | 설명 |
|------|----------|------|
| **생존 시간** | TTL (Time to Live) | 패킷 만료 타이머. 목적지 도달 실패 시 네트워크 마비 방지 |
| **소스 주소** | Source Address | 패킷을 보낸 장치의 IP 주소 |
| **목적지 주소** | Destination Address | 패킷을 받을 장치의 IP 주소 |
| **소스 포트** | Source Port | 발신자가 열어둔 포트 (0~65535 무작위 선택) |
| **목적지 포트** | Destination Port | 수신 측 서비스가 실행 중인 포트 (무작위 아님) |
| **데이터** | Data | 전송 중인 파일의 실제 바이트 |

UDP는 TCP보다 헤더 수가 적고 구조가 단순하다. 체크섬이나 확인 응답 번호 같은 무결성 검증 필드가 없다.

| 퀴즈 | 정답 |
|------|------|
| "UDP"는 무슨 약자인가? | `User Datagram Protocol` |
| UDP는 어떤 유형의 연결 방식인가? | `stateless` |
| 파일을 전송할 때 사용할 프로토콜은? | `TCP` |
| 영상 통화에 사용할 프로토콜은? | `UDP` |

### Task 5 — 포트 101 (Port 101)

**포트(Port)** 는 데이터 교환이 이루어지는 필수적인 논리적 지점이다. 항구(Port)에 선박이 크기에 맞는 부두(Berth)에 접안하듯, 네트워크에서도 데이터는 해당 서비스에 맞는 포트 번호를 통해 전달된다. 선박이 부두에 접안해야만 화물을 내릴 수 있듯, 올바른 포트가 열려 있어야만 데이터를 주고받을 수 있다.

포트는 0번부터 **65535번**까지 사용할 수 있으며, 그중 **0~1024번은 공통 포트(Common Ports)** 로 잘 알려진 서비스에 미리 예약되어 있다. 복잡한 규칙은 그 자체로 중요하지 않지만, 이 공통 포트를 강제하는 이유는 여러 애플리케이션 간에 일관된 방식으로 통신할 수 있기 때문이다.

예를 들어, 어떤 브라우저를 사용하더라도 웹 데이터는 항상 80번 포트를 통해 전송된다. Google Chrome이든 Firefox든 동일한 방식으로 데이터를 처리할 수 있는 이유가 이 공통 규칙 덕분이다.

**주요 프로토콜과 포트 번호**

| 프로토콜 | 포트 번호 | 설명 |
|---------|---------|------|
| **FTP** (File Transfer Protocol, 파일 전송 프로토콜) | 21 | 클라이언트-서버 기반 파일 공유 앱에서 파일을 중앙 위치에서 다운로드하는 데 사용 |
| **SSH** (Secure Shell, 보안 셸) | 22 | 관리자가 텍스트 기반 인터페이스(CLI)로 시스템에 안전하게 원격 로그인하는 데 사용 |
| **HTTP** (HyperText Transfer Protocol) | 80 | 웹 브라우저가 웹 페이지의 텍스트, 이미지, 비디오를 내려받는 데 사용 |
| **HTTPS** (HTTP Secure) | 443 | HTTP와 동일한 기능이지만 암호화로 안전하게 보호 |
| **SMB** (Server Message Block) | 445 | FTP와 유사하지만 파일뿐 아니라 프린터 같은 장치도 공유 가능 |
| **RDP** (Remote Desktop Protocol, 원격 데스크톱 프로토콜) | 3389 | 그래픽 인터페이스로 시스템에 원격 로그인하는 방법. SSH의 GUI 버전 (단, SSL 암호화 필요) |

> 애플리케이션이 표준 포트 번호를 따르지 않고 다른 포트를 사용하는 것도 가능하다(예: 80번 대신 8080번에서 웹 서버 실행). 하지만 이 경우 포트 번호와 함께 콜론(`:`)을 URL에 명시해야 한다.

**실습**

```
$ nc 8.8.8.8 1234
Connection Received: THM{YOU_CONNECTED_TO_A_PORT}
```

`nc`(netcat) 명령으로 IP 주소 `8.8.8.8`의 `1234`번 포트에 직접 연결하면 해당 포트에서 실행 중인 프로그램의 응답을 받을 수 있다.

## 섹션 5 — Extending Your Network (네트워크 확장하기)

### Task 1 — 포트 포워딩 (Port Forwarding)

**포트 포워딩(Port Forwarding)** 은 애플리케이션과 서비스를 인터넷에 연결하는 데 필수적인 기술이다. 포트 포워딩이 없으면 웹 서버 같은 서비스는 **같은 직접 네트워크(인트라넷) 내의 장치에서만** 접근 가능하다.

예를 들어, IP 주소가 `192.168.1.10`인 서버가 80번 포트에서 웹 서버를 실행하고 있다고 하자. 포트 포워딩 없이는 같은 네트워크(Network #1)에 연결된 장치들만 이 서버에 접근할 수 있다.

포트 포워딩을 적용하면 외부 네트워크(Network #2)에서도 Network #1의 공용 IP 주소(예: `82.62.51.70`)를 통해 내부 웹 서버에 접근할 수 있게 된다.

```
[포트 포워딩 전]
Network #1 내부 장치만 192.168.1.10:80 접근 가능

[포트 포워딩 후]
외부(Network #2)에서 82.62.51.70:80 → 192.168.1.10:80 접근 가능
```

포트 포워딩과 방화벽은 혼동하기 쉽지만 역할이 다르다.

| 기능 | 역할 |
|------|------|
| **포트 포워딩** | 특정 포트를 열어 외부에서 내부 서비스에 접근할 수 있도록 경로를 설정 |
| **방화벽** | 포트가 열려 있더라도 해당 포트를 통해 트래픽이 실제로 이동할 수 있는지 여부를 결정 |

포트 포워딩은 **네트워크 라우터**에서 구성한다.

| 퀴즈 | 정답 |
|------|------|
| 포트 포워딩을 설정하는 데 사용되는 장치는? | `router` |

### Task 2 — 방화벽 기초 (Firewall Basics)

**방화벽(Firewall)** 은 네트워크 내에서 **어떤 트래픽이 들어오고 나갈 수 있는지를 결정**하는 장치다. 주로 네트워크의 **경계 보안(Perimeter Security)** 으로 사용되며, 관리자가 트래픽 허용/거부 규칙을 직접 구성할 수 있다.

방화벽이 트래픽을 허용하거나 거부할 때 검토하는 기준은 다음과 같다:

- **출발지(Source)**: 이 트래픽은 어디서 오는가?
- **목적지(Destination)**: 이 트래픽은 어디로 향하는가?
- **포트(Port)**: 이 트래픽은 어느 포트를 대상으로 하는가?
- **프로토콜(Protocol)**: TCP인가, UDP인가, 아니면 둘 다인가?

방화벽은 **패킷 검사(Packet Inspection)** 를 수행하여 위 질문들에 답하고 트래픽을 처리한다.

방화벽은 크기와 형태가 다양하다. 대규모 네트워크용 전용 하드웨어부터 가정용 공유기, Snort 같은 소프트웨어 방화벽까지 존재한다.

**방화벽의 두 가지 주요 유형**

| 유형 | 영문 | 특징 |
|------|------|------|
| **상태 유지 방화벽** | Stateful | 개별 패킷이 아닌 **전체 연결의 상태**를 기반으로 동작 여부 결정. 리소스 소모가 크지만 동적이고 정교함. 연결 상태가 나쁜 호스트는 전체 차단 가능 |
| **무상태 방화벽** | Stateless | **고정 규칙 집합**으로 개별 패킷 허용/거부 결정. 리소스 소모 적지만 기능이 제한적. DDoS 공격처럼 대량 트래픽이 특정 호스트 그룹에서 발생할 때 효과적 |

방화벽은 OSI 모델의 **3계층(Network)** 과 **4계층(Transport)** 에서 작동한다.

| 퀴즈 | 정답 |
|------|------|
| 방화벽은 OSI 모델의 어떤 계층에서 작동하나? | `3 & 4` |
| 전체 연결을 검사하는 방화벽 유형은? | `stateful` |
| 개별 패킷을 검사하는 방화벽 유형은? | `stateless` |

### Task 4 — VPN 기본 사항 (VPN Basics)

**VPN(Virtual Private Network, 가상 사설 네트워크)** 은 서로 다른 네트워크에 있는 장치들이 인터넷을 통해 **전용 암호화 터널(Tunnel)** 을 만들어 안전하게 통신할 수 있도록 하는 기술이다. 터널 내에 연결된 장치들은 자체적인 **사설 네트워크(Private Network)** 를 형성한다.

예를 들어, 두 개의 서로 다른 사무실 네트워크(Network #1, #2)가 있을 때, VPN을 사용하면 인터넷을 통해 두 사무실을 마치 하나의 사설 네트워크(Network #3)처럼 연결할 수 있다. 이 VPN 터널에 연결된 장치들만 서로 통신할 수 있다.

**VPN의 주요 이점**

| 이점 | 설명 |
|------|------|
| **원격 네트워크 연결** | 지리적으로 떨어진 사무실 네트워크들을 하나로 연결. 서버·인프라 자원을 다른 사무실에서도 접근 가능 |
| **개인정보 보호** | 데이터를 암호화하여 전송 중 도청(스니핑) 방지. 공용 Wi-Fi처럼 암호화가 없는 환경에서 특히 유용 |
| **익명성 제공** | ISP 등 중간 업체가 트래픽을 추적하기 어렵게 함. 표현의 자유가 제한된 국가에서 언론인·활동가들이 활용 |

> TryHackMe는 VPN을 사용해 취약한 실습 시스템에 안전하게 연결한다. 이를 통해 ISP가 사용자가 다른 컴퓨터를 공격하는 것으로 오해하지 않으며, 취약한 시스템이 인터넷에 직접 노출되지 않아 보안이 유지된다.

**VPN 기술 종류**

| 기술 | 전체 이름 | 특징 |
|------|----------|------|
| **PPP** | Point-to-Point Protocol | 개인 키와 공개 인증서로 인증 및 데이터 암호화. 단, 자체적으로 네트워크 밖으로 라우팅 불가 |
| **PPTP** | Point-to-Point Tunneling Protocol | PPP 데이터를 네트워크 밖으로 이동시키는 터널링 프로토콜. 설정 간편, 대부분 기기 지원. 암호화 수준은 낮음 |
| **IPSec** | Internet Protocol Security | 기존 IP 프레임워크를 사용해 데이터 암호화. 설정이 복잡하지만 강력한 암호화 제공, 다양한 기기 지원 |

| 퀴즈 | 정답 |
|------|------|
| 데이터 암호화 및 인증 기능만 제공하는 VPN 기술은? | `PPP` |
| IP 프레임워크를 사용하는 VPN 기술은? | `IPSec` |

### Task 5 — LAN 네트워킹 장치 (Router & Switch)

#### 라우터 (Router)

라우터의 역할은 **네트워크와 네트워크를 연결하고 데이터를 전달**하는 것이다. 이를 위해 **라우팅(Routing)** 이라는 기술을 사용한다. 라우터는 OSI 모델의 **3계층(Network Layer)** 에서 작동하며, 일반적으로 관리자가 포트 포워딩·방화벽 등 다양한 규칙을 구성할 수 있는 인터페이스(웹 UI 또는 콘솔)를 제공한다.

여러 경로로 장치가 연결되어 있을 때, 라우터는 아래 기준으로 최적 경로를 선택한다:

- 어느 경로가 가장 짧은가? (경유하는 장치 수 최소)
- 어느 경로가 가장 신뢰할 수 있는가? (패킷 손실 이력)
- 어느 경로의 물리적 연결 속도가 더 빠른가? (광섬유 vs 구리선)

#### 스위치 (Switch)

스위치는 **여러 장치를 하나의 네트워크 내에서 연결**하는 데 특화된 장치다. 이더넷 케이블을 사용해 최대 63대까지 장치를 연결할 수 있다.

스위치는 OSI 모델의 **2계층** 또는 **3계층**에서 동작하며, 두 계층은 서로 배타적이다.

| 유형 | 동작 계층 | 역할 |
|------|---------|------|
| **Layer 2 스위치** | 2계층 (Data Link) | MAC 주소를 기반으로 프레임을 올바른 장치로 전달 |
| **Layer 3 스위치** | 3계층 (Network) | 프레임 전달(Layer 2 기능) + IP 프로토콜로 패킷을 다른 네트워크로 라우팅 가능 |

#### VLAN (Virtual Local Area Network)

**VLAN(가상 로컬 영역 네트워크)** 은 같은 스위치에 연결된 장치들을 **가상으로 분리**하는 기술이다. VLAN으로 분리된 장치들은 인터넷 등 공통 기능은 이용할 수 있지만, **서로 직접 통신할 수 없다**. 이를 통해 네트워크 보안을 강화할 수 있다.

예: 같은 스위치에 연결된 "영업부(Sales, VLAN 1)"와 "회계부(Accounting, VLAN 2)"는 인터넷에 모두 접속할 수 있지만, 두 부서 간 직접 통신은 차단된다.

| **Stateful 방화벽** | 전체 연결 상태 검사, 동적·정교함, 리소스 소모 큼 |
| **Stateless 방화벽** | 고정 규칙으로 개별 패킷 검사, 리소스 적음, DDoS 방어에 효과적 |
| **VPN** | 인터넷 통해 암호화 터널 생성, 원격 연결·개인정보 보호·익명성 제공 |
| **PPP** | 인증·암호화 제공, 네트워크 밖 라우팅 불가 |
| **PPTP** | PPP 데이터를 터널링, 설정 쉬움, 암호화 약함 |
| **IPSec** | IP 프레임워크 기반 암호화, 설정 복잡, 강력한 보안 |
| **라우터** | 네트워크 간 데이터 전달, OSI Layer 3, 최적 경로 선택 |
| **Layer 2 스위치** | MAC 주소 기반 프레임 전달, OSI Layer 2 |
| **Layer 3 스위치** | 프레임 전달 + 패킷 라우팅, OSI Layer 3 |
| **VLAN** | 같은 스위치 내 가상 네트워크 분리, 부서 간 통신 차단으로 보안 강화 |
