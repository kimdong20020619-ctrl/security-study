## 섹션 1 — DNS란 무엇인가? (What is DNS?)

**DNS (Domain Name System, 도메인 이름 시스템)** = 사람이 읽기 쉬운 도메인 이름을 컴퓨터가 이해하는 IP 주소로 변환해주는 시스템

인터넷상의 모든 컴퓨터는 **IP 주소**라는 고유한 주소를 가진다. IPv4 기준으로 `104.26.10.229`처럼 0~255 사이의 숫자 4개를 마침표로 구분한 형태다.

이 숫자들을 직접 외우는 것은 사실상 불가능하기 때문에, DNS가 `tryhackme.com` 같은 이름을 입력하면 내부적으로 IP 주소(`104.26.10.229`)로 변환해준다.

> **비유**: 집마다 고유한 우편 주소가 있는 것처럼, 인터넷상의 모든 컴퓨터에는 IP 주소가 있다. DNS는 이름과 주소를 연결해주는 전화번호부 역할을 한다.

## 섹션 2 — 도메인 계층 구조 (Domain Hierarchy)

도메인 이름은 계층 구조로 이루어져 있다. `admin.tryhackme.com`을 예시로 보면:

```
admin      .    tryhackme    .    com
  ↑                ↑               ↑
서브도메인      2차 도메인       최상위 도메인
(Subdomain)    (SLD)            (TLD)
```

### 최상위 도메인 (TLD, Top-Level Domain)

도메인 이름의 **가장 오른쪽 부분** (예: `tryhackme.com`에서 `.com`)

두 종류로 나뉜다:

| 종류 | 정식 명칭 | 용도 | 예시 |
|------|----------|------|------|
| **gTLD** | Generic TLD (일반 최상위 도메인) | 도메인의 목적을 나타냄 | `.com` (상업), `.org` (기관), `.edu` (교육), `.gov` (정부) |
| **ccTLD** | Country Code TLD (국가 코드 최상위 도메인) | 지리적 위치를 나타냄 | `.ca` (캐나다), `.co.uk` (영국) |

> 수요 증가로 `.online`, `.club`, `.website`, `.biz` 등 새로운 gTLD가 계속 등장하고 있다. 현재 전 세계 TLD는 2,000개가 넘는다.

### 2차 도메인 (Second-Level Domain, SLD)

TLD 바로 왼쪽에 있는 부분 (예: `tryhackme.com`에서 `tryhackme`)

| 항목 | 내용 |
|------|------|
| 최대 길이 | 63자 |
| 허용 문자 | 알파벳(a-z), 숫자(0-9), 하이픈(-) |
| 금지 사항 | 하이픈으로 시작·끝 불가, 연속 하이픈 불가 |

### 서브도메인 (Subdomain)

SLD의 **왼쪽에 마침표(.)로 구분**하여 붙이는 추가 레이블 (예: `admin.tryhackme.com`에서 `admin`)

| 항목 | 내용 |
|------|------|
| 최대 길이 | 63자 (각 서브도메인 개별 기준) |
| 허용 문자 | 알파벳(a-z), 숫자(0-9), 하이픈(-) |
| **금지 문자** | **언더스코어(`_`) 사용 불가** |
| 금지 사항 | 하이픈으로 시작·끝 불가, 연속 하이픈 불가 |
| 체인 가능 | `jupiter.servers.tryhackme.com`처럼 여러 개 연결 가능 |
| 전체 도메인 최대 길이 | 253자 |
| 서브도메인 개수 제한 | 없음 |

## 섹션 3 — DNS 레코드 유형 (Record Types)

DNS 레코드는 웹사이트 접속뿐 아니라 이메일, IPv6, 도메인 인증 등 다양한 용도에 사용된다.

| 레코드 유형 | 역할 | 예시 |
|------------|------|------|
| **A** | 도메인 → **IPv4 주소**로 변환 | `104.26.10.229` |
| **AAAA** | 도메인 → **IPv6 주소**로 변환 | `2606:4700:20::681a:be5` |
| **CNAME** | 도메인 → **다른 도메인 이름**으로 변환 (별칭) | `store.tryhackme.com` → `shops.shopify.com` |
| **MX** | 도메인의 **이메일 처리 서버** 주소 | `alt1.aspmx.l.google.com` |
| **TXT** | **텍스트 데이터** 자유 저장 | SPF, DMARC, 도메인 소유권 인증 등 |

### CNAME 레코드 — 작동 방식

CNAME은 IP 주소가 아닌 **다른 도메인 이름을 반환**하기 때문에, 실제 IP 주소를 얻으려면 그 도메인에 대해 DNS를 한 번 더 조회해야 한다.

```
① store.tryhackme.com 조회 요청
        ↓
② CNAME 레코드 반환: shops.shopify.com
        ↓
③ shops.shopify.com에 다시 DNS 조회
        ↓
④ 최종 IP 주소 획득
```

### MX 레코드 — 우선순위 플래그

MX 레코드에는 **우선순위(Priority) 플래그**가 포함된다. 숫자가 낮을수록 우선순위가 높다. 메인 메일 서버에 장애가 발생했을 때 어떤 백업 서버를 먼저 시도할지 순서를 정한다.

### TXT 레코드 — 주요 활용 사례

텍스트를 자유롭게 저장할 수 있어 다양한 목적으로 활용된다:

| 활용 사례 | 설명 |
|-----------|------|
| **스팸·스푸핑 방지 (SPF)** | 이 도메인 대신 이메일을 보낼 수 있는 서버 목록을 기록. 허가받지 않은 서버가 이 도메인을 사칭해 이메일을 발송하는 것을 방지 |
| **이메일 인증 정책 (DMARC)** | 이메일 인증 실패 시 수신 서버가 해당 메일을 거부·격리할지 정책을 기록 |
| **도메인 소유권 증명** | Google Workspace, Microsoft 365 등 외부 서비스 연동 시 "내가 이 도메인의 주인임"을 증명하는 인증 코드를 기록 |

> 보안 관점: SPF, DMARC 레코드가 없는 도메인은 공격자가 그 도메인을 사칭한 피싱 이메일을 보내기 훨씬 쉬워진다.

## 섹션 4 — DNS 요청 과정 (Making A Request)

브라우저에 `www.tryhackme.com`을 입력하면 내부적으로 다음 5단계를 거쳐 IP 주소를 찾는다:

```
[1] 내 컴퓨터 (로컬 캐시 확인)
        ↓ 캐시에 없으면
[2] 재귀 DNS 서버 (ISP 제공 — 자체 캐시 확인)
        ↓ 캐시에 없으면
[3] 루트 DNS 서버 (TLD 서버로 안내)
        ↓
[4] TLD 서버 (.com 담당 — 권한 서버 주소 안내)
        ↓
[5] 권한 DNS 서버 (실제 레코드 보유)
        ↓
    IP 주소 반환 → 재귀 서버 캐시 저장 → 내 컴퓨터로 전달 → 브라우저 접속
```

### 각 단계 상세 설명

**1단계 — 로컬 캐시 확인**
- 컴퓨터는 최근에 조회한 도메인 주소를 **로컬 캐시**에 저장해둔다
- 캐시에 있으면 즉시 반환하고 이후 단계를 거치지 않는다

**2단계 — 재귀 DNS 서버 (Recursive DNS Server)**
- 일반적으로 **ISP(인터넷 서비스 제공업체)**가 제공하며, 사용자가 직접 지정할 수도 있다 (예: Google Public DNS `8.8.8.8`)
- 이 서버도 **자체 캐시**를 가진다
- Google, Facebook처럼 자주 조회되는 도메인은 여기서 바로 응답하는 경우가 많다
- 캐시에 없으면 루트 서버로 요청을 넘긴다

**3단계 — 루트 DNS 서버 (Root DNS Server)**
- 인터넷 DNS 계층의 **최상단**에 위치하는 핵심 인프라
- 특정 도메인의 IP를 직접 알지는 못하지만, **어느 TLD 서버에 물어봐야 하는지**를 알고 있다
- `www.tryhackme.com` 요청 → `.com` TLD를 담당하는 서버 주소 안내

**4단계 — TLD 서버 (Top-Level Domain Server)**
- `.com`, `.org` 등 **특정 TLD를 담당**하는 서버
- 해당 TLD 아래 도메인을 실제로 관리하는 **권한 서버(Authoritative Server)** 정보를 알고 있다
- 권한 서버는 **네임서버(Nameserver)**라고도 불린다
  - 예: `tryhackme.com`의 네임서버 → `kip.ns.cloudflare.com`, `uma.ns.cloudflare.com`
- 네임서버를 **여러 개** 두는 이유: 하나가 장애 나도 다른 네임서버가 백업 역할을 하기 때문

**5단계 — 권한 DNS 서버 (Authoritative DNS Server)**
- 해당 도메인의 **모든 DNS 레코드를 직접 저장**한다
- DNS 레코드를 변경하면 이 서버에 먼저 기록된다
- IP 주소를 재귀 DNS 서버로 반환 → 재귀 서버는 결과를 **캐시에 저장**한 뒤 클라이언트로 최종 전달

### TTL (Time To Live, 레코드 유효 시간)

모든 DNS 레코드에는 **TTL** 값이 포함된다.

| 항목 | 내용 |
|------|------|
| 의미 | DNS 응답을 캐시에 **몇 초 동안** 저장할지 지정하는 값 |
| TTL이 크면 | DNS 서버에 자주 요청하지 않아도 됨 → 응답 빠름 |
| TTL이 작으면 | 변경된 레코드가 더 빨리 전 세계 캐시에 반영됨 |

## 실습 기록

### 실습 — DNS 조회 실습 (Practical)

**목표**: TryHackMe 제공 브라우저 내 DNS 조회 도구로 실제 도메인의 DNS 레코드를 확인하고, DNS 조회 흐름 직접 체험

**플래그**: [Task 5 미진행 — 실습 완료 후 기록]

**배운 점**

1. DNS 조회는 단순히 "도메인 → IP 변환"이 아니라 로컬 캐시 → 재귀 DNS → 루트 서버 → TLD 서버 → 권한 서버의 5단계 계층 구조를 거친다. 이 분산 구조 덕분에 전 세계 수십억 개의 도메인을 중앙 서버 하나 없이 관리할 수 있다.
2. CNAME 레코드는 IP가 아닌 다른 도메인 이름을 반환하기 때문에 DNS 조회가 한 번 더 일어난다. 삭제된 외부 서비스를 여전히 가리키는 CNAME이 남아있으면 공격자가 그 서비스를 재등록해 서브도메인을 탈취(Subdomain Takeover)하는 공격이 가능하므로, 사용하지 않는 DNS 레코드는 즉시 삭제해야 한다.
3. TTL 값이 낮으면 DNS 변경 사항이 전 세계 캐시에 빠르게 반영되지만 DNS 서버로의 요청이 많아진다. 반대로 TTL이 높으면 공격자가 재귀 DNS 서버의 캐시에 가짜 IP를 심는 DNS 캐시 포이즈닝(DNS Cache Poisoning) 공격이 성공했을 때 피해 지속 시간이 그만큼 길어진다.

## 섹션 5 — HTTP와 HTTPS란? (What is HTTP(S)?)

**HTTP (HyperText Transfer Protocol, 하이퍼텍스트 전송 프로토콜)** = 웹 서버와 통신하여 HTML, 이미지, 비디오 등 웹페이지 데이터를 전송하는 규칙 집합

- 팀 버너스-리와 그의 팀이 1989~1991년에 개발
- 웹 브라우저에서 웹사이트를 볼 때마다 사용됨

**HTTPS (HyperText Transfer Protocol Secure)** = HTTP의 보안 버전

| 항목 | HTTP | HTTPS |
|------|------|-------|
| 암호화 | 없음 (평문 전송) | 있음 (제3자가 볼 수 없음) |
| 서버 신원 확인 | 없음 | 있음 (가짜 서버와의 통신 방지) |
| 기본 포트 | 80 | 443 |
| 주소창 표시 | `http://` | `https://` + 자물쇠 아이콘 |

> 브라우저가 HTTPS 인증서 오류를 표시하는 경우, 가짜 서버에 연결됐거나 중간자 공격(MITM, Man-In-The-Middle)을 받고 있을 가능성이 있다.

## 섹션 6 — URL 구조와 요청/응답 형식 (Requests & Responses)

### URL 구조 (Uniform Resource Locator)

URL은 인터넷상의 특정 리소스에 접근하는 방법과 위치를 알려주는 주소다. 모든 구성 요소가 항상 포함되는 것은 아니다.

```
http://user:password@tryhackme.com:80/view-room?id=1#task3
  ↑        ↑               ↑         ↑      ↑      ↑     ↑
스키마    사용자           호스트    포트   경로  쿼리   프래그먼트
```

| 구성 요소 | 역할 | 예시 |
|----------|------|------|
| **스키마 (Scheme)** | 사용할 프로토콜 지정 | `http`, `https`, `ftp` |
| **사용자 (User)** | 인증이 필요한 서비스의 로그인 정보 | `user:password@` |
| **호스트 (Host)** | 접속할 서버의 도메인 이름 또는 IP 주소 | `tryhackme.com` |
| **포트 (Port)** | 연결할 포트 번호. 기본값(80/443)이면 생략 가능 | `:80` |
| **경로 (Path)** | 접근하려는 파일이나 리소스의 위치 | `/view-room` |
| **쿼리 문자열 (Query String)** | 경로에 전달하는 추가 정보. `?`로 시작, `&`로 구분 | `?id=1` |
| **프래그먼트 (Fragment)** | 페이지 내 특정 위치 지정. `#`으로 시작 | `#task3` |

### HTTP 요청 형식

```http
GET / HTTP/1.1
Host: tryhackme.com
User-Agent: Mozilla/5.0 Firefox/87.0
Referer: https://tryhackme.com/

```

| 줄 | 의미 |
|----|------|
| 1번째 줄 | HTTP 메서드(`GET`) + 요청 경로(`/`) + HTTP 프로토콜 버전(`HTTP/1.1`) |
| 2번째 줄 | 요청하는 웹사이트의 호스트명 |
| 3번째 줄 | 사용 중인 브라우저 소프트웨어와 버전 |
| 4번째 줄 | 이 페이지로 연결된 이전 웹페이지 주소(참조 URL) |
| 빈 줄 | 요청이 완료됐음을 서버에게 알리는 신호 (반드시 필요) |

### HTTP 응답 형식

```http
HTTP/1.1 200 OK
Server: nginx/1.15.8
Date: Fri, 09 Apr 2021 13:34:03 GMT
Content-Type: text/html
Content-Length: 98

<html>
<head>
  <title>TryHackMe</title>
</head>
<body>
  Welcome To TryHackMe.com
</body>
</html>
```

| 줄 | 의미 |
|----|------|
| 1번째 줄 | HTTP 버전 + 상태 코드(`200 OK`) |
| 2번째 줄 | 웹 서버 소프트웨어와 버전 |
| 3번째 줄 | 서버의 현재 날짜·시간·시간대 |
| 4번째 줄 | 반환되는 데이터 유형(HTML, CSS, 이미지, PDF 등) |
| 5번째 줄 | 응답 데이터의 길이(바이트) — 데이터 누락 여부 확인 가능 |
| 빈 줄 | HTTP 응답 헤더 끝을 나타내는 신호 |
| 이후 줄 | 실제 요청한 콘텐츠(HTML 등) |

## 섹션 7 — HTTP 메서드 (HTTP Methods)

HTTP 메서드는 클라이언트가 요청 시 **어떤 동작을 원하는지** 서버에게 알려주는 방식이다.

| 메서드 | 동작 | 예시 |
|--------|------|------|
| **GET** | 서버에서 정보를 **가져온다** | 페이지 조회, 뉴스 기사 읽기 |
| **POST** | 서버에 데이터를 **제출**하고 새 레코드를 **생성**한다 | 회원가입, 게시글 작성 |
| **PUT** | 서버에 데이터를 제출하여 기존 정보를 **수정**한다 | 이메일 주소 변경, 프로필 수정 |
| **DELETE** | 서버에서 정보/레코드를 **삭제**한다 | 업로드한 사진 삭제, 계정 삭제 |

---

## 섹션 8 — HTTP 상태 코드 (HTTP Status Codes)

서버 응답의 **첫 번째 줄**에는 항상 요청 결과를 나타내는 상태 코드가 포함된다.

### 상태 코드 5가지 범위

| 범위 | 분류 | 설명 |
|------|------|------|
| **100-199** | 정보 응답 | 요청의 첫 부분 수신 완료, 나머지를 보내도 됨. 현재는 거의 사용되지 않는다. |
| **200-299** | 성공 | 요청이 성공적으로 처리됐다. |
| **300-399** | 리디렉션 | 클라이언트를 다른 리소스나 웹사이트로 보낸다. |
| **400-499** | 클라이언트 오류 | 요청 자체에 문제가 있다. |
| **500-599** | 서버 오류 | 서버 측에서 요청 처리 중 심각한 문제가 발생했다. |

### 자주 만나는 상태 코드

| 코드 | 이름 | 의미 |
|------|------|------|
| **200** | OK | 요청 성공 |
| **201** | Created | 리소스 생성 성공 (예: 새 사용자, 새 게시글) |
| **301** | Permanently Moved | 영구 이동 — 브라우저/검색 엔진에 페이지 위치가 영구적으로 변경됐음을 알림 |
| **302** | Found | 임시 이동 — 가까운 시일 내에 다시 변경될 수 있는 일시적 리디렉션 |
| **400** | Bad Request | 요청에 오류 또는 누락된 정보가 있음 |
| **401** | Unauthorized | 인증이 필요함 (로그인 전 상태) |
| **403** | Forbidden | 로그인 여부와 관계없이 접근 권한 없음 |
| **404** | Not Found | 요청한 페이지/리소스가 존재하지 않음 |
| **405** | Method Not Allowed | 해당 리소스에서 허용되지 않는 HTTP 메서드로 요청함 |
| **500** | Internal Server Error | 서버가 요청 처리 중 오류 발생, 처리 방법을 알지 못함 |
| **503** | Service Unavailable | 서버 과부하 또는 점검 중으로 요청 처리 불가 |

> **401 vs 403 구분**: 401은 "누구인지 모름(미인증)", 403은 "누군지 알지만 권한 없음(인가 거부)"이다.

## 섹션 9 — HTTP 헤더 (Headers)

헤더는 요청/응답 시 함께 전송되는 **추가 메타데이터**다. 필수는 아니지만, 없으면 웹사이트가 제대로 작동하지 않는 경우가 많다.

### 공통 요청 헤더 (클라이언트 → 서버)

| 헤더 | 역할 |
|------|------|
| **Host** | 하나의 서버에 여러 웹사이트가 있을 때, 어떤 사이트를 원하는지 지정. 없으면 서버 기본 사이트가 응답함 |
| **User-Agent** | 사용 중인 브라우저 소프트웨어와 버전. 서버가 브라우저에 맞게 콘텐츠를 포맷하는 데 사용됨 |
| **Content-Length** | 서버로 전송하는 데이터의 양(바이트). 서버가 데이터 누락 여부를 확인하는 데 사용됨 |
| **Accept-Encoding** | 브라우저가 지원하는 압축 방식. 서버가 이에 맞춰 데이터를 압축해 전송 크기를 줄임 |
| **Cookie** | 이전에 서버가 저장해둔 쿠키 데이터를 요청마다 서버로 다시 전송 |

### 공통 응답 헤더 (서버 → 클라이언트)

| 헤더 | 역할 |
|------|------|
| **Set-Cookie** | 브라우저에 저장할 쿠키 데이터를 지정. 이후 요청마다 서버로 자동 전송됨 |
| **Cache-Control** | 브라우저가 응답 내용을 캐시에 얼마나 오래 저장할지 지정 |
| **Content-Type** | 반환되는 데이터 유형(HTML, CSS, JS, 이미지, PDF 등)을 클라이언트에게 알림 |
| **Content-Encoding** | 전송 시 데이터 크기를 줄이기 위해 사용된 압축 방식 |

## 섹션 10 — 쿠키 (Cookies)

**쿠키** = 웹 서버가 브라우저에 저장시키는 작은 데이터 조각

### 쿠키가 필요한 이유

HTTP는 **상태 비저장(Stateless)** 프로토콜이다. 각 요청은 독립적으로 처리되며, 서버는 이전 요청을 기억하지 못한다. 쿠키는 이 한계를 극복하기 위해 사용된다.

### 쿠키 동작 흐름

```
① 클라이언트 → 서버: GET / 요청 (쿠키 없음, 이름 입력 폼 요청)
        ↓
② 서버 → 클라이언트: 응답 + 이름 입력 폼 HTML 반환
        ↓
③ 클라이언트 → 서버: POST / 요청 (name=adam 전송)
        ↓
④ 서버 → 클라이언트: 응답 + Set-Cookie: name=adam 헤더 (브라우저가 쿠키 저장)
        ↓
⑤ 클라이언트 → 서버: 이후 모든 요청에 Cookie: name=adam 헤더 포함
        ↓
⑥ 서버: 쿠키 데이터를 보고 사용자를 식별 → "Welcome back adam" 반환
```

### 쿠키의 주요 용도

| 용도 | 설명 |
|------|------|
| **인증 (가장 흔함)** | 로그인 상태 유지. 쿠키 값은 비밀번호가 아닌 **토큰(Token)** 형태로 저장됨 |
| **개인화 설정** | 언어, 테마 등 웹사이트 설정 기억 |
| **방문 기록** | 사용자의 웹사이트 방문 이력 추적 |

> **토큰(Token)**: 사람이 쉽게 추측할 수 없는 고유한 비밀 코드. 쿠키에 비밀번호를 직접 저장하면 탈취 시 위험하기 때문에, 서버가 발급한 토큰을 대신 저장한다.

### 쿠키 확인 방법

브라우저 개발자 도구 → **Network 탭** → 리소스 클릭 → **Cookies 탭**에서 브라우저가 전송하는 쿠키를 확인할 수 있다.

## 실습 기록 (HTTP in Detail)

### 실습 1 — HTTPS 인증서 오류 식별

**목표**: 가상 웹페이지에서 유효하지 않은 HTTPS 인증서 문제를 찾아 클릭

**플래그**: `THM{INVALID_HTTP_CERT}`

**배운 점**

1. HTTPS는 데이터 암호화뿐만 아니라 **서버 신원 검증**도 수행한다. 인증서가 유효하지 않으면 브라우저가 경고를 표시하는데, 이는 가짜 서버에 연결됐거나 중간자 공격(MITM, Man-In-The-Middle)을 받고 있을 가능성을 의미한다. 이 경고를 무시하고 로그인 정보를 입력하면 제3자에게 그대로 노출된다.
2. HTTP와 HTTPS의 차이는 브라우저 주소창의 자물쇠 아이콘과 `https://` 여부로 구분된다. 보안이 필요한 서비스(로그인, 결제 등)는 반드시 HTTPS를 사용해야 하며, 유효한 SSL/TLS 인증서가 있어야 한다. 인증서가 만료됐거나 자체 서명된 경우도 경고 대상이다.
3. 보안 테스트에서 HTTPS 인증서 상태 확인은 기본 점검 항목이다. 인증서가 신뢰할 수 있는 기관(CA, Certificate Authority)에서 발급됐는지, 만료되지 않았는지, 도메인이 일치하는지 세 가지를 모두 확인해야 한다.

### 실습 2 — HTTP 메서드 실습 (Making Requests)

**목표**: TryHackMe HTTP 요청 시뮬레이터로 GET/POST/PUT/DELETE 요청을 직접 보내 각 메서드 동작 체험

**사용 도구**: TryHackMe 브라우저 내 HTTP 요청 시뮬레이터

**실습 내용 및 플래그**

| 메서드 | 경로 | 추가 파라미터 | 플래그 |
|--------|------|--------------|--------|
| GET | `/room` | — | `THM{YOU'RE_IN_THE_ROOM}` |
| GET | `/blog` | 쿼리 문자열: `id=1` | `THM{YOU_FOUND_THE_BLOG}` |
| DELETE | `/user/1` | — | `THM{USER_IS_DELETED}` |
| PUT | `/user/2` | 본문: `username=admin` | `THM{USER_HAS_UPDATED}` |
| POST | `/login` | 본문: `username=thm`, `password=letmein` | `THM{HTTP_REQUEST_MASTER}` |

**배운 점**

1. HTTP 메서드(GET/POST/PUT/DELETE)는 단순한 이름 규칙이 아니라 **서버가 요청을 처리하는 동작 자체**를 결정한다. 같은 경로(`/user/1`)라도 GET은 조회, DELETE는 삭제라는 전혀 다른 동작이 수행된다. 서버가 메서드를 제대로 검증하지 않으면 허가받지 않은 삭제나 수정이 발생할 수 있다.
2. GET 요청의 쿼리 문자열(`?id=1`)은 URL에 포함되어 브라우저 주소창, 서버 로그, 공유 링크에 그대로 노출된다. 반면 POST 요청의 본문(Body) 데이터는 URL에 노출되지 않는다. 비밀번호나 토큰 같은 민감한 데이터를 URL에 담지 않아야 하는 이유가 바로 이 차이에 있다.
3. `/login`에 POST로 `username=thm&password=letmein`을 전송해 인증에 성공했다. 이 과정은 실제 로그인 폼을 제출할 때 브라우저가 내부적으로 수행하는 동작과 동일하다. 공격자는 이 요청을 직접 조작해 자동화된 로그인 시도(Brute Force)를 수행할 수 있으므로, 로그인 엔드포인트에는 속도 제한(Rate Limiting)과 계정 잠금 정책이 반드시 필요하다.

## 섹션 11 — 웹사이트의 두 가지 구성 요소 (Frontend & Backend)

웹사이트를 방문하면 브라우저가 웹 서버에 요청을 보내고, 서버는 페이지를 표시하는 데 필요한 데이터를 응답으로 돌려준다.

```
브라우저 (Browser)
    ↕ Request / Response
인터넷 (Internet)
    ↕ Request / Response
웹 서버 (Server)
```

웹사이트는 크게 두 가지 구성 요소로 이루어진다:

| 구성 요소 | 역할 | 실행 위치 |
|----------|------|----------|
| **프런트엔드 (Frontend, 클라이언트 측)** | 브라우저가 웹사이트를 화면에 보여주는 방식. HTML/CSS/JS로 구성 | 사용자의 브라우저 |
| **백엔드 (Backend, 서버 측)** | 사용자의 요청을 받아 처리하고 결과를 반환. 데이터베이스 연동 포함 | 웹 서버 |

> 웹 서버는 전 세계 어딘가에 있는, 사용자의 요청을 처리하는 전용 컴퓨터다.

**로그인 폼을 예시로 두 역할의 차이를 이해하면 쉽다**:

| 동작 | 담당 | 이유 |
|------|------|------|
| 아이디·비밀번호 입력 필드와 버튼을 화면에 표시 | 프런트엔드 | 브라우저가 HTML을 렌더링해 보여줌 |
| "이 비밀번호가 이 계정과 일치하는가?" DB에서 확인 | 백엔드 | 서버에서 실행, 사용자는 볼 수 없음 |

> 비밀번호 검증은 반드시 백엔드에서 해야 한다. 프런트엔드에서만 검증하면 공격자가 JavaScript를 우회해 로그인 로직 자체를 건너뛸 수 있다.

## 섹션 12 — HTML

웹사이트는 주로 세 가지 기술로 만들어진다:

| 기술 | 역할 |
|------|------|
| **HTML** | 웹사이트의 구조와 콘텐츠 정의 |
| **CSS** | 웹사이트의 시각적 스타일링 (색상, 폰트, 레이아웃 등) |
| **JavaScript** | 페이지에 상호작용과 동적 기능 추가 |

**CSS (Cascading Style Sheets)** = HTML로 만든 구조에 색상, 폰트, 크기, 여백, 레이아웃 등 시각적 디자인을 입히는 스타일 언어.

- HTML이 "어떤 내용이 있는가(구조)"를 담당한다면, CSS는 "어떻게 보일 것인가(디자인)"를 담당한다
- 예: 제목 글자를 빨간색으로 만들거나, 버튼 크기를 키우거나, 메뉴를 화면 상단에 고정하는 것이 모두 CSS의 역할이다
- CSS는 별도 파일(`.css`)로 분리하거나 HTML 내 `<style>` 태그 안에 작성한다

---

**HTML (HyperText Markup Language)** = 웹사이트를 작성하는 언어. 엘리먼트(태그)가 빌딩 블록 역할을 하며 브라우저에게 콘텐츠를 어떻게 표시할지 알려준다.

### HTML 기본 구조

```html
<!DOCTYPE html>
<html>
  <head>
    <title>Page Title</title>
  </head>
  <body>
    <h1>Example Heading</h1>
    <p>Example paragraph.</p>
  </body>
</html>
```

### 주요 태그 설명

| 태그 | 역할 |
|------|------|
| `<!DOCTYPE html>` | HTML5 문서 선언. 브라우저에게 HTML5 표준으로 해석하라는 지시 |
| `<html>` | HTML 페이지의 루트(최상위) 요소. 모든 요소가 이 안에 들어간다 |
| `<head>` | 페이지 정보 포함 (타이틀, 스타일시트 링크 등). 화면에 직접 표시되지 않음 |
| `<body>` | 문서 본문. **body 내부 콘텐츠만 브라우저 화면에 표시된다** |
| `<h1>` | 대형 제목 (h1이 가장 크고 h6이 가장 작다) |
| `<p>` | 단락(문단) |
| `<button>` | 클릭 가능한 버튼 |
| `<img>` | 이미지 |

### 태그 속성 (Attributes)

태그에는 추가 정보를 담는 **속성**을 붙일 수 있다.

```html
<!-- class 속성: 스타일 적용 (여러 요소가 같은 class 공유 가능) -->
<p class="bold-text">굵은 텍스트</p>

<!-- src 속성: 이미지 파일 경로 지정 -->
<img src="img/cat.jpg">

<!-- 여러 속성을 동시에 사용 가능 -->
<p attribute1="value1" attribute2="value2">내용</p>
```

### id 속성 — class와의 차이

```html
<p id="example">이 요소의 고유 ID는 "example"이다</p>
```

| 속성 | 특징 |
|------|------|
| **class** | 여러 요소가 동일한 class를 가질 수 있다 (스타일 공유 목적) |
| **id** | 하나의 요소에만 사용할 수 있는 고유 식별자. JavaScript로 특정 요소를 선택할 때 사용 |

### 페이지 소스 보기

어떤 웹사이트든 HTML 소스 코드를 직접 볼 수 있다:
- **Chrome**: 우클릭 → "페이지 소스 보기" / 단축키 `Ctrl+U`
- **Safari**: 우클릭 → "Show Page Source"

## 섹션 13 — JavaScript

**JavaScript(JS)** = 세계에서 가장 인기 있는 프로그래밍 언어 중 하나로, HTML로 구조를 만들고 CSS로 스타일을 입힌 뒤 **상호작용과 동적 기능**을 추가한다.

- JavaScript 없이는 페이지가 정적인 상태로만 존재한다
- JavaScript가 있으면 페이지를 실시간으로 업데이트하고, 사용자 이벤트에 반응하며, 애니메이션을 구현할 수 있다

### JavaScript 포함 방법

**방법 1 — 페이지 내 인라인 작성** (`<script>` 태그 안에 직접 코드 작성):

```html
<script>
  document.getElementById("demo").innerHTML = "Hack the Planet";
</script>
```

**방법 2 — 외부 파일로 분리** (`src` 속성으로 외부 파일 불러오기):

```html
<script src="/location/of/javascript_file.js"></script>
```

### getElementById 예시

```javascript
// ID가 "demo"인 HTML 요소의 내용을 "Hack the Planet"으로 변경
document.getElementById("demo").innerHTML = "Hack the Planet";
```

### 이벤트 (Events)

이벤트는 특정 동작이 발생했을 때 JavaScript 코드를 실행시키는 트리거다.

```html
<!-- onclick: 버튼을 클릭했을 때 demo 요소의 내용을 변경 -->
<button onclick="document.getElementById('demo').innerHTML = 'Button Clicked!'">
  Click Me!
</button>
```

| 이벤트 | 발동 조건 |
|--------|----------|
| `onclick` | 요소를 클릭했을 때 |
| `onhover` | 마우스를 요소 위에 올렸을 때 |

> 이벤트는 HTML 요소의 속성으로 정의할 수도 있고, JavaScript 코드 내부에서 직접 정의할 수도 있다.

## 섹션 14 — 민감 정보 노출 (Sensitive Data Exposure)

**민감 정보 노출** = 웹사이트가 민감한 정보를 평문으로 소스 코드에 남겨두어 제3자에게 노출되는 취약점

### 발생 원인

개발자가 임시로 작성한 로그인 자격증명, 숨겨진 링크, API 키 등을 HTML 주석이나 JavaScript에 그대로 남긴 채 배포한 경우 발생한다.

### 소스 코드 확인 방법

- 브라우저에서 `Ctrl+U` → 현재 페이지의 HTML 소스 코드를 새 탭에서 확인

### 실제 발견 사례 (Ctrl+U로 확인한 소스 코드)

실습에서 `Ctrl+U`로 `https://vulnerable-site.com`의 소스를 확인했을 때 로그인 폼 HTML 안에 다음 주석이 발견됐다:

```html
<!--
    TODO: Remove test credentials!
    Username: admin
    Password: testpasswd
-->
```

**문제**: HTML 주석(`<!-- ... -->`)은 브라우저 화면에는 표시되지 않지만, `Ctrl+U`로 소스를 열면 누구나 읽을 수 있다. 개발 중 임시로 작성한 자격증명이 그대로 운영 서버에 배포된 전형적인 사례다.

### 보안 관점

웹 애플리케이션 보안 점검 시 **가장 먼저 확인하는 사항** 중 하나가 페이지 소스 코드다. 자격증명, 숨겨진 경로, 내부 API 엔드포인트 등이 소스 코드에 그대로 노출된 경우가 실제로 많다.

## 섹션 15 — HTML 인젝션 (HTML Injection)

**HTML 인젝션** = 필터링(검증)되지 않은 사용자 입력이 그대로 페이지에 출력될 때, 공격자가 HTML 코드를 삽입해 페이지 구조와 동작을 조작하는 취약점

### 취약한 코드 예시

```javascript
function sayHi() {
    const name = document.getElementById('name').value
    document.getElementById("welcome-msg").innerHTML = "Welcome " + name
}
```

사용자가 입력한 `name` 값을 검증 없이 `innerHTML`에 바로 삽입하기 때문에, 입력 내용이 순수한 HTML로 렌더링된다.

### 공격 흐름

```
① 사용자가 입력 필드에 텍스트 입력
        ↓
② 입력 내용이 JavaScript 함수(sayHi)를 통해 페이지에 출력
        ↓ (입력 검증 없음)
③ 공격자가 일반 텍스트 대신 HTML 코드를 입력
        ↓
④ 브라우저가 HTML 코드를 렌더링 → 공격자가 원하는 콘텐츠가 페이지에 표시됨
```

### 공격 예시

입력 필드에 아래 HTML 코드를 삽입하면:

```html
<a href="http://hacker.com"></a>
```

페이지에 실제 링크로 렌더링되어 사용자가 클릭하면 악성 사이트로 이동하게 된다.

### 방어 방법 — 입력 검증 (Input Validation / Sanitization)

- **프론트엔드 검증**: 브라우저에서 JavaScript로 입력값 검사
- **백엔드 검증**: 서버에서 HTML 태그 제거 또는 이스케이프 처리
- **주의**: 프론트엔드 검증만으로는 불충분하다. 공격자가 HTTP 요청을 직접 조작해 우회할 수 있기 때문에, **백엔드 검증이 반드시 필요하다**

> HTML 인젝션의 더 위험한 형태가 **XSS(Cross-Site Scripting)**다. `<script>` 태그를 삽입하면 피해자의 브라우저에서 악성 JavaScript가 실행될 수 있다.

## 실습 기록 (How Websites Work)

### 실습 1 — HTML 에디터 실습

**목표**: TryHackMe HTML 에디터에서 고양이 웹사이트의 깨진 이미지를 수정하고 개 이미지를 추가

**실습 내용 및 플래그**

| 과제 | 내용 | 플래그 |
|------|------|--------|
| 깨진 이미지 수정 | `src` 경로 오류로 표시되지 않던 이미지를 수정 | `HTMLHERO` |
| 개 이미지 추가 | 11번째 줄에 `<img src='img/dog-1.png'>` 태그 추가 | `DOGHTML` |

**배운 점**

1. `<img>` 태그의 `src` 속성에 잘못된 경로가 지정되면 이미지가 깨져 표시된다. 브라우저는 이 오류를 조용히 처리하기 때문에, 화면만 봐서는 원인을 알기 어렵다. 소스 코드를 직접 열어 경로를 확인하는 습관이 필요한 이유다.
2. HTML 태그 하나(`<img src='img/dog-1.png'>`)를 추가하는 것만으로도 페이지에 완전히 새로운 콘텐츠가 렌더링된다. 이처럼 HTML 삽입이 가능한 환경이라면 공격자는 의도하지 않은 콘텐츠를 페이지에 표시할 수 있다.
3. 브라우저가 렌더링한 결과물과 소스 코드 사이에는 차이가 있다. 화면에 보이는 것만 보지 않고, `Ctrl+U`로 소스 코드를 직접 확인하는 습관은 웹 보안 분석의 첫걸음이다.

### 실습 2 — JavaScript 코드 삽입 실습

**목표**: TryHackMe 에디터에서 JavaScript 코드를 추가해 HTML 요소의 내용을 동적으로 변경

**실습 내용 및 플래그**

| 과제 | 내용 | 플래그 |
|------|------|--------|
| JS로 텍스트 변경 | `demo` 요소의 내용을 "Hack the Planet"으로 변경하는 JS 추가 | `JSISFUN` |
| 버튼 이벤트 추가 | 클릭 시 텍스트가 "Button Clicked!"로 변경되는 버튼 추가 | 답변 불필요 |

**배운 점**

1. `document.getElementById("demo").innerHTML = "Hack the Planet"` — 이 코드는 ID가 `demo`인 HTML 요소의 내용을 JavaScript로 실시간 변경한다. 이처럼 JavaScript는 페이지가 로드된 후에도 HTML 구조를 자유롭게 수정할 수 있다. 공격자가 악성 JavaScript를 삽입할 수 있는 환경이라면 페이지 전체를 조작하는 것도 가능하다.
2. `onclick` 이벤트 핸들러를 HTML 요소에 직접 정의하면 버튼 클릭 시 JavaScript 코드가 실행된다. 이를 악용하면 외관상 정상적으로 보이는 버튼이 클릭 시 악성 동작을 수행하도록 만들 수 있다.
3. JavaScript 코드는 외부 파일로 분리해 `<script src="...">` 형태로 불러올 수 있다. 소스 보기를 통해 외부 JS 파일 경로를 확인하고 직접 접근해 코드를 분석하는 것은 보안 테스트에서 중요한 정보 수집 방법이다.

### 실습 3 — 민감 정보 노출 (Ctrl+U 소스 확인)

**목표**: `Ctrl+U`로 취약한 웹사이트의 HTML 소스를 확인해 숨겨진 자격증명 발견

**발견한 정보**:
- 파일: 24~27번 줄 HTML 주석 내부
- 내용: `Username: admin` / `Password: testpasswd`

**플래그**: `testpasswd`

**배운 점**

1. HTML 주석(`<!-- ... -->`)은 브라우저 화면에는 보이지 않지만 `Ctrl+U`(소스 보기)를 누르면 누구나 읽을 수 있다. "TODO: Remove test credentials!"라는 메모가 그대로 운영 서버에 배포된 것처럼, 개발 단계의 임시 자격증명이 실수로 노출되는 일이 실제로 자주 발생한다.
2. 소스 코드에서 발견한 자격증명(`admin:testpasswd`)으로 해당 사이트에 바로 로그인이 가능하다. 더 나아가 사용자가 다른 사이트에서도 같은 비밀번호를 사용한다면, 이를 이용한 크리덴셜 스터핑(Credential Stuffing) 공격으로 다른 계정까지 탈취당할 수 있다.
3. 웹 애플리케이션 보안 점검 시 가장 먼저 수행하는 작업 중 하나가 소스 코드 검토다. HTML 주석, JavaScript 변수, 외부 파일 경로 등에 숨겨진 민감 정보가 있는지 확인하는 것은 기본 체크리스트에 반드시 포함된다.

### 실습 4 — HTML 인젝션 실습

**목표**: 입력 필드에 HTML 코드를 삽입해 페이지에 악성 링크가 렌더링되는 취약점 체험

**삽입한 코드**:
```html
<a href="http://hacker.com"></a>
```

**플래그**: `HTML_INJECTION`

**배운 점**

1. 사용자 입력이 검증(Sanitization) 없이 `innerHTML`에 바로 삽입되면 HTML 인젝션이 가능하다. `<a href="http://hacker.com"></a>`를 입력 필드에 넣었을 때 실제 링크로 렌더링된 것처럼, 공격자는 원하는 HTML 코드를 페이지에 삽입할 수 있다.
2. HTML 인젝션의 더 위험한 형태가 XSS(Cross-Site Scripting)다. `<script>` 태그를 삽입하면 피해자의 브라우저에서 악성 JavaScript가 실행되어 세션 탈취, 키로깅, 피싱 페이지 표시 등이 가능해진다. 입력 필드는 항상 잠재적인 공격 진입점으로 취급해야 한다.
3. 입력 검증은 프론트엔드(브라우저)와 백엔드(서버) 양쪽에서 반드시 수행해야 한다. 프론트엔드 검증만 있으면 공격자가 HTTP 요청을 직접 조작(예: Burp Suite로 가로채기)해 검증 로직을 완전히 우회할 수 있기 때문이다.

## 섹션 16 — 전체 웹 요청 흐름 개요

이 섹션은 모듈 6에서 배운 DNS, HTTP, 웹 서버, 브라우저 렌더링이 실제 웹 요청 한 번 안에서 어떤 순서로 연결되는지 정리한다.

### 브라우저에서 tryhackme.com을 입력했을 때 일어나는 일 (11단계)

```
 1. 브라우저에서 tryhackme.com 요청
        ↓
 2. 내 PC의 로컬 DNS 캐시 확인 (이전에 방문한 적 있으면 여기서 끝)
        ↓ 없으면
 3. ISP의 재귀 DNS 서버에 IP 주소 질의
        ↓ 없으면
 4. 루트 DNS 서버에 .com TLD 서버 위치 질의
        ↓
 5. 권한 있는 DNS 서버가 최종 IP 주소 응답
        ↓
 6. 요청이 WAF(웹 애플리케이션 방화벽) 통과 — 악성 요청 여부 검사
        ↓
 7. 요청이 로드 밸런서 통과 — 여러 서버 중 하나로 분배
        ↓
 8. 웹 서버에 80(HTTP) 또는 443(HTTPS) 포트로 연결
        ↓
 9. 웹 서버가 GET 요청 수신 및 처리
        ↓
10. 웹 애플리케이션이 데이터베이스에서 필요한 데이터 조회
        ↓
11. 브라우저가 받은 HTML/CSS/JS를 렌더링해 화면에 표시
```

### 각 단계와 배운 섹션 연결

| 단계 | 관련 기술 | 이 모듈에서 배운 섹션 |
|------|----------|----------------------|
| 1–5 (IP 탐색) | DNS | 섹션 1–4 |
| 6 (WAF) | 보안 인프라 | 섹션 17 |
| 7 (로드 밸런서) | 가용성 인프라 | 섹션 17 |
| 8–9 (웹 서버) | HTTP/HTTPS, 가상 호스트 | 섹션 5–10, 섹션 18 |
| 10 (데이터베이스) | 백엔드 | 섹션 17, 섹션 18 |
| 11 (브라우저 렌더링) | HTML, CSS, JavaScript | 섹션 12–13 |

> 이 흐름에서 보안 취약점은 어느 단계에도 존재할 수 있다. DNS 단계에서는 DNS 스푸핑, HTTP 단계에서는 MITM 공격, 웹 서버 단계에서는 인젝션 공격, 브라우저 단계에서는 XSS가 발생한다.

## 섹션 17 — 기타 구성 요소

### 로드 밸런싱 (Load Balancing)

트래픽이 매우 많은 웹사이트는 서버 한 대로 모든 요청을 처리할 수 없다. 이때 **로드 밸런서**가 여러 서버 사이에 트래픽을 분산한다.

**로드 밸런서의 두 가지 역할**:

| 역할 | 설명 |
|------|------|
| **트래픽 분산** | 들어오는 HTTP/HTTPS 요청을 여러 서버에 나눠 전달. 알고리즘: 라운드 로빈(Round-Robin), 가중치 기반(Weighted) 등 |
| **상태 확인 (Health Check)** | 서버가 정상 동작 중인지 주기적으로 확인. 응답 없는 서버로는 요청을 전송하지 않음 |

```
클라이언트 요청
      ↓
  로드 밸런서
   ↙   ↓   ↘
서버1  서버2  서버3
```

### CDN (콘텐츠 전송 네트워크, Content Delivery Network)

**CDN** = 정적 파일(JavaScript, CSS, 이미지, 동영상 등)을 전 세계 수천 개의 서버에 분산 저장하는 서비스

- 사용자의 요청이 들어오면 **가장 가까운 서버**에서 파일을 전달 → 응답 속도 향상
- 지구 반대편 원본 서버 대신 물리적으로 가까운 서버가 파일을 보내므로 지연(Latency)이 줄어든다

### 데이터베이스 (Database)

웹사이트는 사용자 정보, 게시물, 상품 데이터 등을 저장하고 불러와야 한다. 웹 서버는 **데이터베이스와 통신**해 데이터를 저장·조회한다.

| 종류 | 유형 |
|------|------|
| MySQL, MSSQL, PostgreSQL | 관계형(RDBMS) |
| MongoDB | NoSQL 도큐먼트 |
| GraphQL | 쿼리 언어 (API 레이어) |

### WAF (웹 애플리케이션 방화벽, Web Application Firewall)

**WAF** = 웹 서버와 인터넷 사이에 위치하여 악성 트래픽으로부터 웹 서버를 보호하는 방화벽

**주요 기능**:
- **요청 분석**: 실제 사용자 요청처럼 보이는 악성 패턴 탐지 및 차단
- **속도 제한 (Rate Limiting)**: 특정 IP가 일정 시간 내에 너무 많은 요청을 보내는 것을 차단 (DDoS 방어)
- **IP 필터링**: 알려진 악성 IP로부터의 요청 차단

```
클라이언트 요청
      ↓
    WAF (악성 요청 차단)
      ↓
  로드 밸런서
      ↓
   웹 서버
```

## 섹션 18 — 웹 서버 작동 방식

### 웹 서버 소프트웨어

**웹 서버**는 HTTP(S) 요청을 수신하고 파일이나 데이터를 반환하는 소프트웨어다.

| 소프트웨어 | 특징 |
|-----------|------|
| **Apache** | 오픈소스, 가장 널리 사용 |
| **Nginx** | 고성능, 리버스 프록시로도 사용 |
| **IIS** | Windows Server용 마이크로소프트 웹 서버 |
| **NodeJS** | JavaScript 기반 비동기 서버 |

### 가상 호스트 (Virtual Hosts)

하나의 웹 서버에서 **여러 도메인**을 동시에 호스팅할 수 있다. 웹 서버는 HTTP 요청의 `Host` 헤더를 확인해 어떤 도메인으로 들어온 요청인지 판단하고, 각각 다른 파일 경로로 라우팅한다.

```
one.com 요청   →  /var/www/website_one
two.com 요청   →  /var/www/website_two
```

하나의 웹 서버에 호스팅할 수 있는 웹사이트 수에는 제한이 없다.

### 정적 콘텐츠 vs 동적 콘텐츠

| 구분 | 정의 | 예시 |
|------|------|------|
| **정적 (Static)** | 변경되지 않는 파일. 서버에서 클라이언트로 그대로 복사 | HTML, CSS, JS, 이미지, 동영상 |
| **동적 (Dynamic)** | 요청할 때마다 실시간으로 생성되는 콘텐츠 | 블로그 최신 게시물, 사용자별 맞춤 페이지 |

### 백엔드 스크립트 언어

동적 콘텐츠는 **백엔드 언어**로 서버에서 처리된다. 클라이언트는 처리 결과(HTML)만 받으며, **백엔드 코드 자체는 볼 수 없다**.

```php
<!-- 서버에서 실행되는 PHP 코드 (클라이언트에게 보이지 않음) -->
<html><body>Hello <?php echo $_GET["name"]; ?></body></html>
```

서버가 처리한 뒤 클라이언트에게 전달되는 결과:

```html
<!-- 브라우저가 받는 최종 HTML -->
<html><body>Hello adam</body></html>
```

주요 백엔드 언어: PHP, Python, Ruby, NodeJS, Java 등

## 실습 기록 (Putting It All Together)

### 실습 1 — 전체 웹 요청 단계 순서 맞추기 퀴즈

**목표**: 브라우저에서 웹사이트를 요청할 때 일어나는 11단계를 올바른 순서로 배열

**정답 순서**:

| 순서 | 단계 |
|------|------|
| 1 | 브라우저에서 tryhackme.com 요청 (Request tryhackme.com in your browser) |
| 2 | 로컬 캐시에서 IP 주소 확인 (Check Local Cache for IP Address) |
| 3 | 재귀 DNS 서버에서 주소 확인 (Check your Recursive DNS Server for Address) |
| 4 | 루트 서버에 권한 있는 DNS 서버 위치 질의 (Query root server to find authoritative DNS Server) |
| 5 | 권한 있는 DNS 서버가 IP 주소 응답 (Authoritative DNS server advises the IP address for the website) |
| 6 | 요청이 WAF를 통과 (Request passes through a Web Application Firewall) |
| 7 | 요청이 로드 밸런서를 통과 (Request passes through a Load Balancer) |
| 8 | 80번(HTTP) 또는 443번(HTTPS) 포트로 웹 서버에 연결 (Connect to Webserver on port 80 or 443) |
| 9 | 웹 서버가 GET 요청 수신 (Web server receives the GET request) |
| 10 | 웹 애플리케이션이 데이터베이스와 통신 (Web Application talks to Database) |
| 11 | 브라우저가 HTML을 렌더링해 웹사이트 표시 (Your Browser renders the HTML into a viewable website) |

**플래그**: `THM{YOU_GOT_THE_ORDER}`

**배운 점**

1. 웹사이트 하나를 보기까지 DNS 조회 → WAF 통과 → 로드 밸런서 → 웹 서버 → 데이터베이스 → 브라우저 렌더링의 11단계가 순서대로 일어난다. 각 단계가 이 모듈에서 배운 별개의 기술들이며, 실제 운영 환경에서는 이 모든 단계가 수백 밀리초 안에 완료된다.
2. WAF와 로드 밸런서가 DNS 응답 이후, 웹 서버 연결 이전에 위치한다는 점이 중요하다. 공격 트래픽은 웹 서버에 도달하기 전에 WAF에서 먼저 걸러지고, 정상 트래픽은 로드 밸런서가 분산시킨다. 이 순서를 알면 보안 아키텍처를 이해하는 데 도움이 된다.
3. 웹 서버가 GET 요청을 받은 후 데이터베이스와 통신한다는 점은 SQL Injection 공격의 진입점을 이해하는 데 직결된다. 웹 서버가 사용자 입력을 검증 없이 데이터베이스 쿼리에 삽입하면 공격자가 임의의 쿼리를 실행할 수 있다.
